# 컴파일 언어 vs 인터프리터 언어: 실행 방식의 차이가 만드는 개발의 결

![컴파일러와 인터프리터의 차이](/public/images/compiler-vs-interpreter.png)

개발을 시작하며 가장 먼저 접하게 되는 개념 중 하나가 바로 "이 언어는 컴파일 언어인가, 아니면 인터프리터 언어인가?" 하는 질문이다. C, Java, Go 같은 언어들은 컴파일 과정을 거쳐야 한다고 하고, JavaScript나 Python은 실행 버튼만 누르면 바로 동작한다고 한다.

도대체 코드를 기계가 이해하도록 만드는 과정에서 어떤 차이가 있길래 실행 속도나 개발 편의성이 달라지는 걸까? 이 블랙박스를 열어보기 위해 두 방식의 핵심 메커니즘을 시뮬레이션해 보며 차이점을 정리해 보았다.

---

## 1. 번역의 시점: 미리 하느냐, 그때그때 하느냐

핵심적인 차이는 **"인간이 쓴 소스 코드가 언제 기계어로 번역되는가"**에 있다.

### 컴파일러 (Compiler)
컴파일러 방식은 소스 코드 전체를 흝어보고, 실행하기 전에 미리 통째로 기계어(이진 코드)로 번역해 버린다. 번역이 완료된 결과물(실행 파일)은 기계가 즉시 읽을 수 있는 상태이므로, 실행 시점에 매우 빠른 속도를 자랑한다. 하지만 코드를 한 줄만 고쳐도 다시 전체를 번역(재컴파일)해야 하는 번거로움이 있다.

### 인터프리터 (Interpreter)
인터프리터 방식은 통역가와 같다. 소스 코드를 한 줄씩 읽어 내려가며 그 즉시 기계 어로 번역하고 실행한다. 별도의 번역 완료 파일이 생성되지 않으며, 코드를 수정하고 바로 실행해 볼 수 있어 개발 피드백 루프가 매우 짧다. 다만 실행 시점에 매번 번역 과정을 거치므로 순수 실행 속도는 컴파일 언어보다 느릴 수밖에 없다.

---

## 2. 코드로 시뮬레이션해 본 동작 원리

이 차이를 더 명확히 이해하기 위해, TypeScript로 가상의 명령어 세트를 실행하는 간단한 시뮬레이터를 구현해 보았다.

### 인터프리터 방식
인터프리터는 코드를 순회하며 그 자리에서 기능을 수행한다.

```typescript
// src/examples/compiler-interpreter-example.ts 중 발췌
export class Interpreter {
  interpret(code: Instruction[]): string[] {
    const logs = [];
    for (const instruction of code) {
      // 읽는 즉시 실행!
      if (instruction.type === 'PRINT') {
        logs.push(`[Interpreter Output]: ${instruction.value}`);
      }
      // ... 생략
    }
    return logs;
  }
}
```

### 컴파일러 방식
컴파일러는 코드를 실행하는 대신, 나중에 실행될 수 있는 '번역된 함수'를 반환한다.

```typescript
// src/examples/compiler-interpreter-example.ts 중 발췌
export class Compiler {
  compile(code: Instruction[]): () => string[] {
    // 미리 번역된 로직을 담은 클로저를 생성하여 반환
    const compiledProgram = () => {
      const logs = [];
      for (const instruction of code) {
        // 이미 기계가 가공 완료된 로직을 실행하는 단계
        if (instruction.type === 'PRINT') {
          logs.push(`[Binary Exec Output]: ${instruction.value}`);
        }
      }
      return logs;
    };

    return compiledProgram;
  }
}
```

이 테스트 코드를 실행해 본 결과, 인터프리터는 `interpret` 호출 시점에 모든 일이 벌어지지만, 컴파일러는 `compile`과 `execute` 단계가 명확히 분리되는 것을 확인할 수 있었다.

## 3. 어떤 방식이 더 우월한가?

사실 오늘날 현대의 프로그래밍 언어들은 이 두 방식의 경계가 희미해지고 있다. 

- **Java**: 소스 코드를 바이트코드로 컴파일한 후, JVM 내부의 JIT(Just-In-Time) 컴파일러가 실행 시점에 다시 기계어로 번역하여 속도를 끌어올린다. (컴파일 + 인터프리터의 혼합)
- **V8 엔진 (JS)**: 인터프리터 방식으로 시작하지만, 자주 실행되는 코드(Hot Spot)는 실시간으로 컴파일하여 최적화한다.

결국 "어떤 방식이 더 낫다"기보다는, 대규모 시스템의 안정성과 압도적인 성능이 필요할 때는 컴파일 언어가 강점을 발휘하고, 빠른 개발과 유연한 테스트가 우선일 때는 인터프리터 기반의 언어들이 더 매력적인 선택지가 된다는 점을 배웠다.

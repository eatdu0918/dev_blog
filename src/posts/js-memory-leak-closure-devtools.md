# 내 탭은 왜 점점 느려지는가: 클로저의 역기능과 메모리 누수 생존기

입사 초창기, 나에게 메모리 누수(Memory Leak)란 그저 C 코딩하던 시절, `malloc`을 쓰고 반납하지 않은 선배들의 무용담 혹은 서버 개발자들만의 비극쯤으로 치부되었다. "자바스크립트엔 그딴 거 없지, 구글의 V8 엔진이 알아서 청소도 다 해주고, 느려지면 사용자가 F5(새로고침) 한 번 누르면 백지장처럼 가벼워지는걸!" 이 얼마나 건방지고 무지했던 나의 착각이었던가.

문제는 내가 리액트로 개발한 사내용 대시보드 애플리케이션에서 터졌다. 
운영팀 동료의 슬랙 창이 불을 뿜었다. "개발자님! 대시보드 켜놓고 3시간쯤 지나면 스크롤이 끊기고 버튼이 안 눌려요. 그러다가 브라우저 탭이 **'앗, 이런!(Aw Snap!)'** 화면과 함께 죽어버려요."

내 코드가 프론트엔드의 메모리를 갉아먹고 있었다. 리프레시 없는 끈적한 Single Page Application(SPA) 환경을 만들어놓고, 그 무게를 감당하지 못한 나 자신의 무책임을 깨달은 순간이었다.

---

## 1. V8도 치워주지 못하는 '버려진 끈'을 찾아서

원인을 디버깅하며 자바스크립트 세계의 가비지 컬렉션(Mark-and-Sweep) 방식을 파고들었다. 나는 더 이상 쓰지 않는 변수들을 가차 없이 버린다고 생각했는데, 코드 어딘가에서는 그 변수들을 가리키는 포인터(끈)를 계속 몰래 쥐고 놓지 않는 상황들이 발생하고 있었다. 바로 **논리적 누수** 패턴이었다.

가장 먼저 나를 배신한 건 그토록 사랑했던 마법의 문법, **클로저(Closure)** 였다.

```javascript
let cacheData = null;

const createProcessor = function() {
    const originalCache = cacheData;
    const somethingUnused = function() {
        // [사건의 전말] 아무도 호출하지 않는 유령 함수지만, 
        // 클로저 문맥 때문에 거대한 originalCache의 끈을 부여잡고 있다.
        if (originalCache) console.log("이 끈을 절대로 놓지 않을 테야");
    };

    // 거대한 API 모수값. 원래 같으면 쓰이고 지워져야 맞다.
    // 하지만 위클로저가 감싸고 있어서 메모리에서 절대 지워지지 않는다.
    cacheData = {
        heavyPayload: new Array(500000).fill('Data'),
        someMethod: function() {}
    };
};

setInterval(createProcessor, 5000); 
```

5초마다 수십만 개의 데이터가 할당되고, 이것들이 클로저의 마법진 안에 갇힌 채 영원한 좀비 데이터가 되어 메인 스레드의 메모리를 집어삼키고 있었던 것이다.

## 2. Chrome DevTools 부검실에 들어서다

막연히 눈치코치로 소스를 읽어서 이걸 수백 개의 파일 속에서 찾아내는 건 미친 짓이었다. 나는 결국 모니터 화면 분할을 하고 그 무섭다던 Chrome DevTools의 **`Memory` 탭** 부검실을 열었다. 

### 1) 범행 현장 타임라인 (Allocation Timeline)
의심되는 특정 동작(리스트를 무한 스크롤 하거나 트리를 열고 닫는 부분)을 계속 찍먹해 보며 타임라인을 기록해 보았다.
메모리가 점점 상승선을 그리다가, 내가 화면을 끄거나 페이지를 이탈하면 파란 막대기(할당량)가 바닥으로 떨어져야 정상이다. 그런데 내 그래프는 파란색 선이 회색의 좀비 계단으로 변형되며 죽지 않고 시체처럼 계속 쌓여가고 있었다. 누수가 확실했다.

### 2) 힙 덤프 두 개로 틀린 그림 찾기 (Heap Snapshot)
더 정밀한 타격을 위해 페이지 진입 직후에 스냅샷을 찍고, 누수 의심 행위를 10번 반복한 후 또 스냅샷을 찍었다.
상단 보기 옵션을 `Comparison(비교)`으로 맞추고 두 덤프 사이에 "새로 태어났으나 영원히 지워지지 못하고 남은" 객체들만 솎아냈다.

의심 가는 결과물이 떴다. **"Detached DOM tree"**. 화면에서는 분명히 지워진(`Unmount`된) DOM 노드였건만 자바스크립트의 힙 메모리가 그 껍데기를 주렁주렁 매달고 있었다.
"맙소사. `useEffect`에서 `addEventListener`를 걸어놓고 반환 함수인 `Cleanup (removeEventListener)` 처리를 완전히 까먹고 있었어." 
화면 렌더링은 날아갔을지언정 브라우저 `window` 최상단 객체에서는 내 컴포넌트를 계속 짝사랑하며 참조 밧줄을 묶고 놔주질 않았던 것이다.

## 맺으며

프론트엔드 환경에서의 메모리 관리는 더 이상 브라우저만 믿고 손을 놔버릴 영역이 아니었다. 사용자는 우리가 만든 SPA 세상 안에 며칠씩 상주하기도 한다.

에러 콘솔에 아무것도 뜨지 않는 침묵 속에서 메인 스레드가 서서히 늪으로 빨려 들어가는 현상. 이 컴파일 에러 없는 침묵의 살인마를 잡기 위해 타임라인을 파헤치던 며칠간의 시간은, 나를 코더(Coder)에서 벗어나 앱의 수명주기(Lifecycle) 전체를 통제해야만 직성이 풀리는 한 명의 아키텍트로 단련시켜준 선물이었다. 개발자의 길은 생각보다 무거운 책임감을 요한다는 걸 실감하는 요즘이다.

# 브라우저 렌더링 파이프라인: HTML이 픽셀로 변하는 과정

![브라우저 렌더링 파이프라인 도식](/public/images/browser-rendering-pipeline.png)

웹 개발자로서 코드를 작성할 때, 우리가 작성한 텍스트 파일이 어떻게 사용자 모니터의 조밀한 픽셀로 변환되는지 그 과정을 이해하는 것은 매우 중요함을 느꼈습니다. 단순히 "브라우저가 알아서 그려준다"는 정적 믿음을 넘어, 내부에서 일어나는 **Critical Rendering Path(주요 렌더링 경로)**를 파헤쳐 보며 웹 애플리케이션의 성능 최적화가 어느 시점에서 이루어지는지 학습해 보았습니다.

---

## 1. 데이터의 형상화: DOM과 CSSOM의 탄생

웹 페이지의 여정은 서버로부터 받은 바이트(Bytes) 데이터를 문자로 파싱하는 것에서 시작됩니다.

- **DOM(Document Object Model) 트리**: 브라우저는 HTML 태그를 읽어 들이며 각 요소를 객체 노드로 변환합니다. 이들이 서로 부모-자식 관계를 맺으며 형성되는 것이 바로 DOM 트리임을 알게 되었습니다.
- **CSSOM(CSS Object Model) 트리**: HTML 파싱 중 CSS 파일을 마주하면, 브라우저는 스타일 정보를 파싱하여 별도의 CSSOM 트리를 만듭니다. 스타일은 상속(Cascade)되는 특성이 있어, 이 트리 역시 계층적인 구조를 가짐을 보았습니다.

---

## 2. 결합과 계산: Render Tree와 Layout

두 개의 트리가 준비되면 브라우저는 이들을 하나로 합쳐 실제 화면에 그릴 요소들만 골라냅니다.

- **Render Tree**: DOM과 CSSOM이 결합하여 만들어지는 트리입니다. 여기서 흥미로운 점은 `display: none`과 같이 화면에 보이지 않는 요소들은 렌더 트리에 포함되지 않는다는 사실이었습니다.
- **Layout (Reflow)**: 렌더 트리의 각 노드가 화면의 어느 위치에, 어떤 크기로 배치될지 계산하는 단계입니다. 뷰포트(Viewport)의 크기에 따라 상대적인 위치가 결정되는 과정을 보며, 왜 빈번한 레이아웃 변화가 성능에 치명적인지 이해할 수 있었습니다.

---

## 3. 시각화의 완성: Paint와 Composite

이제 계산된 위치에 실제 색상을 입히고 레이어를 합치는 단계로 넘어갑니다.

- **Paint**: 레이아웃 단계에서 계산된 기하학적 정보들을 바탕으로 노드에 실제 픽셀(색상, 텍스트, 이미지 등)을 채워 넣습니다.
- **Composite**: 페이지의 일부분을 별도의 레이어로 나누어 그린 후, 이들을 하나로 합성합니다. 특정 요소에 `transform`이나 `opacity` 같은 속성을 주었을 때 레이아웃과 페인트를 생략하고 바로 합성 단계로 넘어갈 수 있다는 점이 성능 최적화의 핵심임을 깨달았습니다.

---

## 💡 마치며: 렌더링 최적화의 방향성

브라우저의 렌더링 파이프라인을 학습하며 느낀 점은, 우리가 작성하는 코드 한 줄이 이 긴 여정의 어느 단계를 다시 실행(Reflow, Repaint)하게 만드는지 아는 것이 중요하다는 사실이었습니다. 레이아웃 변화를 최소화하고, 가능한 한 합성(Composite) 단계에서 처리될 수 있는 속성을 활용하는 습관이 더 부드러운 사용자 경험을 만드는 밑바탕이 됨을 알게 되었습니다. 앞으로는 화면의 변화를 설계할 때 브라우저의 내부 호흡을 한 번 더 고려해 보려 합니다.

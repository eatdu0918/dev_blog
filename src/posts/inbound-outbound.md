# 클라우드 방화벽의 기본: 인바운드와 아웃바운드 트래픽에 대한 이해

스프링 부트로 백엔드 API를 개발한 뒤 AWS EC2 클라우드 환경에 성공적으로 배포했다고 생각했던 첫날, 로컬에서는 잘되던 브라우저 요청이 클라우드 IP만 타면 무한 타임아웃에 빠지는 당혹스러운 상황을 마주했다.

서버 로그 자체에는 아무런 에러가 찍히지 않았고, 수 시간의 삽질 끝에 원인이 애플리케이션 코드가 아닌 **"AWS 보안 그룹(Security Group) 방화벽 설정"**에 있다는 것을 깨달았다. 이 트러블슈팅을 계기로 머릿속에 혼재되어 있던 인바운드(Inbound)와 아웃바운드(Outbound) 네트워크 트래픽의 방향성 개념을 다시금 확립하게 된 과정을 기록한다.

---

## 🏰 방향성의 절대 기준: 내가 띄운 서버 (내부 네트워크)

네트워크 흐름을 파악할 때 가장 헷갈렸던 점은 '누구를 기준으로' 방향을 잡느냐였다. 
수많은 문서와 설정 콘솔에서 말하는 IN과 OUT의 기준은 **언제나 내가 구축한 서버 인스턴스(내부 네트워크) 그 자체**임을 명확히 인지하고 나서야 개념이 잡히기 시작했다.

### ⬇️ 인바운드 (Inbound) : 밖에서 내 시스템 안으로 들어오는 트래픽
외부망 통신망이나 클라이언트에서 **내 서버 내부로** 뚫고 들어오려는 데이터 흐름이다. 기본적으로 방화벽은 인터넷의 모든 익명 접근을 차단하므로, 내 서비스가 받아야 할 요청에 대해서만 명시적으로 포트를 열어주어야 했다.

-   **막혔을 때의 증상**: 클라이언트가 `www.myapi.com`을 호출하면 연결 자체가 성립되지 않음. 내가 원격으로 EC2 터미널에 SSH(22포트) 접속을 하려 해도 튕겨 나감.

### ⬆️ 아웃바운드 (Outbound) : 내 시스템 안에서 밖으로 뻗어나가는 트래픽
**내 서버 내부에서** 시작되어 외부 인터넷망의 목적지를 향해 나가는 데이터 흐름이다. 클라우드 보안 그룹 구성을 살펴보니 아웃바운드는 보통 기본적으로 "모든 포트(0.0.0.0/0) 개방 허용" 상태로 세팅되어 있었다. 내부망에서 스스로 요청해 나가는 패킷은 어느 정도 신뢰할 수 있다고 간주하기 때문이다.

-   **실무적 상황**: 내 스프링 서버가 사용자 결제를 위해 외부 포트원(Iamport) API를 HTTP 요청으로 찌르거나, 터미널에서 `npm install` 명령을 쳐서 외부 패키지 레지스트리에서 파일을 다운로드 받아올 때 발생하는 모든 트래픽.

---

## 🛡️ 실전 적용: AWS 보안 그룹(Security Group) 방어선 구축

EC2 인스턴스에 방화벽 정책을 세팅하면서, 내가 서비스할 목적에 맞는 필수 구멍(포트)만을 인바운드 규칙에 명시해주는 화이트리스트(Whitelist) 방식을 실무에 적용했다.

| 서비스 목적              | 프로토콜 (포트 번호)    | 개방 논리 (의존성)                                                                                                 |
| :----------------------- | :---------------------- | :----------------------------------------------------------------------------------------------------------------- |
| **원격 관리자 접속**     | SSH (`TCP 22`)          | 터미널 내부 관리를 위해 필수. (보안 강화를 위해 내 IP 주소 대역에서만 접근 가능하도록 타이트하게 제한함)           |
| **일반 웹 접속 API**     | HTTP (`TCP 80`)         | 클라이언트의 기본 HTTP 통신 수용을 위해 전체 대역 개방.                                                            |
| **안전한 웹 접속 (SSL)** | HTTPS (`TCP 443`)       | SSL 인증서가 씌워진 안전한 통신 패킷을 수신하기 위해 개방.                                                         |
| **데이터베이스 연결**    | PostgreSQL (`TCP 5432`) | 백엔드 서버가 별도의 RDS(데이터베이스) 서버로 접속해야 할 때, RDS 측의 인바운드로 백엔드 EC2의 접근을 허용 처리함. |

## 💡 최종 회고

단순히 코드만 에러 없이 컴파일된다고 해서 라이브 서비스가 정상 동작하는 것이 아님을 뼈저리게 깨달은 경험이었다. 

서버 인프라를 다루기 위해서는 애플리케이션 외부를 감싸고 있는 방화벽의 존재를 반드시 인지해야 하며, 타 서버 API와의 연동 시 Timeout 예외가 터지면 로직 내부를 뒤지기 전에 클라우드 인프라의 아웃바운드/인바운드 막힘 여부부터 의심해 보는 사고 단위의 전환을 이뤄낼 수 있었다.

## Q1. 캐시는 어디에 두는 게 좋나요?

**A.** 한 군데가 아니라 **계층**입니다.

브라우저 → CDN → Reverse Proxy → 애플리케이션(Redis) → DB. 각 계층은 TTL과 무효화 비용이 다릅니다.

선택 기준:
- **거의 안 변하는 정적 자산** → CDN + 긴 max-age + 해시 파일명.
- **자주 안 변하는 사용자별 데이터** → Redis(짧은 TTL) + 변경 시 삭제.
- **실시간성 요구** → 캐시 안 쓰거나 SWR 패턴.
- **개인정보** → `private, no-store` 또는 서버 전용 캐시.

## Q2. HTTP 캐시 헤더의 핵심을 정리해 주세요.

**A.** 가장 자주 쓰는 헤더들입니다.

- **`Cache-Control: max-age=N`**: N초 동안 캐시 사용. `public`(공유) vs `private`(브라우저만).
- **`no-store`**: 절대 저장 금지(민감 정보).
- **`no-cache`**: 저장은 하지만 사용 전 재검증. **`no-store`와 다름**.
- **`ETag` / `If-None-Match`**: 컨텐츠 해시. 304로 본문 전송 절약.
- **`stale-while-revalidate=N`**: 만료된 응답을 즉시 반환하면서 백그라운드 갱신. 체감 성능에 큰 효과.
- **`stale-if-error`**: 오리진 장애 시 만료 응답으로 fallback.

정적 자산은 **immutable + 1년 max-age + 파일명 해싱**(`app.abc123.js`)이 정석입니다.

## Q3. Redis 캐시 패턴 중 무엇을 가장 자주 쓰시나요?

**A.** **Cache-Aside(Lazy Loading)** 가 가장 흔합니다.

- 읽기: 캐시 miss → DB 조회 → 캐시 저장.
- 쓰기: DB 갱신 후 **캐시 삭제**(갱신이 아닙니다).

다른 패턴:
- **Write-Through**: DB+캐시 동시 쓰기. 일관성↑, 지연↑.
- **Write-Behind**: 캐시에 먼저 쓰고 DB 비동기. 빠르지만 유실 위험.
- **Read-Through**: 캐시 라이브러리가 miss 시 DB까지 처리.

## Q4. DB 갱신 시 캐시 삭제가 갱신보다 안전한 이유는?

**A.** **동시성 시나리오에서 stale 값이 캐시에 남는 문제를 회피**하기 위해서입니다.

캐시 갱신 시나리오에서는 두 트랜잭션이 동시에 들어오면:
1. T1이 DB 갱신.
2. T2가 DB 갱신.
3. T2가 캐시에 자기 값 저장.
4. T1이 캐시에 자기(이미 stale) 값 저장 → 캐시에 옛 값 남음.

**삭제만 하면** 다음 읽기에서 DB 최신 값으로 채워지므로 안전합니다. race condition이 완전히 사라지진 않으니 **TTL은 최후의 보험**입니다.

## Q5. Cache Stampede는 무엇이고 어떻게 막나요?

**A.** 핫 키가 만료되는 순간 수많은 요청이 동시에 DB로 몰리는 문제입니다.

해결 패턴:
- **Probabilistic Early Expiration**: 만료 전부터 일정 확률로 미리 갱신.
- **단일 갱신 락(SETNX)**: 한 요청만 DB 조회, 나머지는 대기 또는 stale 반환.
- **stale-while-revalidate**: 만료 후 잠시 stale 반환하면서 백그라운드 갱신.
- **single-flight**: 같은 키 in-flight 요청을 메모리에서 통합.

## Q6. Hot Key, Big Key 문제는 어떻게 해결하나요?

**A.**
- **Hot Key**: 한 키에 트래픽 집중 → Redis 단일 노드 CPU 병목. **로컬 캐시 한 단계 추가**(에어리어 캐시) 또는 **키 분할**(슬롯 N개로 분산).
- **Big Key**: 한 키 값이 수 MB → 네트워크/메모리 부담. **분할 저장**(예: 사용자별 리스트를 여러 키로 나눔).

운영 모니터링: `redis-cli --hotkeys`, `--bigkeys`로 탐지.

## Q7. TTL은 어떻게 결정하시나요?

**A.** **비즈니스가 허용하는 stale 시간**으로 정합니다.

체크리스트:
- 사용자가 5초 stale을 봐도 되는가, 1시간 stale을 봐도 되는가?
- 쓰기 빈도가 얼마나 되는가?
- 캐시 적중률 측정 후 조정.

원칙: **TTL은 안전장치, 무효화가 정확성 도구**. 긴 TTL + 정확한 무효화에만 의존하면 분산 환경에서 무효화 누락이 반드시 발생해 사고가 납니다.

---

## Q8. no-cache와 no-store는 어떻게 다른가요?

**A.** 자주 헷갈리는 부분입니다.

- **`no-cache`**: 캐시는 **저장 가능**, 사용 전 **재검증** 강제. 304 응답 받을 수 있음.
- **`no-store`**: **저장 자체 금지**. 진짜 캐시 안 함. 민감 정보용.

이름과 의미가 직관과 반대라 항상 헷갈립니다. "캐시 진짜 끄려면 `no-store`"로 외워두면 됩니다.

---

## Q9. Redis 장애 시 어떻게 fallback하시나요?

**A.** **origin(DB)으로 fallback** 가능하게 설계합니다.

- 캐시 호출에 timeout 설정(예: 100ms).
- timeout/에러 시 DB 직접 조회.
- 회로 차단기(circuit breaker)로 Redis 장애 시 일시적으로 캐시 우회.
- DB가 그 트래픽을 견딜 수 있는지 사전에 부하 테스트.

캐시는 빠르게 응답하기 위함이지 의존성이 되어선 안 됩니다.

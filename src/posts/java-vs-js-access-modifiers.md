# Java와 JavaScript의 접근 제어자: 캡슐화의 벽을 넘나들며 느낀 설계의 차이

![접근 제어자와 캡슐화 개념도](/public/images/java-vs-js-access-modifiers.png)

소프트웨어 개발을 시작하며 가장 먼저 배웠던 객체 지향의 원칙 중 하나는 **캡슐화(Encapsulation)**였다. 데이터를 외부의 무분별한 접근으로부터 보호하고, 정해진 통로(Method)를 통해서만 소통하게 만드는 것. 이 원칙을 지키기 위해 우리는 '접근 제어자'라는 도구를 사용한다.

최근 백엔드 서비스(Java)와 프론트엔드 모듈(TypeScript/JS)을 동시에 다루는 프로젝트를 진행하면서, 두 진영이 '정보를 숨기는 방식'에 대해 얼마나 다른 철학을 가지고 있는지 피부로 느끼게 되었다. 단순히 문법이 다른 것이 아니라, 언어가 개발자를 신뢰하는 방식 자체가 달랐다.

---

## 1. Java: 철저한 계급 기반의 보안망

Java를 사용하며 느낀 접근 제어자는 마치 견고한 성벽(~~마치 진격의 거인의 성벽과 같이...~~)과 같았다. `private`, `default`, `protected`, `public`으로 나뉘는 4단계의 레벨은 패키지 단위의 설계까지 고려된 아주 정밀한 제어 수단이었다.

1. **`private`**: 클래스 외부에서는 절대 넘볼 수 없는 철저한 은닉.
2. **`default` (package-private)**: 같은 패키지라는 '동료'들에게만 허용되는 공유 공간.
3. **`protected`**: 상속 관계라는 '혈연'에게만 열어주는 통로.
4. **`public`**: 세상 모든 곳에 공개된 광장.

Java의 컴파일러는 이 규칙을 어기는 순간 맹렬하게 빨간 줄을 그으며 실행 자체를 막아버린다. 덕분에 협업 과정에서 "누군가 내 의도와 다르게 이 내부 변수를 건드리면 어떡하지?"라는 불안감을 가질 필요가 없었다. 설계자의 의도는 컴파일 타임에 강제되었기 때문이다.

---

## 2. JavaScript: 유연함 속에서 피어난 진정한 프라이빗 (#)

반면, JavaScript의 캡슐화 역사는 꽤나 험난했다. 초기 런타임 환경에서는 진정한 의미의 프라이빗 변수가 존재하지 않았다. 관례상 변수명 앞세 밑줄(`_password`)을 붙여 "제발 건드리지 말아주세요"라고 부탁하는 수준이었다.

하지만 ECMAScript 2022에 등장한 **`#` (Private Fields)** 문법을 보며 나는 신선한 충격을 받았다.

```javascript
class SecureVault {
  #secretKey = "SUPER_SECRET"; // 진정한 프라이빗 필드

  getSecret() {
    return this.#secretKey;
  }
}

const vault = new SecureVault();
console.log(vault.#secretKey); // Syntax Error: 런타임 이전에 엔진 수준에서 차단됨
```

재미있는 점은 TypeScript의 `private` 키워드와의 차이였다. TypeScript의 `private`은 컴파일 후 JavaScript로 변환되면 일반 public 변수와 다름없이 동작한다(개발 환경에서의 경고일 뿐이다). 하지만 `#` 접두사는 런타임 엔진 자체가 접근을 봉쇄한다. "문법적 제어"를 넘어선 "물리적 격리"를 JavaScript에서 비로소 마주하게 된 것이다.

---

## 3. 문법의 차이가 만든 사고의 확장

이 두 언어의 차이를 직접 경험하며, 나는 단순히 '어떤 키워드를 쓰느냐'보다 **'어떤 수준에서 보호하느냐'**가 설계에 미치는 영향을 깨닫게 되었다.

- **Java**는 설계자가 세운 규칙이 곧 법이며, 이를 어기면 시스템이 돌아가지 않게 함으로써 안정성을 확보한다.
- **JavaScript**는 기술적인 한계나 유연성 때문에 오랫동안 개발자 간의 '약속(Convention)'에 의존해 왔지만, 이제는 언어 자체적으로 더 강력한 캡슐화 도구를 제공하며 현대적인 설계를 지원하고 있다.

나 역시 처음에는 JS의 `#` 문법이 어색하게만 느껴졌다. 하지만 "우연히라도 내부 상태를 오염시키지 않겠다"는 강력한 의지를 표현하기에 이보다 명확한 방법은 없음을 깨달았다.

---

## 💡 회고: 보호받는 코드가 주는 안정감

그동안 프로젝트를 진행하며 '동작하는 코드'를 만드는 데 급급해 접근 제어자를 단순히 귀찮은 제약 사항으로만 생각했던 적이 있었다. 하지만 Java의 엄격한 제어와 JavaScript의 진화 과정을 함께 보며, 캡슐화는 타인을 막기 위한 벽이 아니라 **내 코드가 오해받지 않도록 보호하는 최소한의 장치**라는 사실을 가슴 깊이 이해하게 되었다.

이제는 변수 하나를 선언할 때도 한 번 더 고민하게 된다. "이 정보는 정말로 밖에서 알아야 할까?" "이 벽은 충분히 견고한가?" 이러한 고민의 흔적들이 모여 더 단단하고 유지보수하기 좋은 아키텍처를 만든다는 것을, 나는 서로 다른 두 언어를 넘나들며 조금씩 배워나가고 있다.

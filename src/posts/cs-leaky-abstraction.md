# 추상화 누수: 완벽한 도구는 없다는 겸손함

프레임워크나 라이브러리를 쓰기만 하면 모든 복잡함이 해결될 거라 믿었던 때가 있었습니다. 하지만 프로젝트를 진행하다 보니, 깔끔한 인터페이스 너머로 로우레벨의 기술들이 불쑥 튀어나와 발목을 잡는 상황(Leaky Abstraction)을 자주 마주하게 되었습니다.

조엘 스폴스키(Joel Spolsky)의 **"모든 부실하지 않은 추상화는 실질적으로 누수가 발생한다(All non-trivial abstractions, to some degree, are leaky)"**라는 명언을 통해 이 현상의 본질을 깨달으며 배운 인사이트를 공유합니다.

---

## 1. 추상화 누수(Leaky Abstraction)란?

추상화는 복잡한 내부 동작을 감추고 사용자에게 편리한 도구를 제공하는 과정입니다. 하지만 **추상화가 완벽하지 않아 내부의 로우레벨 구현 내용이 상위 레벨로 새어 나오는 현상**을 추상화 누수라고 합니다.

![추상화 누수의 시각화](/images/leaky_abstraction_concept.png)

위 그림처럼 견고하게 감싸진 추상화 계층(Pipe/Box)이라 하더라도, 때로는 내부의 복잡한 태엽(Gears)이나 코드 조각들이 외부로 노출되어 우리가 그것을 직접 신경 써야만 하는 상황이 발생합니다.

---

## 2. 실전 예제: ORM은 DB를 완벽히 숨기지 못한다

가장 대표적인 예시는 데이터베이스 연동을 편리하게 해주는 ORM(Object-Relational Mapping)에서 볼 수 있습니다.

### 추상화의 약속
"SQL을 직접 짜지 않아도 됩니다. 그냥 객체의 메서드만 호출하세요!" (예: `User.findAll()`)

### 누수의 발생: N+1 문제

```typescript
// 추상화된 메서드 호출
const posts = await postRepository.find(); 

// 누수 발생: 하지만 성능을 고려한다면 내부 SQL이 어떻게 나가는지 알아야 합니다.
// find()만 하면 연관된 유저 정보를 가져오기 위해 100번의 추가 쿼리가 나갈 수 있죠(N+1).
// 결국 우리는 "추상화된" 코드를 짤 때 "로우레벨 SQL의 성능"을 고민해야 합니다.
```

이렇게 편리한 `find()` 메서드를 쓰면서도, 정작 성능 이슈를 해결하기 위해 로우레벨의 JOIN이나 Fetch Join 전략을 직접 개입시켜야 하는 상황이 바로 전형적인 추상화 누수입니다.

---

## 3. 누수로부터 얻은 깨달음: 겸손한 설계

추상화 누수를 겪으며 제가 깨달은 세 가지 교훈은 다음과 같습니다.

1. **도구를 맹신하지 말 것**: 어떤 도구도 복잡성을 100% 제거해주지 않습니다. 도구의 내부 원리를 모르면 결국 누수가 발생했을 때 대처할 수 없습니다.
2. **최소한의 추상화**: 필요 이상으로 거대한 추상화 벽을 치는 것보다, 누수가 발생했을 때 유연하게 대응할 수 있는 '투명한 추상화'를 고민하게 되었습니다.
3. **학습의 범위**: 추상화된 상위 기술만 배우는 게 아니라, 그 아래에서 실제로 어떤 일이 벌어지는지(예: ORM 아래의 SQL, JS 프레임워크 아래의 DOM 조작 등)를 함께 공부해야 함을 실감했습니다.

---

## 4. 회고와 성장

예전에는 "프레임워크가 이상해서 이런 문제가 생겼어"라며 도구 탓을 하곤 했습니다. 하지만 이제는 **"추상화는 단지 복잡성을 잠시 빌리는 것일 뿐"**이라는 사실을 겸허히 받아들입니다.

누수는 골치 아픈 버그가 아니라, 시스템의 깊은 곳을 이해할 수 있게 해주는 초대장과도 같다는 점을 배웠습니다. 이제는 추상화 레이어를 만들 때도 "이 레이어가 언제 누수될 수 있을까?"를 미리 자문하며, 더 견고하고 예측 가능한 설계를 하려 노력하고 있습니다.

# RESTful API란 무엇인가? 개념과 실무 설계 규칙

웹 개발을 하면서 가장 많이 듣고, 또 끊임없이 고민하게 되는 부분 중 하나가 "나의 API 엔드포인트는 과연 **RESTful** 한가?"라는 명제였다.

아무런 규칙 없이 API 주소를 마구잡이로 찍어내다 보면, 프론트엔드 개발자와의 소통 비용이 기하급수적으로 증가하는 것을 뼈저리게 겪었다. 이를 타파하고 현대 웹 아키텍처의 표준인 REST 규칙을 체화하기 위해 학습했던 가이드라인을 회고한다.

---

## 📖 REST (REpresentational State Transfer) 란?

**REST**는 웹(HTTP)의 뼈대를 최대한 활용하기 위해 제안된 **'소프트웨어 아키텍처 스타일'**이다.

웹의 본질인 네트워크 상의 '자원(Resource)'을 주고받기 위해, 다음과 같은 철학을 기반으로 한다.
> "모든 자원에 고유한 **URI**를 부여하고, **HTTP 메서드**를 통해 해당 자원의 스냅샷(상태 및 표현)을 주고받자."

이러한 **REST 아키텍처 제약 조건을 철저히 준수한 시스템을 가리켜 'RESTful'하다고 칭한다.**

---

## 🛑 REST의 핵심 제약 조건

RESTful 체계를 유지하기 위해선 시스템이 다음 조건들을 충족시켜야 한다.

1.  **Client-Server (클라이언트-서버 구조)**: 사용자 인터페이스(UI) 렌더링 영역과 데이터 저장소(DB) 관할 영역을 완벽히 분리해야 한다.
2.  **Stateless (무상태성)**: 서버는 클라이언트의 상태(세션 등)를 메모리에서 직접 들고 있지 않는다. 클라이언트의 넘겨주는 토큰과 파라미터 요청 덩어리만으로 독립적 처리가 보장되어야 한다.
3.  **Cacheable (캐싱 처리)**: 클라이언트는 응답을 캐싱할 수 있어야 하며, 프로토콜 단에서 웹 인프라의 조건부 헤더 등을 이용해 리소스 절약을 꾀해야 한다.
4.  **Uniform Interface (인터페이스 일관성)**: 단연코 REST의 가장 핵심이다. URI로 식별된 자원을 특정 형태(JSON)로 조작하며, API 응답 메시지만 보고도 스스로 렌더링 상태를 해석할 수 있어야(Self-descriptive) 한다는 강력한 규제다.

---

## 🛠️ 실무 API 설계 규칙: 명사와 메서드의 분리

가장 실무에서 크게 부딪히는 부분은 Uniform Interface의 식별성, 즉 **URI 식별과 HTTP 메서드 조합**이었다.

### 규칙 1: URI(경로)는 행위 동사가 아닌 명사(대상 자원)만 허용한다
과거 무지했을 때는 `POST /createUser`, `POST /updatePost/1` 과 같이 행위를 URI에 하드코딩했다. 이는 통일성을 파괴하고 구조적 결함만 남겼다.
-   **개선된 방향**: `/users` (사용자 데이터 자원망 전체), `/users/123/posts` (123번 사용자가 소유한 게시물 자원) 형태로 명사(복수형 권장)만으로 경로를 세팅했다.

### 규칙 2: 행위(Action)는 오직 HTTP Method로만 지시한다
대상 자원이 `/users/123/posts` 라면, 이를 어떻게 지지고 볶을지는 HTTP 프로토콜의 표준 메서드가 전담하도록 관심사를 찢었다.

| HTTP Method | 자원 제어 역할 (CRUD)               | 멱등성 (Idempotent: 여러 번 쳐도 동일한가?)*  |
| :---------- | :---------------------------------- | :-------------------------------------------- |
| **GET**     | 대상 자원 데이터 조회 (Read)        | O (조회만 하므로 서버 상태 변화 X)            |
| **POST**    | 신규 자원 객체 생성 (Create)        | X (계속 새로운 자원이 중복 파생됨)            |
| **PUT**     | 대상 자원 통째 덮어쓰기 (Update)    | O (완전히 교체되므로 최종 결과는 언제나 동일) |
| **PATCH**   | 대상 자원 일부 속성만 수정 (Update) | - (구현 로직에 따라 상이함)                   |
| **DELETE**  | 대상 자원 삭제 (Delete)             | O (한 번 지우나 역으로 지우나 사라진 건 동일) |

## 💡 최종 회고 정리

수많은 프로젝트에서 REST API를 구축한다고 자부하지만, 엄밀한 의미(HATEOAS 등)를 완벽하게 충족하는 진정한 RESTful 생태계는 실무에서 찾아보기 극히 어려웠다. 대부분은 URI를 명사 자원으로 잘 설계하고, HTTP의 의미론을 규격에 맞춰 활용하는 **'HTTP 기반 인터페이스(HTTP API)'** 수준인 경우가 흔했다.

하지만 그 수준의 규격을 지키는 것만으로도, 백엔드와 클라이언트(프론트, 앱) 간의 협업 예측 가능성과 구조 생산성이 폭발적으로 상승한다는 것을 체감했다. **"API 주소 경로 모양만 봐도 무슨 데이터 타겟에 접근하는지 투명히 알 수 있고, HTTP 메서드명만 봐도 백엔드가 DB에 무슨 짓을 하려는지 굳이 묻지 않아도 알도록 직관을 일치시키는 것"**. 이것이 내가 체득한 REST 설계의 실무적 본질이다.

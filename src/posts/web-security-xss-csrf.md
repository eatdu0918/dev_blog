# 웹 보안의 기초: XSS와 CSRF의 공격 원리와 방어 전략

![웹 보안 XSS와 CSRF 도식](/public/images/web_security_xss_csrf.png)

웹 애플리케이션을 개발하면서 보안은 언제나 마지막에 챙기기 쉬운 요소였습니다. 하지만 보안은 사고가 발생했을 때 비로소 그 가치를 실감한다는 점에서 매우 중요함을 깨달았습니다. 

대표적인 웹 보안 취약점인 **XSS(Cross-Site Scripting)**와 **CSRF(Cross-Site Request Forgery)**의 공격 원리를 다시 학습하며, 개발 초기에 이를 방어하기 위해 어떤 설계가 필요한지 정리해 보았습니다.

---

## 1. 사용자의 권한을 훔치는 XSS (Cross-Site Scripting)

XSS는 공격자가 작성한 악성 스크립트가 사용자의 브라우저에서 실행되게 하여 쿠키나 세션 토큰을 탈취하는 방식입니다.

- **공격 경로**: 게시판 글, URL 파라미터 등 사용자의 입력값이 검증 없이 화면에 출력될 때 발생합니다. 
- **방어 핵심**: 모든 사용자의 입력은 신뢰하지 않는 것이 기본이었습니다. 
  - **이스케이프(Escaping)**: `<`, `>` 등 특수 문자를 HTML 엔티티로 치환하여 실행되지 않게 합니다. 
  - **CSP(Content Security Policy)**: 브라우저가 신뢰할 수 있는 출처의 스크립트만 실행하도록 정책을 세움으로써 추가적인 방어막을 칩니다.
  - **HttpOnly 쿠키**: 자바스크립트가 쿠키에 직접 접근하지 못하게 설정하는 것만으로도 대부분의 토큰 탈취를 막을 수 있음을 알게 되었습니다.

---

## 2. 사용자가 의도치 않게 요청하게 만드는 CSRF (Cross-Site Request Forgery)

CSRF는 사용자가 특정 사이트에 로그인된 상태에서, 공격자가 만든 악성 링크를 통해 브라우저가 서버로 원치 않는 요청을 보내게 만드는 공격입니다.

- **공격 경로**: 브라우저가 모든 요청에 자동으로 쿠키를 실어 보내는 특성을 악용합니다. 
- **방어 핵심**: 요청이 사용자의 의지에 의한 것인지 확인해야 합니다. 
  - **CSRF 토큰**: 서버에서 발급한 숨겨진 랜덤 토큰을 모든 상태 변경 요청(POST, PUT 등)에 실어 보내 서버에서 검증합니다. 
  - **SameSite 쿠키 속성**: 쿠키가 다른 사이트의 요청에 실려 갈 수 없도록 설정하는 것(Strict, Lax)이 현대 웹 보안에서 얼마나 강력한 도구인지 확인했습니다.

---

## 💡 마치며: 보안은 선택이 아닌 필수 요건

XSS와 CSRF를 학습하며 느낀 것은, 웹의 기본 동작 원리인 '스크립트 실행'과 '자동 쿠키 전송'이 그 자체로 보안상의 취약점이 될 수도 있다는 사실이었습니다. 

앞으로는 화려한 기능을 완성하는 것에만 몰두하기보다, 사용자의 입력을 철저히 필터링하고 쿠키와 세션의 보안 설정을 꼼꼼히 점검하는 습관을 들이려 합니다. 보안 사고를 예방하는 것은 기술적 완결성을 넘어, 우리 서비스를 믿고 사용하는 사용자에 대한 가장 기본적인 예의임을 마음속에 새겨봅니다.

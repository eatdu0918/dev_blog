# JVM 메모리 누수 진단과 해결: OutOfMemoryError 원인 파악을 위한 힙 덤프(Heap Dump) 분석 기법

자바를 주 언어로 사용하면서 내가 누렸던 가장 큰 호사는 '가비지 컬렉터(GC)'의 존재였다. 개발자가 `free()`나 메모리 해제를 직접 신경 쓰지 않아도 뒤에서 쓱싹쓱싹 청소를 해주는 우렁각시 같은 존재. 그래서 나는 "자바는 아무렇게나 짜도 메모리 누수가 없다"는 치기 어린 망상에 젖어있었다.

어느 금요일 오후, 내가 배포했던 운영 서버의 메모리 그래프가 이상했다. 평화롭던 곡선이 톱니바퀴처럼 요동치더니 허공을 찌른 채 멈춰버렸고, 가차 없이 에러 로그가 터졌다.
`java.lang.OutOfMemoryError: Java heap space`

과거의 나라면 두 눈을 질끈 감고 AWS 인스턴스의 '재시작(Restart)' 버튼을 눌러 임시방편으로 눈앞의 불만 껐을 것이다. 하지만 이제는 더 깊이 있는 개발자가 되어야 한다고 생각했고, 이번 주엔 반드시 이 문제를 근본적으로 파헤쳐서 "어디서 피가 새고 있는지" 원인을 밝혀내겠다고 내 자신과 약속했다.

---

## 1. GC가 지워주지 않는 객체의 함정에 빠지다

"자바인데 도대체 왜 가비지 컬렉터가 작동하지 않았지?"
답은 내가 작성했던 얄팍한 '전역 캐싱 로직'에 있었다.

나는 사용자의 특정 활동 로그를 외부 API로 한 번에 쏘기 위해, `Map` 구조의 싱글톤 객체에 꾸역꾸역 모아두고 배치로 보내는 로직을 짰다.
문제는 외부 API 장애가 났을 때, 나는 이 큐를 지워버리는(`clear` 하거나 `remove` 하는) 예외 처리를 어디에도 작성하지 않았던 거다.

GC는 '아무도 보지 않는 쓰레기'만 치운다. 하지만 내 코드 어딘가에선 싱글톤 `Map`이라는 컬렉션이 그 활동 로그 객체들을 계속 "내 것!"이라며 꽉 쥐고 참조의 끈을 놓지 않고 있었다. 이걸 곧 **논리적 누수(Logical Memory Leak)** 라고 부른다는 사실을 그제야 배웠다.

---

## 2. 현장 보존의 최우선: Heap Dump라는 검시관

OOM이 터진 후 재부팅하면 메모리 정보는 모두 허공으로 증발해버린다. 선배들의 노하우를 훔쳐보며, 운영 서버 시작 인자에 이 두 줄을 삽입하는 것의 엄청난 가치를 깨달았다.

```bash
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/logs/jvm-oom-dump.hprof
```

이 옵션을 넣어두고 다시 한번 사건이 재발했을 때, 내 서버는 죽어가면서 1GB 크기의 유언장(`.hprof` 파일 덤프)을 남겨주었다. 죽기 직전의 메모리 전체 상태가 스냅샷처럼 찍혀 나온 것이었다!

---

## 3. 떨리는 손으로 Eclipse MAT를 돌려보다

수 기가바이트의 덤프 파일을 단순 터미널 텍스트로 읽는 건 인간에게 불가능했다.
나는 `Eclipse MAT(Memory Analyzer Tool)` 라는 부검 도구를 처음 내 로컬 피씨에 다운로드했고 분석 리포트를 돌렸다.

### 1) 범인은 내 곁에 숨어있었다
"Leak Suspects(누수 의심자)" 버튼을 눌러 원형 차트를 보는 순간 소름이 돋았다.
`One instance of 'java.util.concurrent.ConcurrentHashMap' occupies 85.3% of the heap.`

아! 내 캐시 변수였다.
전역에 `static`으로 선언해둔 녀석 하나가 전체 램(RAM)의 85% 지분을 독식하고 있었던 거다.

### 2) Dominator Tree를 까뒤집다
그 안의 잔가지들을(`Dominator Tree`) 열어보니, 외부 API 통신 실패에 분노하여 밖으로 빠져나가지 못하고 차곡차곡 쌓여있던 내 사용자 행동 로그 DTO들이 300만 개나 버글거리고 있었다.
이걸 두 눈으로 목격하고 나서야 나는 에러 핸들링 구문에 `map.clear()` 한 줄을 조심스레 추가했다.

## 맺으며

OOM을 겪고 식은땀을 뻘뻘 흘리며 힙 덤프와 며칠을 씨름하던 그 기간.
아마도 내 개발자 인생에서 가장 하드코어하면서 짜릿한 성장통이 아니었을까 싶다.

재시작 버튼으로 도망가던 과거를 넘어, 이젠 `Eclipse MAT`를 띄워놓고 "어떤 변수가 참조를 안 뱉어내고 있나요?"라고 침착하게 분석할 수 있는 용기를 얻었다. 메모리 최적화란 결국 내가 짠 코드의 생명주기를 내가 온전히 끝장낼 수 있도록 책임지는 일. 아직 가야 할 길이 멀지만, 적어도 JVM 위에서 내가 지은 죄악을 마주할 힘이 생겼다.

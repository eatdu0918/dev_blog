# V8 엔진의 소스코드 실행 파이프라인 이해하기: Ignition과 TurboFan

![V8 엔진 파이프라인](/public/images/v8-engine-pipeline.png)

자바스크립트는 '인터프리터 언어'라고 흔히 배웁니다. 한 줄씩 읽어서 바로 실행하기 때문에 컴파일 과정이 필요 없고, 덕분에 생산성이 높다는 점이 가장 큰 특징이죠. 하지만 현대의 브라우저에서 돌아가는 자바스크립트의 놀라운 속도를 보며, 단순히 한 줄씩 읽는 것만으로 이 정도 성능이 가능한지 의구심이 들 때가 있었습니다. 

엔진의 동작 방식에 대해 찾아보며, 저는 제가 알고 있던 '인터프리터'라는 개념이 현대의 복잡한 런타임 환경에서는 얼마나 일부분에 불과했는지 깨닫게 되었습니다. 오늘은 자바스크립트 소스코드가 어떻게 구조화된 트리(AST)를 거쳐 바이트코드(Bytecode)가 되고, 마침내 최적화된 기계어(Machine Code)로 변모하는지 그 여정을 추적해 보았습니다.

---

## 1. 텍스트에서 구조로: 파싱(Parsing)의 세계

가장 먼저 맞이하는 단계는 우리가 짠 텍스트 파일(코드)을 엔진이 이해할 수 있는 구조로 바꾸는 과정이었습니다.

- **토크나이징(Tokenizing)**: 소스코드를 `let`, `x`, `=`, `10` 같이 의미 있는 최소 단위인 '토큰'으로 쪼개는 것으로 시작합니다.
- **파싱(Parsing)**: 이 토큰들을 문법적으로 분석하여 거대한 나무 구조인 **AST(Abstract Syntax Tree, 추상 구문 트리)**를 만드는 단계입니다. 엔진이 비로소 코드의 '의미'를 파악하고 논리적 오류를 검사하는 첫 관문임을 알게 되었습니다.

---

## 2. Ignition 인터프리터를 통한 바이트코드 실행

과거의 V8은 곧바로 기계어로 컴파일했지만, 메모리 소모가 심하고 초기 실행 속도가 늦다는 한계가 있었다고 합니다. 이를 해결하기 위해 등장한 것이 바로 **Ignition(이그니션)** 인터프리터라는 사실을 새로 배웠습니다.

- **바이트코드(Bytecode) 생성**: AST를 직접 실행하는 대신, '바이트코드'라는 중간 단계 언어로 변환합니다. 이는 기계어보다는 추상적이지만 소스코드보다는 실행에 최적화된 형태입니다.
- **실행(Hotness Check)**: Ignition은 바이트코드를 한 줄씩 실행하면서 동시에 어떤 함수가 자주 불리는지(Hot Function) 감시합니다. 여기서 'JIT(Just-In-Time) 컴파일'의 진가가 발휘되기 시작함을 느꼈습니다.

---

## 3. TurboFan을 통한 기계어 최적화

자주 실행되는 코드가 발견되면, V8은 이를 더 빠르게 만들기 위해 '최적화 컴파일러'인 **TurboFan(터보팬)**에게 일을 넘깁니다.

- **JIT 컴파일**: 바이트코드를 실제 CPU가 즉각 실행할 수 있는 **최적화된 기계어(Machine Code)**로 직접 번역해버립니다. 
- **추측(Speculation)**: 이 단계에서 가장 흥미로운 점은 V8이 '타입'을 과감하게 추측한다는 것이었습니다. "이 함수는 계속 숫자만 들어왔으니, 앞으로도 숫자만 들어올 거야!"라고 가정하고 불필요한 타입 체크를 생략한 극도로 최적화된 기계어를 만듭니다.

---

## 4. 성능의 함정: 최적화 해제(Deoptimization)

V8의 추측이 빗나갔을 때 발생하는 'Deoptimization'은 저에게 큰 경각심을 주었습니다.

- **현상**: 숫자만 받던 함수에 갑자기 문자열이 들어오면, TurboFan이 만든 최적화 기계어는 무용지물이 됩니다. 
- **복구**: 엔진은 실행 중이던 기계어를 즉시 폐기하고, 다시 안전한 바이트코드(Ignition) 상태로 돌아갑니다. 
이 과정이 반복되는 상황을 보며, 왜 자바스크립트에서 객체의 모양(Shape)을 일정하게 유지하거나 타입 시스템(`TypeScript`)을 활용하는 것이 엔진의 최적화 효율에 직접적인 기여를 하는지 비로소 이해할 수 있었습니다.

---

## 💡 코드로 느껴보는 타입의 일관성

V8이 내부적으로 객체의 구조를 캐싱할 때 도움을 주는 'Hidden Class' 개념을 염두에 두고, 어떻게 코드를 짜는 것이 최적화에 유리할지 생각하며 예제를 구성해 보았습니다.

```typescript
// src/examples/v8-optimization.ts

/**
 * 객체의 구조(Hidden Class)가 고정될 때 V8은 최적화 효율을 극대화한다.
 */
class Point {
    constructor(public x: number, public y: number) {}
}

export function performHeavyCalculation() {
    const points = [];
    // 1. 일관된 타입/모양의 객체 생성
    for (let i = 0; i < 10000; i++) {
        points.push(new Point(i, i + 1));
    }

    // 2. 동적으로 속성을 추가하거나 모양을 바꾸는 행위를 자제하면 
    // V8의 최적화(TurboFan) 파이프라인을 온전히 타게 됨을 배웠다.

    return points.reduce((acc, p) => acc + p.x + p.y, 0);
}
```

```typescript
// test/v8-optimization.test.ts
import { describe, it, expect } from 'vitest';
import { performHeavyCalculation } from '../src/examples/v8-optimization';

describe('V8 엔진 동작 원리 학습', () => {
    it('일관된 객체 구조를 활용한 연산이 성공적으로 수행되는지 검증한다', () => {
        const result = performHeavyCalculation();
        expect(result).toBeGreaterThan(0);
        // TurboFan 최적화가 이 코드를 어떻게 처리할지 상상하며 결과를 확인해 봅니다.
    });
});
```

## 💡 마치며: 성능을 고려하는 코드 작성의 필요성

자바스크립트가 내부적으로 얼마나 복잡한 최적화 과정을 거쳐 실행되는지 학습하며, 단순히 '동작하는 코드' 이상의 고민이 필요함을 느꼈습니다. 특히 V8 엔진이 타입을 추측하고 최적화 기계어를 생성하는 파이프라인을 이해하고 나니, 객체의 구조를 일정하게 유지하는 것과 같은 습관이 실제 성능에 어떤 영향을 주는지 구체적으로 그려볼 수 있었습니다.

추상화된 고수준 언어를 사용하고 있지만, 그 이면에서 동작하는 엔진의 최적화 원리를 이해하는 것은 더 견고한 애플리케이션을 만드는 밑바탕이 된다고 생각합니다. 앞으로는 엔진의 최적화 효율을 저해하지 않는 코드를 작성하기 위해 더 세심하게 고민해 보려 합니다.

## 핵심 요약

상태를 **서버 상태(Server State)**와 **클라이언트 상태(Client State)**로 나누는 게 출발점입니다. 둘을 한 통에 담으려고 하면 어떤 도구를 써도 복잡해집니다.

## 두 가지 상태의 본질적 차이

- **서버 상태**: 원본이 서버에 있고, 가져온 시점부터 stale. 캐싱, 무효화, 재시도, 동기화가 본질.
- **클라이언트 상태**: 원본이 브라우저에 있고, 컴포넌트 트리 안에서 살아있는 동안만 유효. UI 상태, 폼 상태, 모달, 토스트 등.

이 분리를 안 하면 Redux에 fetch 코드가 잔뜩 쌓이고, 캐시 무효화는 수동으로, 로딩 상태는 액션으로 관리하는 흔한 사고가 납니다.

## 제 디폴트 조합

저는 신규 React 프로젝트에 다음 조합을 디폴트로 둡니다.

- **서버 상태 → Tanstack Query**: 캐싱, 백그라운드 refetch, mutation invalidation이 표준화되어 있어 손으로 짤 일이 없습니다.
- **클라이언트 상태 → Zustand 또는 Jotai**: 보일러플레이트가 거의 없고, slice 단위로 작게 시작해 점진적으로 키워갈 수 있습니다.
- **폼 상태 → React Hook Form + Zod**: 리렌더 비용이 작고, 스키마 기반 검증이 깔끔합니다.

Redux Toolkit이 나쁘다는 게 아니라, **대부분의 프로덕트에서 과합니다**. 미들웨어, devtools, time-travel 같은 기능이 정말 필요한 시점이 오기 전에는 도구의 무게에 끌려갑니다.

## 어떤 경우에 Redux를 다시 고르는가

- 매우 큰 SPA에서 시간 여행 디버깅이 실제로 도움이 되는 경우.
- 복잡한 워크플로우 상태(예: 그래프 에디터, 캔버스)에서 모든 액션을 직렬화/리플레이해야 하는 경우.
- 팀이 이미 Redux에 깊이 익숙하고 다른 라이브러리 학습 비용이 더 큰 경우.

도구 선택은 **팀의 학습 곡선과 운영 비용**도 같은 무게로 봐야 합니다.

## 자주 묻는 디테일

- **"전역 상태에 무엇을 둬야 하나요?"** → 여러 라우트나 트리 전반에서 필요한 것만. "props drilling이 귀찮아서"는 잘못된 이유입니다. 컴포지션이나 Context로 푸는 편이 가벼울 때가 많습니다.
- **"Tanstack Query 캐시도 전역 상태인가요?"** → 그렇게 다뤄도 됩니다. URL이나 쿼리 파라미터로 표현 가능한 상태(검색어, 필터)는 URL을 single source of truth로 두고 Tanstack Query가 그걸 키로 캐시하게 만들면 새로고침/공유에도 강해집니다.
- **"낙관적 업데이트는 어디서 하나요?"** → Tanstack Query의 `onMutate`로. 실패 시 `onError`에서 롤백하는 패턴.

## 면접 follow-up

- "Context API는 안 쓰시나요?" → 거의 변하지 않는 값(테마, i18n)에는 적합. 자주 변하는 상태에 두면 트리 전체 리렌더가 발생.
- "SSR과 함께 쓸 때 주의점은?" → Tanstack Query의 hydration 패턴, Zustand의 server snapshot 처리, 직렬화되지 않는 값 주의.
- "RSC 환경에서는?" → 서버 컴포넌트는 상태가 없으니, 클라이언트 컴포넌트로 경계를 그어 그 안에서만 상태 라이브러리를 사용. 가능한 한 데이터 fetching은 서버로 옮기고 클라이언트 상태를 줄이는 방향이 트렌드.

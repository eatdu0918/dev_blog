# MSA 로컬 인프라 구동의 벽: Docker Compose 도입기

마이크로서비스 아키텍처(MSA) 학습 프로젝트의 백엔드 코드를 분리 작성하는 것 이상으로 제 진을 뺐던 부분은 **"이 복잡하고 파편화된 환경을 로컬 PC 영역 수준에서 통합 테스트하고 빌드/구동할 것인가"** 하는 인프라 운영 문제였습니다.

여러 개의 스프링 부트 포트와 이를 백그라운드에서 뒷받침하는 기술 스택들을 수동 런타임으로 통제하려다 한계를 맞고, **Docker Compose**를 도입해 개발 생산성을 확보했던 과정을 적어봅니다.

---

## 🚨 수동 구동의 한계와 비효율성

프로젝트 시스템 파이프의 근간을 이루는 인프라 베이스 요소는 총 6가지가 넘었습니다. 
- MySQL 인스턴스 (각 서비스별 완전 런타임 격리를 위해 다수 분할 필요)
- Redis 컨테이너망
- Kafka & Zookeeper (메시징 브로커 데몬)
- 그 밖의 서버 스택 라인

초반부엔 인텔리제이를 무작정 열어 스프링 애플리케이션을 하나씩 수동으로 Run을 누르고, 백엔드에는 직접 DB 프로그램들을 설치하여 파이프라인 실행 명령어를 쳤습니다. 로컬 장비 세팅과 싱크로나이즈에 매일 십 분 이상의 시간이 허투루 지연되었고, 바인딩 포트 충돌이라도 터지면 터미널 창 수십 개를 탭하며 원인 로그 메시지가 뜬 구멍을 찾아 헤매는 소모적인 일상이 다람쥐 쳇바퀴 돌듯 반복되었습니다.

---

## 🏗️ 개선: 단일 스크립트 도커 컴포즈 작성

단일 야믈 스크립트 파일명세만으로 연결된 모든 로컬 환경망을 일괄 정의하고 내릴수 있다는 개발자 커뮤니티의 조언에 따라 `docker-compose.yml` 스크립트 환경 정립을 시작하게 되었습니다. 다수의 독립적인 Docker 컨테이너들을 한 통의 묶음 생태계로 바인딩하고 가상 로컬 통신 허브망을 깔아주는 본질적인 연결 관리 도구였습니다.

### 1. 실행 순서 보장하기 (Depends On 체인 구조)
도입기 가장 애를 먹었던 부분은 **서비스 간의 순차적 기동 순서**를 강제하는 것이었습니다. 스프링 기반의 서버 코드가 가장 먼저 무겁게 올라와 구동을 마쳤는데 정작 통신 타겟인 DB 컨테이너 블록이 아직 포트를 활성화하지 못한 지연 상태라면 서버 엔진이 곧장 DB 커넥션 셧다운 예외를 터뜨리고 시스템이 아예 죽어버리는 패닉이 일어났었습니다.

이를 해결하기 위해 타깃 컨테이너의 내부망 기동 상태를 점검하는 Ping 헬스체크 정책을 강제 스크립트로 반영했습니다.
```yaml
services:
  order-mysql:
    image: mysql:8.0
    # ... 
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      retries: 5

  order-service:
    # ...
    depends_on:
      order-mysql:
        condition: service_healthy # 컨테이너 내부 통신이 가능한 건강한 활성 상태까지 대기 걸림
```

### 2. 가상 네트워크 도메인 연결 설정
도커 컴포즈가 짜놓은 가이드 내부 생태계 통신망으로 진입하고 나니, 이전 환경처럼 IP 주소의 하드코딩 배열이나 지루한 `localhost` 고정 바인딩 타겟망에 집착할 필요성이 사라졌습니다. 
가령 환경세팅 안에 `spring.datasource.url=jdbc:mysql://order-mysql:3306/` 형태처럼 설정 파일 내의 서비스명칭 문자열만으로 내부 컨테이너의 도메인 라우팅 아이피가 알아서 탐색 매핑되어 인스턴스끼리의 로컬 통신 테스트 단계가 크게 매끄러워졌습니다.

## 💡 회고

명령어 단 한 줄(`docker-compose up -d`)만 치팅 입력하면 수십 개의 인프라가 엮여 규격화된 상태로 깔끔하게 통합 구동되는 것을 목격했을 때 개발의 피로도가 크게 절감되었습니다. 

시스템적으로 마이크로서비스는 코드 덩어리뿐만 아니라 주변을 서포팅하는 인프라 요건 체계가 막대하게 늘어나는 아키텍처 특성이라는 점, 그리고 이를 로컬 개발자 환경에서 균등하게 가상 복제하고 띄워 올릴 수 있는 인프라 제어 통제권(Dx) 확보 지식이 선행되지 못하면 개발 본연업인 비즈니스 로직 작성 파이프 유지보수 조차가 극도로 고통스러워진다는 구조적 시스템 관점의 교훈을 확실히 남길 수 있었습니다.

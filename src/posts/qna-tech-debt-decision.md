## Q1. 기술 부채를 어떻게 분류하시나요?

**A.** **Martin Fowler의 4사분면**으로 나눕니다.

| | 의도적 | 비의도적 |
|---|---|---|
| **신중함** | "지금 출시 우선, 다음에 갚자" | "더 좋은 방법이 있었구나" |
| **무모함** | "테스트? 시간 없어" | "코드 안 보고 짜다 부채가 생김" |

면접에서는 **신중한 의도적 부채**를 어떻게 가시화하고 갚는지를 중심으로 답하는 게 깊이가 드러납니다.

---

## Q2. 리팩토링 우선순위는 어떻게 매기시나요?

**A.** 5가지 기준으로 줄을 세웁니다.

1. **변화 빈도**: 자주 건드리는 코드의 부채가 가장 비쌈. `git log`로 측정.
2. **결함률**: 버그가 자주 나는 모듈. 이슈 트래커로 매핑.
3. **온보딩 마찰**: 새 멤버가 헤매는 영역.
4. **장애 영향도**: 결제/인증 같은 핵심 경로.
5. **변경 비용 추세**: 한 기능 추가에 드는 시간이 늘어나면 부채 누적 신호.

"감"이 아니라 **데이터로 우선순위**를 정하는 게 시니어의 역량입니다.

## Q3. 부채만 갚는 스프린트가 가능한가요?

**A.** 비즈니스 설득이 어렵습니다. **기능 개발과 섞는 4가지 전략**이 현실적입니다.

- **Boy Scout Rule**: 작업한 영역은 살짝 더 깨끗하게. 점진 개선.
- **기능 PR 전에 리팩토링 PR 선행**: 깔끔히 끼워 넣기 위한 사전 정리. PR 분리로 리뷰 부담 감소.
- **20% 룰**: 매 스프린트 일정 비율을 부채 상환에 고정.
- **부채 백로그**: 발견 즉시 티켓 가시화. 망각 방지.

## Q4. PM이 기능을 우선한다면 어떻게 설득하시나요?

**A.** **비즈니스 언어로 비용/효과 환산**합니다.

좋은 표현:
- "이 모듈에 새 기능 추가가 평균 3일에서 7일로 늘어났습니다. 한 번에 5일 정리하면 다음 4건이 6일 절약됩니다."
- "최근 장애 3건 중 2건이 이 영역에서 발생했습니다."

추정이 어렵다면 **타임박스 실험** 제안: "1주 정리해 보고 다음 PR의 lead time을 측정하겠다."

## Q5. 리팩토링이 위험한 순간은 언제인가요?

**A.** 3가지 상황입니다.

- **테스트가 없는 영역**: 회귀 테스트(Characterization Test)부터 만들고 시작.
- **출시 직전**: 동결 기간엔 표면 변경 금지. 작은 PR만 머지.
- **요구사항이 곧 바뀔 영역**: 어차피 다시 쓸 코드를 정리하는 건 낭비.

이 셋 중 하나라도 해당하면 리팩토링을 미루거나 범위를 좁히는 게 안전합니다.

---

## Q6. 테스트 없는 레거시는 어떻게 리팩토링하시나요?

**A.** **특성 테스트(Characterization Test)** 로 현재 동작을 먼저 잠급니다.

순서:
1. 현재 동작을 그대로 검증하는 테스트 작성(올바른 동작이 아니라 **현재 동작**).
2. 작은 단위로 메서드/클래스 추출.
3. 재배치하고 의존성 정리.
4. 새 동작에 맞춰 테스트 보강.

핵심은 **"리팩토링 전에 변경 안전망부터"**입니다.

## Q7. "그냥 갈아엎기"는 왜 위험한가요?

**A.** **기존 시스템의 모든 엣지 케이스를 새 시스템이 모를 가능성**이 큽니다.

흔한 함정:
- 새 시스템 구축에 6개월, 운영 검증에 1년.
- 그 사이 기존 시스템에 새 기능이 추가되어 격차 확대.
- 결국 빅뱅 이전 실패 → 두 시스템 동시 운영의 부담.

**Strangler Fig 패턴**이 정공법입니다: 기존 위에 점진적으로 새 코드를 얹고 트래픽을 단계적으로 옮김.

## Q8. 안전한 마이그레이션은 어떻게 진행하시나요?

**A.** 4가지 도구를 같이 씁니다.

1. **Feature flag**: 새 코드 경로를 플래그로 켜고 끔.
2. **Dual-write**: 두 시스템에 동시 쓰기 → 데이터 동기화 검증.
3. **단계적 트래픽 이전**: 1% → 10% → 50% → 100%.
4. **롤백 경로 사전 정의**: 문제 발견 즉시 되돌릴 수 있는 절차.

이 패턴으로 가면 위험을 잘게 쪼개 시스템 안정성을 유지하면서 옮길 수 있습니다.

## Q9. 기술 부채와 나쁜 코드는 같은 건가요?

**A.** **다릅니다**. 부채는 의도적 트레이드오프가 가능한 자원입니다.

- 출시 데드라인 맞추려고 단순한 구조로 가는 것 = **합리적 부채**.
- 단순히 신경 안 써서 더러워지는 것 = **나쁜 코드**.

YAGNI 위반(너무 일찍 추상화)도 부채의 흔한 원인입니다. **부채를 만들지 않는 코드 = 가장 단순한 코드**가 종종 답입니다.

---

## Q10. 부채 상환의 효과를 어떻게 측정하시나요?

**A.** 직접 ROI는 어렵고, **간접 지표**를 봅니다.

- **Lead time**: 새 기능 추가에 드는 시간 추세.
- **결함 비율**: 해당 모듈의 버그 발생 빈도.
- **온보딩 시간**: 새 멤버가 코드 이해까지 걸리는 시간.
- **PR 사이즈**: 같은 변경에 필요한 라인 수.

리팩토링 전후로 이 지표들의 변화를 측정하면 비즈니스에도 설득력 있는 데이터가 됩니다.

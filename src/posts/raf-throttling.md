# requestAnimationFrame(rAF) 기반 스로틀링: 부드러운 스크롤 애니메이션 최적화 경험

과거 프로모션 페이지 개발 중 스크롤바 위치에 비례하여 DOM 요소들이 모션과 함께 나타나고 사라지는 시각 효과(Parallax)를 구현한 적이 있습니다. 이벤트 과부하를 막기 위해 일전에 적용해 보았던 `setTimeout` 기반 스로틀링(Throttling)을 걸어두었는데, 결과적으로 화면 요소가 움직일 때 미세하게 툭툭 끊기는(Janky) 어색한 느낌을 지울 수 없었습니다.

이 불쾌한 경험의 원인을 분석하며 일반적인 시간 기반 스로틀링의 한계와, 브라우저 렌더링 프레임의 관계를 깨닫고 아키텍처를 개선한 과정을 남깁니다.

---

## 🚫 타이머 기반 스로틀링이 초래한 렌더링 한계

브라우저는 통상적으로 1초에 약 60회(60Hz, 16.6ms 간격) 화면 레이아웃을 재구성하여 모니터에 페인트해 냅니다(Frame Rate).

스크롤 이벤트 성능 제어를 위해 스로틀링 호출 한계치를 임의의 `50ms` 타이머로 강제 적용해 둔 것이 원인이었습니다. 
이 타이머 구동 방식은 모니터의 화면 갱신 주기와는 조금의 연관도 없이 자바스크립트 스레드를 독자적으로 실행합니다. 따라서 스크롤 위치 재계산 연산이 브라우저의 렌더링 준비 시간과 엇박자로 매칭될 경우 페인팅되지 못하고 중간 프레임이 대거 폐기(Frame Drop)되면서 미세하게 툭툭 프레임 끊어짐 현상이 발생했던 것입니다.

---

## ✨ 해결: 프레임 갱신 주기에 맞춘 rAF 스로틀링 도입

이 렌더링 간격 불일치 문제를 통제하기 위해, 고정된 시간(`ms`) 구속을 과감히 포기하고 브라우저의 화면 그리기 타이밍에 전적으로 의존하는 `requestAnimationFrame` API를 스로틀링의 기준으로 교체했습니다.

### rAF 스로틀링 구현 전략

**"어떠한 이벤트 트리거가 아무리 쏟아져 들어오더라도, 브라우저가 허용하는 1 프레임(약 16.6ms) 당 오직 한 번의 시각 반영 액션만 다음 렌더링 큐에 등록한다"**는 메커니즘을 적용했습니다.

```javascript
function rafThrottle(callback) {
  let isTicking = false; // 현재 렌더링 큐 예약 여부 플래그
  
  return function(...args) {
    if (!isTicking) {
      isTicking = true; // 무중복 작업 예약 상태로 잠금
      
      requestAnimationFrame(() => {
        // 다음 화면 렌더링 사이클(리페인트) 업데이트 직전에 콜백 실행 보장
        callback.apply(this, args);
        
        isTicking = false; // 프레임 반영 완료 후 다음 허용을 위해 대기
      });
    }
  };
}

// 스크롤 이벤트에 커스텀 rAF 제어 파이프 연동
const handleScroll = rafThrottle(() => {
  const currentScrollY = window.scrollY;
  // 패럴랙스 등 시각 요소 좌표 업데이트 컴포넌트 로직 구성
});

window.addEventListener('scroll', handleScroll);
```

이러한 로직 구조로 전환한 결과, 스크롤링 위치에 비례하여 시각 요소의 CSS 픽셀 좌표값을 할당하는 타이밍이 브라우저의 페인팅 공정 파이프라인과 완벽하게 동기화되었고 시야가 극도로 부드러워진 모션(60FPS)을 관찰할 수 있었습니다.

## 💡 회고

본인의 시행착오 경험을 바탕으로, 요구사항에 맞는 성능 최적화 도구를 적재적소에 분리 채택하는 기준을 세웠습니다.

- **setTimeout 기반 스로틀링**: 데이터 패칭 로직(API 통신), 특정 분기 연산 등 DOM 엘리먼트의 직접적인 시각적 렌더링 동기화 여부가 중요하지 않은 백그라운드 모델 컨트롤.
- **rAF 기반 스로틀링**: 무한 스크롤, 실시간 DOM 객체의 위치 재조정, CSS 스타일의 프레임 변환이 동반되어 사용자 눈에 직접적으로 도출되는 시각 처리 모델 컨트롤.

프론트엔드 환경에서는 로직이 단순히 '오류 없이 잘 작동하는가'를 넘어서 아키텍처의 채택 설계 방식이 사용자 체감 품질(UX 지표)에 극명히 작용한다는 사실을 피부로 느낄 수 있었던 사례였습니다.

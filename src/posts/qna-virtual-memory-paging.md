## 핵심 요약

- **가상 메모리**: 각 프로세스가 보는 **연속된 메모리 환상**. 실제로는 흩어진 물리 메모리에 매핑.
- **페이징**: 가상 메모리를 고정 크기 페이지(보통 4KB)로 나눠 물리 프레임에 매핑.
- **MMU + 페이지 테이블**이 가상→물리 변환 담당. **TLB**가 변환을 캐시.

## 왜 가상 메모리?

1. **격리**: 한 프로세스가 다른 프로세스 메모리 못 봄.
2. **단순성**: 모든 프로세스가 0번지부터 보이는 깔끔한 주소 공간.
3. **물리 메모리 효율**: 사용하는 페이지만 실제로 할당.
4. **swap**: 사용 안 하는 페이지를 디스크로 옮겨 큰 메모리처럼 사용.

## 페이지 테이블

- 가상 페이지 번호 → 물리 프레임 번호 매핑 + 권한(읽기/쓰기/실행).
- 64bit 주소 공간 전체를 평면 테이블로 두면 너무 큼 → **다단계 페이지 테이블**(x86-64는 4단계).

## TLB (Translation Lookaside Buffer)

- MMU의 캐시. 최근 변환 결과 저장.
- TLB hit = 빠름. miss = 페이지 테이블 워크.
- 컨텍스트 스위칭 시 보통 flush — 프로세스 간 스위칭이 비싼 이유 중 하나(스레드 간은 같은 주소 공간이라 flush 없음).

## Page Fault

가상 페이지에 접근했는데 물리 메모리에 없을 때:
1. CPU가 page fault 인터럽트.
2. OS가 처리:
   - **Minor fault**: 메모리에는 있지만 매핑 안 됨(공유 라이브러리 등) → 빠른 매핑.
   - **Major fault**: 디스크에서 읽어야 함 → 느림(ms 단위).
3. 매핑 후 재시도.

## Demand Paging

프로그램 시작 시 모든 페이지 로드 X. **접근하는 페이지만 로드** → 시작 빠름.

`fork` + `exec` 시 **Copy-On-Write**: 자식이 같은 페이지 읽기로 공유 → 쓰기 시 복사. 메모리 절약.

## 페이지 교체 알고리즘

물리 메모리 부족 시 어떤 페이지를 swap out?

- **LRU(Least Recently Used)**: 최근 안 쓴 것. 이상적이지만 구현 비용.
- **Clock(LRU 근사)**: 참조 비트 + 원형 큐. 실용적.
- **LFU**: 사용 빈도 적은 것.
- **FIFO**: 단순. **Belady 이상 현상**(메모리 늘렸는데 fault 증가) 가능.

## 세그멘테이션 vs 페이징

- **세그먼트**: 가변 크기, 의미 단위(코드/스택/힙). 외부 단편화.
- **페이지**: 고정 크기. 내부 단편화 가능, 외부 X.

현대는 페이징 위주. x86은 둘 다 지원하지만 OS가 사실상 페이징만 사용.

## 메모리 영역

- **Text(Code)**: 실행 코드. 읽기 전용.
- **Data**: 초기화된 전역 변수.
- **BSS**: 미초기화 전역.
- **Heap**: 동적 할당. 위로 증가.
- **Stack**: 함수 호출. 아래로 증가.

스택과 힙이 충돌 안 하게 가상 주소 공간 가운데에 거대한 빈 공간.

## mmap

파일을 가상 메모리에 매핑 → 파일 I/O를 메모리 접근처럼.
- 큰 파일 부분 읽기에 유리.
- 데이터베이스/검색엔진 내부에서 광범위 사용.
- 단점: page fault 처리가 일반 read보다 예측 어려움.

## Thrashing

대부분 시간을 page fault 처리에 씀 → CPU 활용률 급락.
- 메모리 부족 + 워킹셋이 안 맞을 때.
- 해결: 메모리 추가, 워킹셋 줄이기, 프로세스 수 줄이기.

## 실무에서 만나는 곳

- **컨테이너 메모리 한도** + JVM heap이 안 맞으면 OOM Killer 발동.
- **Redis maxmemory** + page cache가 디스크 I/O 영향.
- **PostgreSQL의 shared_buffers** vs OS page cache 균형.
- **/proc/meminfo**, `vmstat`로 swap, page fault, 캐시 모니터링.

## 자주 헷갈리는 디테일

- "32bit OS는 4GB까지만"은 가상 주소 공간 한계. PAE로 물리는 더.
- 가상 메모리 = swap이 아님. 가상 메모리는 추상화, swap은 그 일부 메커니즘.
- huge page(2MB/1GB)는 TLB miss를 줄임 — DB/JVM이 이득.

## 면접 follow-up

- "프로세스 메모리 누수 디버깅?" → RSS 추세 모니터링 + heap dump + valgrind/jemalloc 프로파일러.
- "fork가 큰 프로세스에서도 빠른 이유?" → COW로 페이지 테이블만 복사 + 쓰기 시점에 실제 복사.
- "swap이 켜진 서버 vs 꺼진 서버 차이?" → 켜지면 OOM 회피 가능하지만 swap thrashing 시 응답 폭발. DB는 보통 끔.

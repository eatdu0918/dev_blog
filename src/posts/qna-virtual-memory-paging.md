## Q1. 가상 메모리는 왜 필요한가요?

**A.** 4가지 이유입니다.

1. **격리**: 프로세스 간 메모리 보호.
2. **단순성**: 모든 프로세스가 0번지부터 보이는 깔끔한 주소 공간.
3. **물리 메모리 효율**: 사용하는 페이지만 실제 할당.
4. **swap**: 안 쓰는 페이지를 디스크로 옮겨 큰 메모리처럼 사용.

각 프로세스는 자기만의 가상 주소 공간을 가지고, OS와 MMU가 물리 메모리로 변환해줍니다.

---

## Q2. 페이징의 동작 원리를 설명해 주세요.

**A.** 가상 메모리를 **고정 크기 페이지**(보통 4KB)로 나눠 물리 프레임에 매핑합니다.

- **페이지 테이블**: 가상 페이지 번호 → 물리 프레임 번호 + 권한.
- **MMU**: 메모리 접근 시마다 페이지 테이블을 참조해 변환.
- **TLB**: MMU의 캐시. 최근 변환 결과 저장.

64bit 주소 공간 전체를 평면 테이블로 두면 너무 커서 **다단계 페이지 테이블**(x86-64는 4단계)을 사용합니다.

---

## Q3. TLB는 무엇이고 왜 중요한가요?

**A.** **Translation Lookaside Buffer**: MMU의 가상→물리 변환 캐시.

- TLB hit = 빠름.
- TLB miss = 페이지 테이블 워크 → 다단계 트리 탐색.
- **컨텍스트 스위칭 시 flush** 되어 프로세스 간 스위칭이 비싼 큰 이유.

스레드 간 스위칭은 같은 주소 공간이라 TLB flush가 없어 훨씬 저렴합니다.

---

## Q4. Page Fault는 무엇이고 종류가 어떻게 되나요?

**A.** **가상 페이지 접근 시 물리 메모리에 없을 때 발생하는 인터럽트**.

- **Minor fault**: 메모리엔 있지만 매핑 안 됨(공유 라이브러리 등) → 빠른 매핑.
- **Major fault**: 디스크에서 읽어야 함 → 느림(ms 단위).
- **Invalid fault**: 권한 없는 영역 접근 → segfault.

Major fault가 자주 발생하면 swap thrashing → 응답 폭발의 원인입니다.

---

## Q5. Copy-On-Write는 어떻게 동작하나요?

**A.** **fork 시 즉시 복사 안 하고 쓰기 시점까지 미루는** 최적화입니다.

```
fork() 호출
↓
부모와 자식이 같은 물리 페이지를 읽기로 공유
↓
한쪽이 쓰기 시도 → page fault → 그 페이지만 복사
```

큰 프로세스를 fork해도 페이지 테이블만 복사하고 실제 메모리는 거의 안 늘어납니다. Redis BGSAVE, Postgres fork 모델이 이 덕분에 빠릅니다.

---

## Q6. 페이지 교체 알고리즘은 어떤 게 있나요?

**A.** 물리 메모리 부족 시 어떤 페이지를 swap out할지.

- **LRU**: 최근 안 쓴 것. 이상적이지만 구현 비용.
- **Clock**(LRU 근사): 참조 비트 + 원형 큐. 실용적, 실제 OS가 자주 사용.
- **LFU**: 사용 빈도 적은 것.
- **FIFO**: 단순. **Belady 이상 현상**(메모리 늘렸는데 fault 증가) 가능.

Linux는 active/inactive 두 LRU 리스트를 유지하는 변종을 씁니다.

---

## Q7. 세그멘테이션과 페이징은 어떻게 다른가요?

**A.**
- **세그먼트**: 가변 크기, 의미 단위(코드/스택/힙). **외부 단편화** 발생.
- **페이지**: 고정 크기. **내부 단편화**(페이지 끝 남는 공간)는 있지만 외부 단편화 없음.

현대는 페이징이 표준. x86은 둘 다 지원하지만 OS가 사실상 페이징만 사용합니다.

---

## Q8. mmap은 무엇이고 언제 쓰이나요?

**A.** **파일을 가상 메모리에 매핑**해 메모리 접근처럼 파일 I/O를 하는 시스템콜.

장점:
- 큰 파일의 일부만 읽기에 유리.
- 운영체제 page cache와 자연스럽게 통합.

활용: **데이터베이스, 검색엔진, 로그 처리** 내부에서 광범위 사용. SQLite, MongoDB의 WiredTiger, Lucene 등.

단점: page fault 처리가 일반 read보다 예측 어려워 실시간성 요구에 주의.

---

## Q9. Thrashing은 무엇이고 어떻게 막나요?

**A.** **대부분 시간을 page fault 처리에 쓰는 상태** → CPU 활용률 급락.

원인:
- 메모리 부족.
- 워킹셋(자주 쓰이는 페이지 집합)이 물리 메모리를 초과.

해결:
- 메모리 추가.
- 프로세스 수 줄이기.
- 워킹셋 축소(쿼리 패턴 개선, 캐시 정리).
- 일부 프로세스를 swap out 후 메모리 회복.

## Q10. 컨테이너/JVM 환경에서 가상 메모리 관련 함정은?

**A.** 4가지가 자주 보입니다.

- **컨테이너 메모리 한도 + JVM heap 미스매치**: heap이 한도를 넘으면 OOM Killer가 컨테이너를 죽임. `MaxRAMPercentage` 설정.
- **swap 정책**: DB 서버는 swap을 보통 끔(`swappiness=0~10`). swap 시 응답 폭발.
- **huge page**: 2MB/1GB 페이지로 TLB miss 감소. DB/JVM이 이득.
- **Linux page cache**: free 명령의 "available"이 진짜 가용. "free"만 보면 오해.

`/proc/meminfo`, `vmstat`로 swap/page fault/캐시를 모니터링하는 습관이 운영의 출발점입니다.

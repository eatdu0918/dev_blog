## Q1. 복합 인덱스의 컬럼 순서는 어떻게 결정하시나요?

**A.** 우선순위 3단계입니다.

1. **WHERE 절의 등호(=) 조건이 가장 앞**.
2. **범위/정렬 조건이 그 뒤**.
3. **선택도가 높은 컬럼을 더 앞으로** (단, 1·2 원칙이 우선).

```sql
-- (status, created_at) 인덱스가 가장 효율
SELECT * FROM orders
WHERE status = 'PAID'
ORDER BY created_at DESC
LIMIT 20;
```

선택도(distinct ratio)만 보고 등호 조건을 뒤로 보내는 건 잘못된 선택입니다.

## Q2. 복합 인덱스 `(a, b, c)`는 어떤 쿼리에 사용되나요?

**A.** **앞에서부터 매칭**되어야 인덱스를 탑니다.

- `a = ?` → 사용.
- `a = ? AND b = ?` → 사용.
- `a = ? AND b = ? AND c = ?` → 사용(전체).
- `b = ?`만 → **사용 안 됨**.
- `a = ? AND c = ?` → `a`만 사용, `c`는 인덱스 효과 없음.

이 규칙 때문에 컬럼 순서가 매우 중요합니다.

---

## Q3. 범위 조건이 들어가면 왜 그 뒤 컬럼이 효과를 잃나요?

**A.** **B+Tree 인덱스는 정렬된 자료구조**라서, 범위 조건이 만난 시점부터 뒤 컬럼의 정렬이 흐트러지기 때문입니다.

```sql
-- (a, b, c) 인덱스
WHERE a = 1 AND b > 5 AND c = 10
```

`b > 5`로 매칭된 row 안에서 `c`는 정렬되어 있지 않아 추가 필터링은 가능하지만 **인덱스로 좁히지는 못합니다**. 그래서 등호 조건을 인덱스 앞에 두는 게 핵심입니다.

## Q4. EXPLAIN으로 인덱스 사용을 어떻게 검증하시나요?

**A.** 핵심 컬럼들을 확인합니다.

- **`type`**: `const`/`ref`/`range`면 좋음. `ALL`(full scan)이면 인덱스 못 탐.
- **`key`**: 실제 사용된 인덱스. NULL이면 안 탄 것.
- **`rows`**: 옵티마이저가 추정한 스캔 수. 실제와 큰 차이면 통계 갱신 필요.
- **`Extra`**: `Using index`(커버링), `Using filesort`(추가 정렬), `Using temporary`(임시 테이블)는 모두 신호.

PostgreSQL은 `EXPLAIN ANALYZE`로 실제 실행 시간 + 추정 vs 실제 비교가 가능합니다.

## Q5. 인덱스를 추가하면 어떤 비용이 드나요?

**A.** **읽기를 빠르게 하지만 쓰기를 느리게 만듭니다**.

- 모든 INSERT/UPDATE/DELETE가 인덱스 트리를 갱신.
- 인덱스 N개면 쓰기 비용도 대략 N배(해당 컬럼이 포함된 경우).
- 디스크 공간 차지(큰 테이블은 base만큼 큰 경우도).
- 옵티마이저 선택지가 많아져 잘못된 인덱스를 고를 위험도 증가.

"혹시 모르니까" 인덱스를 만드는 건 사고를 키우는 행동입니다.

---

## Q6. 어떤 쿼리에 인덱스를 추가할지 어떻게 결정하시나요?

**A.** **이 인덱스로 사라지는 비싼 쿼리가 무엇인지** 명확해야 추가합니다.

체크 흐름:
1. **slow query log / pg_stat_statements**로 비싼 쿼리 식별.
2. **EXPLAIN**으로 실행 계획 분석.
3. **인덱스 가설**을 세우고 dev 환경에서 검증.
4. 운영 적용 후 **쓰기 처리량 모니터링**.

쓰기가 많은 테이블에서는 사용되지 않는 인덱스(`sys.schema_unused_indexes`, `pg_stat_user_indexes`)를 주기적으로 정리하는 것도 중요합니다.

## Q7. 커버링 인덱스가 무엇이고 언제 효과적인가요?

**A.** **SELECT 절의 모든 컬럼이 인덱스에 포함되어 테이블 읽기 자체가 필요 없는** 경우입니다.

```sql
-- (user_id, created_at, status) 인덱스
SELECT user_id, created_at, status FROM orders WHERE user_id = 1;
```

테이블 페이지를 안 읽으니 매우 빠릅니다. 좁은 범위의 핵심 쿼리에 효과 큼. EXPLAIN의 `Extra: Using index`로 확인.

비용: 인덱스 크기 증가. 대부분의 컬럼을 다 포함시키면 인덱스가 base 테이블만큼 커집니다.

---

## Q8. 함수를 WHERE에 쓰면 인덱스를 못 탄다고 하던데요?

**A.** 맞습니다. **컬럼에 함수를 씌우면 일반 인덱스가 무력화**됩니다.

```sql
-- ❌ email 인덱스 못 탐
WHERE LOWER(email) = 'kim@a.com'

-- ✅ 함수 기반 인덱스
CREATE INDEX idx_email_lower ON users (LOWER(email));
```

또는 데이터 저장 시 정규화(소문자로 저장)하면 일반 인덱스로 충분합니다. 자주 마주치는 함정이라 의식적으로 점검해야 합니다.

## Q9. B-Tree 외 다른 인덱스 타입은 언제 쓰나요?

**A.** PostgreSQL을 기준으로:

- **B-Tree**: 기본. 등호/범위/정렬.
- **Hash**: 등호 전용. B-Tree로 충분해 거의 안 씀.
- **GIN**: 다대다 매핑. 전문 검색(`tsvector`), 배열, JSONB.
- **GiST**: 공간 데이터(PostGIS), 유사도 검색.
- **BRIN**: 매우 큰 테이블 + 시계열 데이터. 블록 단위 요약.

워크로드 특성을 알면 적합한 타입이 보입니다. 일반 OLTP는 B-Tree로 충분합니다.

## Q1. 정규화는 왜 하나요?

**A.** **중복 제거 + 이상 현상 방지**가 본질입니다.

이상 현상은 3가지:
- **갱신 이상**: 같은 정보가 여러 행에 중복돼 일부만 갱신되어 불일치.
- **삽입 이상**: 부수적인 데이터를 위해 무관한 데이터까지 입력해야 함.
- **삭제 이상**: 한 정보를 지우려다 다른 정보까지 사라짐.

정규화는 이 셋을 데이터 모델로 막아서 애플리케이션 코드의 부담을 줄여줍니다.

---

## Q2. 정규화 단계 1NF부터 BCNF까지 짧게 설명해 주세요.

**A.**

- **1NF**: 모든 컬럼이 **원자 값**. 콤마로 구분된 문자열, 반복 그룹 금지.
- **2NF**: 1NF + **부분 함수 종속 제거**. 복합 PK의 일부에만 의존하는 컬럼 분리.
- **3NF**: 2NF + **이행 함수 종속 제거**. A → B → C면 C를 별도 테이블로.
- **BCNF**: 3NF + 모든 결정자가 후보 키.

4NF/5NF는 실무에서 거의 안 다룹니다. **대부분의 시스템은 3NF면 충분**합니다.

## Q3. 역정규화는 언제 결정하시나요?

**A.** **측정 후**입니다. 처음부터 역정규화로 가는 건 함정입니다.

역정규화 동기:
- **JOIN 비용**이 병목으로 측정됨.
- **읽기 >> 쓰기** 비율(카탈로그/리포트).
- **분산 환경**에서 여러 노드의 데이터를 묶는 비용.
- **집계 캐싱**(합계/카운트를 매번 계산할 수 없음).

먼저 인덱스 + 쿼리 튜닝으로 시도하고, 그래도 부족할 때 역정규화로 갑니다.

---

## Q4. 역정규화의 흔한 패턴을 알려주세요.

**A.** 4가지가 자주 쓰입니다.

1. **계산 컬럼 저장**: 주문 합계를 주문 행에 직접. 트리거나 이벤트로 동기화.
2. **참조 데이터 복제**: 주문에 상품명 스냅샷. 상품명이 바뀌어도 주문 시점이 보존됨 + JOIN 회피.
3. **요약 테이블**: 일/월별 집계를 미리 계산해서 저장.
4. **머터리얼라이즈드 뷰**: 옵티마이저가 자동으로 활용.

---

## Q5. 역정규화의 비용은 무엇인가요?

**A.** 세 가지가 늘어납니다.

- **동기화 책임**: 두 곳의 데이터를 항상 일치시켜야 하므로 누락이 곧 버그.
- **쓰기 경로 복잡도**: 한 변경이 여러 행/테이블을 건드림.
- **디스크 사용량**.

특히 동기화는 분산 환경에서 까다로워서 비동기 처리 + 재시도 + 정합성 점검 배치가 같이 가야 합니다.

---

## Q6. NoSQL과 역정규화는 어떤 관계인가요?

**A.** 도큐먼트 DB(MongoDB 등)에서 자주 같이 읽는 데이터를 한 도큐먼트에 묶는 것이 **의도적 역정규화**입니다.

다만 본질은 "JOIN이 안 되니까"가 아니라 **읽기 패턴 최적화**입니다. 어떻게 쿼리할지를 먼저 설계하고 그에 맞춰 데이터 모양을 정합니다. RDBMS의 정규화와 사고 방향이 반대입니다.

## Q7. "정규화했는데 왜 느린가요?"라는 상황은 어떻게 진단하시나요?

**A.** JOIN 자체보다 다른 원인이 더 흔합니다.

- **누락된 인덱스**: 외래키나 자주 조회하는 컬럼.
- **옵티마이저 통계 부정확**: ANALYZE/통계 갱신.
- **잘못된 데이터 타입 비교**: VARCHAR vs INT 비교가 인덱스를 무력화.
- **N+1 쿼리**: ORM의 lazy 로딩 패턴.

EXPLAIN으로 실제 실행 계획을 보고 원인부터 짚는 게 정석입니다.

## Q8. 대리 키(surrogate key)를 PK로 쓸 때 주의할 점은?

**A.** 자연 키(이메일, 주문번호) 종속이 가려져서 **정규화 검증이 어려워집니다**.

해결: **자연 키에도 UNIQUE 제약**을 같이 둬서 비즈니스 무결성을 보존합니다. 대리 키는 외래키 효율과 PK 변경 회피용이고, 자연 키는 비즈니스 의미를 표현합니다. 둘이 충돌하지 않습니다.

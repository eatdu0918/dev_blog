## 핵심 요약

- **정규화**: 중복 제거 + 이상 현상(insert/update/delete anomaly) 방지.
- **역정규화**: 읽기 성능을 위해 의도적으로 중복 허용.
- 실무 기본은 **3NF까지 정규화 후, 측정 기반으로 역정규화**.

## 정규화 단계

### 1NF (제1정규형)
- 모든 컬럼이 **원자 값**(atomic). 콤마로 구분된 문자열 X.
- 반복 그룹 X.

### 2NF (제2정규형)
- 1NF + **부분 함수 종속 제거**.
- 복합 PK의 일부에만 의존하는 컬럼은 분리.

### 3NF (제3정규형)
- 2NF + **이행 함수 종속 제거**.
- A → B → C일 때 C를 별도 테이블로.

### BCNF
- 3NF + 모든 결정자가 후보 키.

### 4NF, 5NF
- 다치/조인 종속 제거. 실무에서는 거의 다루지 않음.

대부분의 시스템은 **3NF면 충분**.

## 정규화로 막는 이상 현상

- **갱신 이상(Update Anomaly)**: 같은 정보가 여러 행에 중복 → 일부만 갱신되어 데이터 불일치.
- **삽입 이상(Insert Anomaly)**: 부수적인 데이터를 위해 무관한 데이터까지 입력해야 함.
- **삭제 이상(Delete Anomaly)**: 한 정보를 지우려다 다른 정보까지 사라짐.

## 역정규화의 동기

- **JOIN 비용**: 큰 테이블 다중 JOIN이 병목.
- **읽기 < 쓰기 비율**: 카탈로그/리포트 등 읽기 우세.
- **분산 환경**: 여러 노드에 흩어진 데이터를 묶는 비용.
- **집계 캐싱**: 합계/카운트를 매번 계산 X.

## 역정규화 패턴

- **계산 컬럼 저장**: 주문 합계를 주문 행에. 트리거/이벤트로 동기화.
- **참조 데이터 복제**: 주문에 상품명 스냅샷(상품명 변경에도 주문 시점 보존 + JOIN 회피).
- **요약 테이블**: 일/월별 집계 사전 계산.
- **머터리얼라이즈드 뷰**: 옵티마이저가 자동 활용.

## 역정규화의 비용

- 동기화 책임 증가 → 일관성 깨질 위험.
- 쓰기 경로가 복잡해짐.
- 디스크 사용량 증가.

**먼저 정규화 + 인덱스 + 쿼리 튜닝**으로 시도하고, 그래도 부족하면 역정규화. 처음부터 역정규화로 가는 건 함정.

## NoSQL과의 관계

- 도큐먼트(Mongo): 자주 같이 읽는 데이터를 한 도큐먼트에 넣음 = 의도적 역정규화.
- "JOIN 안 됨" 때문에 역정규화하는 게 아니라 **읽기 패턴 최적화**가 본질.

## 자주 헷갈리는 디테일

- 3NF는 BCNF의 약화 버전. 보통 실무는 3NF로 부르고 BCNF까지 자연스럽게 만족.
- ID(대리 키, surrogate key)를 PK로 두면 자연 키 종속이 가려져 정규화 검증이 어려움 — 자연 키도 UNIQUE 제약으로 보존.
- 역정규화는 **트리거**가 아니어도 됨. 애플리케이션 레벨 동기화, 이벤트 소싱, CDC 등.

## 면접 follow-up

- "3NF인데 BCNF는 아닌 예?" → 후보 키가 여러 개 + 비프라임 속성 없는 특수 케이스. 흔하지 않아 실무에서는 동일시.
- "역정규화 후 데이터 불일치 발생 경험?" → 비동기 동기화 지연 + 재시도 정책 + 정합성 점검 배치.
- "정규화 했는데 왜 느린가?" → JOIN 자체보다 **누락 인덱스**, 옵티마이저 통계 부정확, 잘못된 데이터 타입(VARCHAR vs INT 비교)인 경우 다수.

# JavaScript 생태계에서 TypeScript로 전향해야 했던 이유

기초 스택으로서 JavaScript로 시스템을 조립해 나갈 초기 개발 시기에는, 데이터 변수에 어떤 값이든 수용하는 동적 타입(Dynamic Typing) 언어 특유의 자유로움이 마치 마법이자 생산성의 축복인 줄만 알았다.

하지만 서비스의 트래픽 규모나 코드 베이스 모듈 컴포넌트의 가짓수가 거대해지면서, 그 축복이 철저히 부메랑이 되어 시스템 프로덕션을 붕괴시키는 한계 시점에 직면했다. 이 파괴적인 런타임 버그의 폭풍을 억제하기 위해 **TypeScript(타입스크립트)**라는 방어막(정적 타입 생태계)을 필연적으로 도입할 수밖에 없었던 구조적 이유와 회고를 정리한다.

---

## 🛑 동적 타입 자유도가 낳은 런타임 파괴 리스크

JavaScript 엔진 환경에서는 내가 작성한 로직 함수가 실제로 호출 구동되기 전까지(Runtime 이전)는, 내부에 숨은 오타나 객체 구조의 데이터 결함 에러를 절대 겉으로 드러내 주지 않았다.

### 암묵적 타입 허용으로 인한 치명적 장애 연출
협업 중 다른 서버나 컴포넌트에서 넘겨준 객체 데이터 스키마 내부에 미세한 철자 오타가 발생했음에도, V8 컴파일러 단에서는 이를 '에러'로 잡지 않고 강제로 `undefined` 라는 엉뚱한 임시 값을 우겨넣은 채 함수를 태연하게 실행시켜 버렸다.
이는 결제 도메인이나 주요 데이터 파이프라인에서 치명적인 서버 장애를 유발했고, 에러 트레이싱 로그를 까볼 때까지 장애 포인트를 영원히 인지조차 할 수 없는 블라인드 재앙을 초래했다.

```javascript
// JavaScript 특유의 무방비 함수 매개변수
function extractUserPaymentId(userPayload) {
  // 치명적 오타! payment_id가 아니라 결함 텍스트 요청
  // JS 환경에선 런타임이 터지는게 아니라 undefined를 할당해 DB 쿼리가 꼬여버림
  return userPayload.payment_di; 
}
```

## 🛡️ 컴파일 타임의 방패: TypeScript의 강제력

이 악몽의 굴레를 뿌리치기 위한 패러다임 전환으로 TypeScript 엔진 컴파일러를 파이프라인 전위에 도입했다.

코드 빌드 명령어가 내려지는 순간 혹은 에디터에서 타이핑이 발생하는 즉시, 약속된 `interface`나 `type` 규격을 벗어난 변이가 감지되면 화면 전체를 빨간 줄로 수놓으며 **컴파일 빌드 자체를 완벽히 셧다운 전면 차단**시켜버리는 이 칼날 같은 엄격함이 백엔드/프론트엔드 양면에 즉각적인 평화를 가져다주었다.

```typescript
interface UserPayload {
  name: string;
  payment_id: string;
}

// 타입 규격을 탑재한 안전 샌드박스 함수
function extractUserPaymentId(userPayload: UserPayload) {
  return userPayload.payment_di; 
  // [TypeScript Compiler 차단망 가동] 
  // Error: Property 'payment_di' does not exist on type 'UserPayload'. 
}
```

## 📈 실무 생산성의 아이러니한 폭발적 상승 효과

초기에는 명세 규격서(Type) 타이핑 코드를 물리적으로 추가 수동 작성해야 한다는 번거로움에 개발 공수 시간이 지연된다는 불만이 존재했다. 하지만 시스템 규모가 커질수록 파급 체감 효과는 상상을 뛰어넘었다.

1. **자동 완성과 IDE 추론의 시너지(IntelliSense)**: 함수에 마우스만 올려도 리턴되는 객체 형태를 IDE 에디터가 전부 예측 타이핑을 리스트로 나열해 주었다. 일일이 매뉴얼 문서를 검색하거나 소스코드를 위로 스와이프해 까볼 필요가 소멸되었다.
2. **리팩토링의 공포 해방**: 서비스 개편으로 도메인 객체의 어원의 이름이나 구조를 일괄 변경 척결해야 할 때, JavaScript 였다면 "어디 파이프가 터질지 모른다"는 두려움 속에서 눈먼 수동 테스트에 목숨을 걸었을 것이다. 하지만 TS 생태계 안에서는 변경점이 영향을 미치는 모든 노드를 컴파일러 에러가 빠짐없이 추적해 잡아내어 빨간 칠로 지시해주므로, 두려움 없이 과감한 코드의 대규모 파괴와 재건 개선 작업을 단행할 수 있는 시스템 아키텍처적 용기(Safety Net)를 얻어냈다.

## 💡 최종 회고 스탠스

TypeScript 마이그레이션 적용은, 개발 생태계 환경 구조를 단지 "오류를 줄이기 위한 규제 통제 감옥"에 가두는 행위가 아니라, 동적 타입의 시한폭탄 불안 요소를 제거함으로써 거시적 시스템의 **유지보수 구조 안정성 용이성과 코드의 맥락 가독성**을 영구적으로 확보해 주는 막대한 이자의 투자 기술이었다는 이치를 피부로 깨쳤다. 
초반의 학습/타이핑 코스트 낭비라는 진입 장벽 이면에는 런타임 장애 축소라는 황금 동아줄이 매달려 있었음을 확신하게 된 의미 높은 생태계 마이그레이션 여정이었다.

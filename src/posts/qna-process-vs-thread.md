## 핵심 요약

- **프로세스**: 실행 중인 프로그램. **독립된 메모리 공간(가상 주소)**, 파일 디스크립터, 권한을 가짐.
- **스레드**: 한 프로세스 안의 실행 흐름. **코드/힙/파일 디스크립터를 공유**, 스택과 레지스터/PC만 따로.
- 멀티프로세스 = 격리/안전, 통신/생성 비용 큼. 멀티스레드 = 공유/빠름, **동기화 책임**이 개발자에게.

## 메모리 구조에서 보기

프로세스 메모리 = `code | data | heap | (스레드별 stack)`.

- 스레드는 **stack만 분리**. 다른 영역은 공유 → 한 스레드의 잘못된 포인터가 다른 스레드/프로세스 자체를 깨뜨릴 수 있음.
- 프로세스는 가상 주소 공간이 다름 → 한 프로세스가 죽어도 다른 프로세스 영향 없음.

## 컨텍스트 스위칭

CPU가 실행 단위를 바꿀 때 일어나는 일.

- **레지스터/PC 저장 및 복원**.
- 프로세스 간 전환은 추가로 **MMU의 페이지 테이블 교체 + TLB flush** 필요 → 캐시/TLB 미스 폭증.
- 스레드 간 전환은 같은 주소 공간이라 **TLB flush 없음** → 훨씬 저렴.

비용 자체는 마이크로초 단위지만, 캐시가 식는 **간접 비용**이 더 큰 경우가 많습니다.

## IPC vs 스레드 통신

- **프로세스 간 통신(IPC)**: 파이프, 소켓, 공유 메모리, 메시지 큐. 직렬화/시스템콜 오버헤드.
- **스레드 간 통신**: 같은 힙을 직접 읽고 씀. 빠른 대신 **레이스 컨디션** 위험 → 락/원자 연산 필요.

## 멀티프로세스 vs 멀티스레드 선택

- **격리/안정성이 중요**: 프로세스(예: Chrome 탭별 프로세스, Nginx worker).
- **공유 메모리/빠른 통신이 중요**: 스레드(예: JVM, .NET).
- **CPU 바운드**: 코어 수만큼 분배. 언어가 GIL이면(파이썬 CPython) 스레드로 CPU 병렬화 불가 → 멀티프로세싱.
- **I/O 바운드**: 비동기/이벤트 루프가 종종 더 효율적(Node.js).

## 사용자 스레드 vs 커널 스레드

- 커널 스레드: OS가 스케줄링. 진짜 병렬 가능.
- 사용자 레벨(그린 스레드, 코루틴, goroutine, virtual thread): 런타임이 스케줄. 컨텍스트 스위칭 비용이 매우 작아 **수십만 개 동시 운영** 가능.
- Go의 goroutine, Java 21의 virtual thread, Kotlin 코루틴이 이 모델. 블로킹 시스템콜만 조심하면 강력.

## 자주 헷갈리는 디테일

- "스레드가 무조건 빠르다"는 잘못된 일반화. 동기화 비용 + 디버깅 난이도가 추가됨.
- "프로세스는 메모리를 절대 공유 못한다" → mmap/shm으로 의도적 공유 가능.
- 멀티코어 = 멀티스레드 필수가 아님. 멀티프로세스도 코어 활용 가능.

## 면접 follow-up

- "프로세스 컨텍스트 스위칭이 스레드보다 비싼 이유는?" → 페이지 테이블 교체 + TLB flush + 캐시 무효화.
- "Node.js는 싱글 스레드인데 어떻게 동시성을 처리하나?" → 이벤트 루프 + 비동기 I/O. CPU 바운드 작업은 worker_threads 또는 cluster.
- "goroutine과 OS 스레드의 차이?" → goroutine은 사용자 레벨에서 다중화(M:N 스케줄링). 스택 크기도 동적(시작 2KB).

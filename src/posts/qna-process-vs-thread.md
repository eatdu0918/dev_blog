## Q1. 프로세스와 스레드의 차이를 한 줄로 설명해 주세요.

**A.**
- **프로세스**: 실행 중인 프로그램. **독립된 메모리 공간(가상 주소)**, 파일 디스크립터, 권한을 가짐.
- **스레드**: 한 프로세스 안의 실행 흐름. 코드/힙/파일 디스크립터를 **공유**, 스택과 레지스터/PC만 따로.

요약하면 **격리/안전 = 프로세스, 공유/빠름 = 스레드**.

---

## Q2. 메모리 구조 측면에서 둘은 어떻게 다른가요?

**A.** 프로세스 메모리 = `code | data | heap | (스레드별 stack)`.

- 스레드는 **stack만 분리**, 나머지는 공유 → 한 스레드의 잘못된 포인터가 다른 스레드까지 깨뜨릴 수 있음.
- 프로세스는 가상 주소 공간이 다름 → 한 프로세스가 죽어도 다른 프로세스에 영향 없음.

이 차이가 격리성과 공유성의 트레이드오프 본질입니다.

---

## Q3. 컨텍스트 스위칭은 정확히 어떤 비용이 드나요?

**A.** CPU가 실행 단위를 바꿀 때 일어나는 일:

- **레지스터/PC 저장 및 복원**.
- **프로세스 간 전환**: MMU의 페이지 테이블 교체 + **TLB flush** → 캐시/TLB 미스 폭증.
- **스레드 간 전환**: 같은 주소 공간이라 **TLB flush 없음** → 훨씬 저렴.

비용 자체는 마이크로초 단위지만 **캐시가 식는 간접 비용**이 더 큰 경우가 많습니다.

---

## Q4. IPC와 스레드 간 통신은 어떻게 다른가요?

**A.**
- **IPC**: 파이프, 소켓, 공유 메모리, 메시지 큐. 직렬화 + 시스템콜 오버헤드.
- **스레드 통신**: 같은 힙을 직접 읽고 씀. 빠른 대신 **레이스 컨디션 위험** → 락/원자 연산 필수.

빠른 통신을 원하면 스레드, 격리된 안전한 통신을 원하면 IPC를 선택합니다.

---

## Q5. 멀티프로세스와 멀티스레드 중 무엇을 선택하시나요?

**A.** 워크로드 특성으로 결정합니다.

- **격리/안정성이 중요**: 프로세스. Chrome 탭별 프로세스, Nginx worker.
- **공유 메모리/빠른 통신이 중요**: 스레드. JVM, .NET.
- **CPU 바운드 + GIL 언어(Python CPython)**: 스레드로는 CPU 병렬화 불가 → 멀티프로세싱.
- **I/O 바운드**: 비동기/이벤트 루프가 종종 더 효율적.

---

## Q6. 사용자 스레드(virtual thread, goroutine)는 OS 스레드와 어떻게 다른가요?

**A.** **런타임이 스케줄링**한다는 점이 다릅니다.

- **커널 스레드**: OS가 직접 스케줄. 진짜 병렬.
- **사용자 레벨**(virtual thread, goroutine, 코루틴): 런타임이 OS 스레드 N개에 사용자 스레드 M개를 다중화(M:N).

장점: 컨텍스트 스위칭 비용이 매우 작아 **수십만 개 동시 운영** 가능.
주의: **블로킹 시스템콜**을 그대로 호출하면 OS 스레드까지 막혀 다른 사용자 스레드가 멈춤. 런타임이 처리해 주는지 확인 필요.

---

## Q7. Node.js는 싱글 스레드인데 어떻게 동시성을 처리하나요?

**A.** **이벤트 루프 + 비동기 I/O** 모델입니다.

- 메인 스레드는 한 개지만 I/O는 libuv가 비동기로 처리.
- I/O 대기 시간 동안 다른 일을 처리해 동시성 확보.

CPU 바운드 작업이 들어오면 메인 스레드를 막아 다른 모든 요청이 멈춥니다. 해결:
- **worker_threads**: CPU 작업을 별도 스레드로.
- **cluster**: 멀티프로세스로 코어 활용.

---

## Q8. 프로세스 컨텍스트 스위칭이 스레드보다 비싼 이유는?

**A.** **페이지 테이블 교체와 TLB flush** 때문입니다.

프로세스가 바뀌면:
- MMU가 다른 가상 주소 공간을 가리키도록 페이지 테이블 교체.
- TLB(주소 변환 캐시)를 비움 → 이후 매 메모리 접근에 TLB miss.
- L1/L2 캐시도 식음 → 캐시 미스 폭증.

스레드는 같은 주소 공간을 공유하므로 이 비용이 거의 없습니다. 같은 프로세스 안의 스레드 간 스위칭은 레지스터 저장/복원 정도로 끝납니다.

---

## Q9. 스레드가 무조건 빠른가요?

**A.** **잘못된 일반화**입니다.

스레드의 추가 비용:
- **동기화 비용**: 락 경합, atomic 연산.
- **디버깅 난이도**: 레이스 컨디션은 재현이 어려움.
- **데드락 가능성**.

단순 격리만 필요한 워크로드면 멀티프로세스가 더 단순하고 안전합니다. 멀티코어를 쓰기 위해 멀티스레드가 필수도 아닙니다 — 멀티프로세스도 코어를 잘 활용합니다.

## 핵심 요약

영속성 컨텍스트는 EntityManager가 관리하는 **1차 캐시이자 트랜잭션 단위 작업 공간**입니다. 트랜잭션이 끝나는 시점에 컨텍스트가 보유한 엔티티의 스냅샷과 현재 상태를 비교해 변경된 필드만 UPDATE 쿼리로 내보내는 동작이 dirty checking입니다.

## 더 깊이

엔티티가 영속 상태가 되면 JPA는 **로딩 시점의 값을 별도 스냅샷으로 보관**합니다. 트랜잭션 커밋 직전 `flush()`가 호출되면 영속 상태의 모든 엔티티에 대해 스냅샷과 현재 값을 비교하고, 차이가 있는 필드를 묶어 UPDATE를 발행합니다. 즉 `repository.save()`를 다시 부르지 않아도 setter 호출만으로 변경이 반영됩니다.

이 과정에서 자주 같이 묻는 개념들이 있습니다.

- **쓰기 지연(write-behind)**: 영속화된 엔티티의 INSERT/UPDATE는 즉시 발행되지 않고 쓰기 지연 SQL 저장소에 모아두었다가 flush 시점에 한꺼번에 나갑니다. 이 덕분에 batch 처리와 같은 최적화가 가능합니다.
- **동일성 보장**: 같은 트랜잭션 내에서 같은 PK로 조회하면 1차 캐시에서 같은 인스턴스를 돌려줍니다. `==` 비교가 가능한 이유입니다.
- **flush 트리거**: 커밋, JPQL 실행, 명시적 `em.flush()` 세 가지가 일반적인 트리거입니다. JPQL 실행 전에 자동 flush가 일어난다는 점이 dirty checking과 자주 충돌하는 포인트입니다.

## 실무에서의 트레이드오프

장점이 곧 함정이 되는 경우가 많습니다.

- **대량 업데이트에서는 오히려 독**입니다. 수만 건을 setter로 갱신하려고 들면 엔티티가 모두 영속 상태로 올라가면서 메모리/스냅샷 비용이 폭발합니다. 이런 경우 `@Modifying` 벌크 쿼리로 1차 캐시를 우회하고, 끝난 뒤 `entityManager.clear()`로 컨텍스트와의 동기화를 명시적으로 끊어줍니다.
- **준영속/영속 상태 혼동**으로 변경이 반영되지 않는 사례가 자주 보입니다. OSIV가 꺼진 환경에서 컨트롤러가 받은 엔티티는 이미 준영속이라 setter로는 dirty checking이 일어나지 않습니다. 서비스 계층의 트랜잭션 안에서 다시 조회해 변경을 적용하는 패턴이 안전합니다.
- **N+1과 결합되면 폭발적**입니다. 컬렉션을 가진 엔티티에 대해 dirty checking이 일어나면 lazy 로딩이 의도치 않게 풀릴 수 있어, 변경 감지가 트리거되는 흐름에서는 fetch 전략과 같이 검토해야 합니다.

## 면접에서 자주 따라오는 질문

- "save() 호출이 필요 없다면 언제 명시적으로 부르나요?" — 신규 엔티티 생성 시, merge가 필요한 준영속 엔티티 처리 시.
- "스냅샷 비교는 어떤 비용이 드나요?" — 엔티티 수 × 필드 수만큼의 객체 비교 비용이 있고, BLOB/LOB 필드가 많으면 체감됩니다.
- "@DynamicUpdate를 켜야 할 때는 언제인가요?" — 컬럼이 매우 많고 변경되는 컬럼이 일부일 때 UPDATE 문 사이즈를 줄이고, 인덱스 영향 범위를 좁힐 때입니다.

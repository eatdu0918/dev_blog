## Q1. REST, GraphQL, gRPC 중 무엇을 디폴트로 선택하시나요?

**A.** **REST가 디폴트**입니다. 특별한 이유 없으면 REST.

이유:
- HTTP 캐시(브라우저, CDN, 리버스 프록시) 그대로 활용.
- OpenAPI 스키마 + 코드젠 도구 풍부.
- 디버깅이 쉬움(`curl`로 바로 됨).
- 학습 곡선 낮고 서드파티 노출에 적합.

다른 두 가지는 REST의 약점이 명확히 드러날 때만 검토합니다.

---

## Q2. GraphQL은 언제 정당화되나요?

**A.** 다음이 동시에 보일 때:

- 클라이언트가 매우 다양(웹/모바일/서드파티)하고 화면마다 필요한 필드가 다름.
- 한 화면을 그리려고 REST 엔드포인트 5~10개를 호출.
- BFF(Backends for Frontends)를 받쳐줄 백엔드 여력.

"GraphQL이 좋아서"가 아니라 **"REST의 한계가 명확해서"** 가 도입 사유여야 합니다.

---

## Q3. GraphQL의 운영 비용은 무엇이 있나요?

**A.** 4가지가 큽니다.

- **N+1 쿼리**: 그래프 탐색이 본질이라 자연스럽게 N+1이 발생. **DataLoader 패턴 강제**.
- **캐시**: 보통 POST + 쿼리 본문이라 HTTP 캐시 그대로 못 씀. Apollo client 캐시나 persisted query로 우회.
- **권한**: 필드 단위 인가 필요 → 복잡.
- **모니터링**: 한 엔드포인트로 들어오는 다양한 쿼리를 어떻게 분류/제한할지 별도 설계.

REST 대비 인프라 부담이 큽니다.

---

## Q4. gRPC는 언제 적합한가요?

**A.** **내부 서비스 간 통신**에서 다음이 중요할 때.

- **성능**: HTTP/2 + Protobuf로 페이로드 작고 지연 낮음.
- **타입 안전**: `.proto`가 단일 진실 소스. 코드젠으로 양쪽 타입 강제.
- **스트리밍**: server/client/bidi streaming이 1급 시민.
- **다양한 언어**: 코드젠으로 일관된 클라이언트 자동 생성.

대부분 회사가 **외부 = REST, 내부 = gRPC** 형태로 가져갑니다.

---

## Q5. gRPC의 단점은 무엇인가요?

**A.**
- **브라우저에서 직접 호출 어려움**: grpc-web 또는 프록시 필요.
- **디버깅 번거로움**: 바이너리 프로토콜이라 curl 직접 호출 못 함.
- **인프라 의존**: HTTP/2 잘 다루는 로드밸런서/트레이싱 필요.
- **쉬운 변경 어려움**: proto 파일 변경 시 양쪽 코드젠 동기화 필요.

내부 마이크로서비스 메시에는 강점이지만 외부 API에는 부적합합니다.

---

## Q6. 한 시스템에서 셋을 같이 쓰는 게 일반적인가요?

**A.** 네. **각자의 강점이 다른 영역에서 발휘**되므로 조합이 흔합니다.

전형적 구조:
- **외부 API**: REST(서드파티 호환).
- **모바일 앞단**: GraphQL BFF(over/under-fetching 해소).
- **내부 서비스 메시**: gRPC(성능 + 타입 안전).
- **비동기 흐름**: Kafka 등 메시징.

"하나만 써야 한다"는 강박보다 "어디에 무엇이 적합한가"의 결정이 더 중요합니다.

## Q7. REST가 진짜 RESTful한 경우는 드물지 않나요?

**A.** 맞습니다. 대부분의 "REST API"는 **HATEOAS까지는 가지 않은 RPC over HTTP**에 가깝습니다.

Roy Fielding 본인도 HATEOAS가 빠지면 진정한 REST가 아니라고 했지만, 실무에서는 그 정도 엄격함이 가치 대비 비용이 너무 큽니다. **자원 중심 명명 + 메서드 매핑 + 적절한 상태 코드**까지만 해도 충분히 잘 짜인 API입니다.

---

## Q8. gRPC와 메시지 큐(Kafka)의 역할 분담은 어떻게 하나요?

**A.** **동기성과 영속성** 기준으로 갈립니다.

- **gRPC**: 동기 호출. 즉시 응답 필요. 강한 결합.
- **메시지 큐**: 비동기. eventual consistency 허용. 느슨한 결합.

호출 그래프가 동기로 깊어지면 **cascading failure** 위험이 큽니다(A→B→C→D 동기 호출, 한 곳 장애로 전체 다운). 비동기 큐로 끊을 수 있는 곳은 끊는 게 회복성에 도움됩니다.

---

## Q9. API 버저닝은 어떻게 하시나요?

**A.** 3가지 방식 중 **URL 버저닝**이 사실상 디폴트입니다.

- **URL**(`/v1/users`): 단순. 캐시/디버깅 쉬움. 클라이언트 마이그레이션 부담.
- **헤더**(`Accept-Version: 2`): 깔끔하지만 도구 지원 약함.
- **쿼리스트링**: 비권장.

호환성 정책:
- **Additive only**: 필드 추가 OK, 제거/변경은 새 버전.
- **Sunset 기간** 명시 + 마이그레이션 가이드.

## Q10. GraphQL Federation은 언제 검토하나요?

**A.** **여러 GraphQL 서비스를 하나의 통합 스키마로 묶는** 패턴입니다(Apollo Federation).

검토 시점:
- 조직이 크고 도메인이 명확히 갈림.
- 각 도메인 팀이 자기 GraphQL 서비스를 운영.
- 클라이언트는 단일 엔드포인트로 보고 싶음.

작은 조직이 처음부터 도입할 필요는 없습니다. **단일 GraphQL → 분리 필요성 발생 → Federation** 순으로 진화하는 게 안전합니다.

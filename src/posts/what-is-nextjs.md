# React를 넘어 Next.js를 도입해야 했던 이유

React 라이브러리의 컴포넌트 생태계를 학습하며 클라이언트 뷰 렌더링에 자신감이 붙어가던 중, 채용 시장의 요구사항이나 실무 아키텍처 구성에서 'Next.js 프레임워크 경험'이 거의 필수 불가결하게 자리 잡고 있음을 목격하게 되었다. 

단순히 React 환경(CRA 혹은 Vite 등) 만으로는 실 서비스 배포 과정에서 어떤 뼈아픈 한계에 봉착하게 되는지, 그리고 이를 지탱하기 위해 왜 Next.js 생태계를 도입해야 하는지에 대해 직접 고찰해 본 렌더링 전환기를 작성해 본다.

---

## 1. CSR 렌더링의 한계와 마주한 빙산

순수 React로 개발하여 배포한 토이 프로젝트는, 사용자가 최초 접속을 때릴 때 의도치 않게 브라우저 화면이 하얗게 멈추는(White Screen) 아주 짧은 플리커링 데드타임을 동반했다. 

이는 **CSR (Client-Side Rendering)** 구조상 피할 수 없는 병목 패널티였다. 서버가 달랑 텅 빈 뼈대 HTML 껍데기만 넘겨주고, 사용자의 브라우저 연산 자원을 통해 무거운 번들 자바스크립트 엔진 파일들을 전부 내려받고 자체 해석 실행한 뒤에야, 비로소 데이터 기반의 돔(DOM) 트리가 화면에 스케치되는 수동적 동작 구조였기 때문이다.

초기 로딩 시간(FCP) 저하 문제뿐만 아니라 가장 치명적인 약점은, 구글이나 네이버 검색망의 크롤링 봇 군단이 내 사이트를 방문했을 때 텅 빈 `div` 태그만을 마주하고 돌아서며 **SEO(검색엔진 최적화)** 점수가 바닥을 친다는 마케팅적인 심각한 사업 디메리트였다.

---

## 2. 서버 사이드에서의 선(Pre)렌더링: SSR과 SSG의 구제

이 결함을 아키텍처로 타파한 것이 Next.js 프레임워크의 본질인 렌더링 시점의 통제, 즉 **서버(Server)단에서 미리 완성된 HTML 페인팅을 조립해서 클라이언트로 직배송해 주는** 접근이었다.

- **SSR (Server-Side Rendering)**: 요청이 인바운드 될 때마다 외부망에서 최신 데이터를 연산해 완전한 HTML DOM 구조를 서버단에서 조립 렌더링한 후 브라우저에 내려 꽂아주는 방식이다.
- **SSG (Static Site Generation)**: 실시간 갱신이 필요 없는 정적 페이지의 경우, 아예 빌드 옵션 타임에 HTML 산출물을 영구히 박제해 구워두고 클라이언트 요청 시 빛의 속도로 응답 캐싱을 꽂아주는 극강의 속도를 자랑했다.

이를 통해 사용자에게 자바스크립트 다운로드 지연 대기시간 없이 렌더링 결과물을 즉각 제공하고, 봇 크롤러에게도 완성된 마크다운을 쥐여 줄 수 있는 매력이 결정적인 기술 스택 전향의 트리거가 되었다.

## 3. App Router 환경의 컴포넌트 이원화 경험

Next.js 버전을 상위로 올리며 접하게 된 'App Router' 디렉터리 아키텍처는 또 한 번의 패러다임 붕괴를 선사했다. 기존의 React 멘탈 모델과 달리, **모든 컴포넌트 뼈대가 기본 디폴트값으로 서버 측에서 렌더링이 종결되는 '서버 컴포넌트(RSC)'로 취급된다는** 무서운 제약사항이었다. 
브라우저 브라우저 Web API(onClick, useState 등)가 필연적으로 탑재되어야 하는 요소만을 정밀하게 국소 부위로 타겟팅하여 명시적인 '클라이언트 컴포넌트(`"use client"`)' 선언으로 도려내 조립해야만 했다.

```tsx
// app/page.tsx (서버 전용 컴포넌트)
// 브라우저로는 단 1바이트의 JS 파싱 코드도 넘어가지 않는 보안/데이터 무거움의 훌륭한 격리 공간
import InteractiveButton from '@/components/InteractiveButton';

export default async function HomePage() {
  // useEffect 훅의 필요성이 증발했다. 함수 컴포넌트 스펙 자체가 서버에서 async/await로 백엔드를 찌른다.
  const userData = await fetch('https://api.example.com/user').then((res) => res.json());

  return (
    <main className="p-8">
      <h1 className="text-2xl font-bold">환영합니다, {userData.name}님!</h1>
      {/* 상태 렌더링 변이 토글 훅이 필요한 막내 버튼만 클라이언트 컴포넌트로 분배 이식한다. */}
      <InteractiveButton title="더 알아보기" />
    </main>
  );
}
```

초기에는 무심코 페이지 레이아웃 상단에 `onClick` 속성을 바인딩했다가 터져버리는 뷰 에러를 수도 없이 마주해야 했다. 
화면을 구성하는 묵직한 첫 골조 및 비동기 데이터 패칭 레이어는 완전히 **서버 컴포넌트** 계층에 귀속 책임으로 맡기고, 유저의 상태 스위칭이나 마우스 이벤트를 청취 상호작용해야 하는 입력 폼 꼬리 컴포넌트들만 잔가지로 분기하여 **클라이언트 컴포넌트**로 쪼개는 계층형 파티션 분리 설계가 App Router 트리의 가장 큰 아키텍처 숙제임을 체득했다.

## 💡 최종 회고와 아키텍처 인사이트

프레임워크 도구가 제공하는 SSR의 서버 연산 렌더링 마법은 SEO 대응과 초기 성능면에서 막강했지만, 이를 구현하기 위해 프론트엔드 개발자조차도 브라우저 런타임 환경과 Node 서버 런타임 환경의 실행 생명주기 분기를 머릿속에서 완벽하게 분할 컴파일할 줄 아는 입체적인 사고 역량을 요구했다.

무대포식으로 화면의 모든 레이아웃에 클라이언트 선언 지시어 `"use client"`를 남용 배포하면 사실상 비싼 돈 주고 Next.js 인프라 서버를 돌리는 의미가 무색해지는 CSR의 함정으로 다시 퇴보 회귀할 우려가 강했다. 기능이 아닌 '렌더링 환경과 상태 의존성' 기준점 하에 컴포넌트를 예리하게 슬라이싱하는 Next.js만의 설계 철학 아키텍처를 체화할 수 있었던 소중한 이정표였다.

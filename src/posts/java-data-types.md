# Java 자료형의 두 얼굴: 기본형과 참조형, 그 이름보다 깊은 메모리 이야기

![Java 자료형 메모리 구조](/public/images/java-data-types-stack-heap.png)

JavaScript로 개발을 시작했을 때, 나에게 숫자란 그저 `number` 하나면 충분했다. 정수인지 실수인지, 크기가 큰지 작은지는 크게 중요하지 않았다. 엔진이 알아서 적절한 메모리를 할당해 줄 것이라는 막연한 믿음이 있었기 때문이다.

하지만 Java의 세계로 넘어오자마자 나는 거대한 타입의 벽에 부딪혔다. `int`, `long`, `double` 같은 숫자 타입의 세분화부터, 도대체 왜 똑같은 정수인데 `int`와 `Integer`가 따로 존재하는지에 대한 의문까지. 이 의문들을 해결하기 위해 한 걸음 더 깊이, Java가 데이터를 바라보는 두 가지 방식인 **기본형(Primitive Type)**과 **참조형(Reference Type)**에 대해 파헤쳐 보기로 했다.

---

## 1. 기본형(Primitive): "가볍고 명확한 데이터 그 자체"

기본형은 실제 값을 변수 안에 직접 저장하는 타입이다. Java에는 8개의 기본형(byte, short, int, long, float, double, char, boolean)이 있다.

이들은 메모리의 **스택(Stack)** 영역에 할당된다. 스택은 크기가 정해져 있고 매우 빠르게 접근할 수 있는 공간이다. 값을 저장하기 위해 다른 곳을 거칠 필요 없이 그 자리에 값이 바로 존재한다.

### 내가 마주한 당혹감: "왜 이렇게 세세하게 나눠놨지?"
처음에는 `long`이 있는데 굳이 `byte`나 `short`를 써야 하는 이유를 이해하지 못했다. 하지만 임베디드 환경이나 대규모 데이터를 다루는 시뮬레이션을 고민해 보니, 불필요하게 큰 타입을 쓰는 것이 얼마나 큰 메모리 낭비인지 깨닫게 되었다. Java의 세분화된 타입은 설계자가 메모리를 효율적으로 관리할 수 있도록 부여한 '칼날'과 같았다.

---

## 2. 참조형(Reference): "데이터를 가리키는 손가락"

기본형을 제외한 모든 것(클래스, 인터페이스, 배열 등)은 참조형이다.

참조형 변수는 실제 데이터가 아닌, 데이터가 저장된 **힙(Heap)** 메모리의 **주소(Reference)**를 저장한다. 즉, 변수는 스택에 존재하지만 그 안에 담긴 것은 "진짜 데이터는 힙의 몇 번지에 있어"라는 정보뿐이다.

### 참조형이 필요한 이유: 유연성과 확장성
기본형은 크기가 고정되어 있지만, 객체(Object)는 얼마나 거대해질지 알 수 없다. 그래서 유연한 관리가 가능한 힙 메모리에 데이터를 밀어 넣고, 스택에는 고정된 크기의 '주소값'만 보관하는 방식을 택한 것이다.

---

## 3. 결정적인 차이: 값의 복사 vs 주소의 공유

이 두 방식의 차이를 가장 명확하게 실감했던 지점은 데이터를 전달할 때였다.

- **기본형**: 값을 전달할 때 내용을 통째로 복사한다. 원본을 수정해도 복사본에 영향을 주지 않는다 (Deep Copy와 유사한 직관).
- **참조형**: 주소값을 복사한다. 즉, 두 변수가 같은 객체를 바라보게 된다. 한쪽에서 객체의 상태를 바꾸면 다른 쪽에서도 그 변화를 마주하게 된다 (Shallow Copy의 원리).

이 원리를 이해하고 나니, 왜 Java에서 객체를 비교할 때 `==`이 아닌 `equals()`를 써야 하는지, 왜 `NullPointerException`은 참조형에서만 발생하는지 그 퍼즐 조각이 비로소 맞춰지기 시작했다.

---

## 4. 코드로 시뮬레이션하기 (Value vs Reference)

Java의 이 핵심적인 동작 방식을 TypeScript를 이용해 재현해 보았다. JavaScript/TypeScript 역시 원시 타입과 객체 타입을 다루는 방식에서 Java와 매우 흡사한 메모리 철학을 공유한다.

```typescript
// test/java-data-types.test.ts 중 일부 발췌

describe('Value vs Reference Simulation', () => {
    it('Primitive: 값 자체가 복사되어 독립적이다', () => {
        let a = 10;
        let b = a; // 값의 복사
        b = 20;

        expect(a).toBe(10); // a는 영향받지 않음
    });

    it('Reference: 주소가 복사되어 같은 객체를 공유한다', () => {
        const obj1 = { value: 10 };
        const obj2 = obj1; // 주소의 복사
        obj2.value = 20;

        expect(obj1.value).toBe(20); // obj1도 변경됨
    });
});
```

## 💡 마치며: 타입의 엄격함이 주는 안정감

그동안 '귀찮은 제약'으로만 느껴졌던 Java의 자료형 체계를 공부하며, 나는 오히려 그 엄격함에서 오는 안정감을 느꼈다. 내 데이터가 메모리의 어디에, 어떤 모양으로, 얼마나 차지하며 머무는지 명확히 인지하고 코드를 짤 수 있다는 것은 설계자로서 큰 무기였다.

자료형은 단순히 데이터를 담는 그릇이 아니었다. 그것은 **데이터가 생명주기를 어떻게 관리할지 결정하는 규정**이었다. 이제는 변수를 하나 선언할 때도 스택과 힙의 지도를 머릿속에 그려보게 된다. 이 지도들이 모여 더 견고하고 예측 가능한 아키텍처를 그리는 밑바탕이 되리라 믿는다.

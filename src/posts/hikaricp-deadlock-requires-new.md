# HikariCP 커넥션 풀의 역습: REQUIRES_NEW가 부른 첫 데드락(Deadlock)의 아찔함

한창 트랜잭션의 개념을 공부하고, 어설프게 `@Transactional(propagation = Propagation.REQUIRES_NEW)`를 알게 되었던 그날. 
"아! 메인 로직이 실패해도 감사 로그(Audit)는 별개로 남겨야 하니까 여기엔 새로운 트랜잭션을 파서 분리해야겠다!" 라며 무릎을 탁 쳤다. 내 코드의 아키텍처가 꽤 세련돼졌다고 으스대며 배포 스위치를 눌렀다.

그리고 정확히 며칠 뒤 이벤트 트래픽이 몰리자마자, 서버는 비명을 지르기 시작했다.
`HikariPool-1 - Connection is not available, request timed out after 30000ms...`

DB 연동이 멈췄다. 장애 알림은 불났고, 식은땀을 연신 흘리며 DB 부서에 "DB 죽은 거 아니에요?"라고 물었을 때 돌아온 대답은 차가웠다.
"DB 서버 자원은 널널한대요? 애플리케이션 쪽 문제 같습니다."
내가 짠 `REQUIRES_NEW` 한 줄이 서버의 목을 완벽하게 졸라버린 커넥션 풀 데드락(Connection Pool Deadlock)에 빠진 것이다. 

---

## 1. HikariCP의 한정된 자원, 그리고 피 말리는 줄서기

Spring Boot의 기본 커넥션 풀인 HikariCP의 세계관을 난 너무 몰랐다. 기본적으로 한 번에 DB와 대화할 수 있는 창구(`maximum-pool-size`)는 10개로 좁게 세팅되어 있다.
10명의 사용자가 작업을 시작하면 나머지 사용자들은 앞사람 10명 중 누군가가 볼일을 다 보고 창구(커넥션)를 비워주기만 기다리며 타임아웃(30초) 전까지 애가 타게 줄을 서야 한다.

나의 멋들어진 코드를 다시 복기해 보았다.

```java
@Service
public class ParentService {
    @Transactional // (1) 유저가 컨트롤러로 들어와서 여기서 1번 창구를 먼저 차지한다.
    public void executeMainBiz() {
        processMainLogic();
        
        // (2) 이제 자식 메서드를 호출하며 로그를 남기려 한다.
        auditService.saveLogAsRequiresNew();
    }
}

@Service
public class AuditService {
    @Transactional(propagation = Propagation.REQUIRES_NEW) 
    // (3) "나 여기서부터 새 살림 차리니까 DB 창구 1개 '더' 가져오셈!"
    public void saveLogAsRequiresNew() {
        logRepository.save(); 
    }
}
```

## 2. 참극의 시나리오: 부모가 자식을 죽이고, 자식이 부모를 죽였다.

트래픽이 적을 때는 문제가 없었다. 하지만 이벤트 당일, 한 번에 **10명의 유저(스레드)** 가 동시에 `executeMainBiz()`로 밀려 들어왔다.

1. HikariCP는 착실하게 10개의 커넥션을 각 스레드의 **부모 트랜잭션**에게 모두 사이좋게 탈탈 털어 나눠줬다. (이제 풀에 남은 커넥션: 0개)
2. 10개의 스레드가 일제히 1번 로직을 처리한 뒤, 로그를 남기려고 **자식 트랜잭션(`REQUIRES_NEW`)** 발급을 HikariCP에게 요청했다.
3. 자식 로직: *"나 물리적으로 분리된 새 커넥션 창구 1개 더 줘."*
4. HikariCP: *"아까 전에 남은 거 싹 다 나가서 0통인데? 뒤에 줄 서서 기다려."*
5. 여기서 교착(Deadlock)의 악마가 이빨을 드러낸다.
   - **부모들**: *"야식(자식 트랜잭션)이 빨리 끝내고 돌아와야 나도 이걸 완료 치고 DB 창구를 반납하든 말든 하지!"*
   - **자식들**: *"씨방 누구라도 DB 창구 하나를 뱉어줘야 내가 빨리 로직을 끝내고 되돌아갈 거 아니야!"*

서로가 서로의 자원을 한없이 기다리는 기형적인 물고 물림. 스레드 10개가 그 자리에서 숨통이 막혀 얼어붙었고, 30초 뒤 일제히 장렬한 타임아웃 예외를 터뜨리며 앱 서버 전체가 넉다운된 것이었다.

## 3. 이 끔찍한 사슬을 끊어내는 방법의 깨달음

장애를 수습하고 며칠 밤낮 레퍼런스를 뒤져본 후, 내가 해결해야 할 근본 공식이 보이기 시작했다.
풀의 최솟값을 강제로 늘려버려 억지로 동맥경화를 뚫는 수학적 공식(`풀 크기 = 스레드 수 x (중첩 - 1) + 1`)도 있었지만, 더 근본적인 건 아키텍처의 설계 결함이었다.

"로그 기록이 굳이 비즈니스 트랜잭션과 실과 바늘처럼 엮여서 호출되어야 할까?"
트랜잭션 끈을 무의미하게 길게 잡고 있었던 게 문제였다. 나는 의존성을 아예 갈라내 버리기로 마음먹었다. 컨트롤러나 바인딩 단에서 비즈니스 로직(트랜잭션 반납)을 먼저 완벽히 끝낸 직후에 다른 빈을 호출해 비동기로 로깅을 돌리거나, 아예 `Spring ApplicationEventPublisher`를 통한 비동기(`@Async`) 트랜잭션 이벤트로 처리를 찢어 파편화시켰다.

## 맺으며

멋있어 보인다고 인터넷에서 주워다 쓴 코드 스니펫 하나가 동료들에게 어떤 지옥을 겪게 만드는지 체감했다. 
내가 짠 코드는 가상 공간을 떠도는 마법이 아니었다. 한정된 DB 커넥션이라는 물리적 자원을 갉아먹는 엄연한 '할당 시스템'이었다. 트랜잭션의 중첩과 부모/자식 간의 락(Lock) 대기를 뼛속까지 인지하게 된 이 아픈 교훈 덕분에, 이제 나는 커넥션 하나를 얻는다는 무게감을 다르게 느끼며 더 깊은 이해의 길에 접어들었다.

# MSA 환경에서 마주한 Database per Service와 JOIN의 딜레마

마이크로서비스 아키텍처(MSA) 학습 프로젝트를 진행하면서 각 도메인(주문, 상품, 유저 등) 서버를 물리적으로 분리하는 작업에 착수했습니다. 서버 코드는 분리했는데, 의외의 가장 큰 난제는 "데이터베이스"에 있었습니다.

초기에는 기존 모놀리식 시절처럼 하나의 커다란 통합 DB에 여러 개의 마이크로서비스가 연결되도록 설정했습니다. 하지만 이는 진정한 도메인 독립 구조가 아니라는 사실을 인지하고, MSA의 격리성을 완성하는 **Database per Service** 패턴을 적용해보며 뼈저리게 겪은 시행착오를 남깁니다.

---

## 🧱 7개의 독립된 DB 환경 구축

강결합 징후를 뿌리 뽑기 위해, 하나의 인스턴스 안에 모여있던 테이블들을 서비스의 개수만큼 완전히 분할했습니다. `user_db`, `product_db`, `order_db` 등 독립적인 스키마나 인스턴스를 가지도록 분리하고 다른 서비스망의 DB에는 절대 직접 접근하지 못하게 네트워크 단에서 제약을 걸었습니다.

- **체감한 장점**: '주문' 팀에서 테이블 컬럼 명세서를 무단으로 변경하더라도 '상품' 서비스는 아무런 런타임 에러를 겪지 않게 되었습니다. 서비스 에러 장애 격리가 구조적으로 완벽해짐을 확인했습니다.

---

## 🚨 위기 발생: 데이터베이스 JOIN 결합의 불가

데이터베이스 분리가 성공적으로 끝났다고 안도할 무렵, 결제 내역 뷰를 렌더링하는 비즈니스 구현 단계에서 벽에 부딪혔습니다.
화면에 **"주문 내역 정보(order_db)와 함께, 해당 주문에 종속된 상품 이름(product_db)과 주문자 이메일(user_db) 문자를 나란히 렌더링해 주세요"**라는 기능이 주어졌습니다.

과거 통합 DB 시절에는 `SELECT ... JOIN` 쿼리 한 줄이면 거저먹기로 가져왔던 데이터를 추출할 방법이 완전히 차단된 것입니다. 물리적으로 분리 구성된 떨어진 DB끼리는 쿼리문 JOIN이 불가하기 때문입니다.

---

## 🛠️ 분리된 데이터 결합을 위한 접근 방식 비교

### 1. API 컴포지션 (Aggregation) 방식
고민 끝에 처음 시도한 대안 방식입니다. 메인 요청을 받은 서버(주문)가 백엔드 내부의 HTTP 규격이나 FeignClient를 통해 다른 서버망(상품, 유저)의 API를 직접 찔러 단편적인 데이터를 가져온 뒤, 애플리케이션의 메모리 상에서 이를 직접 조합(조인)해 내려주는 방식입니다.
- **평가**: 로직이 단순하고 직관적으로 파악됩니다. 하지만 내부 API 교차 호출 빈도 수가 급증하며 응답 지연(Latency)이 크게 생기고, 통신 대상인 상품 서버 하나만 죽어 있어도 주문 내역 전체 조회가 실패해버리는, 결국 MSA의 장점(독립성)을 까먹는 결합도 문제가 잔존했습니다.

### 2. 이벤트 기반 데이터 복제 (Data Replication & CQRS)
API 실시간 동기 호출 통신의 의존 문제를 해결하고자, 카프카(Kafka) 브로커 망을 곁들인 비동기 패턴을 시도해 보았습니다.
상품의 이름 데이터가 변경될 때마다 "상품 정보 변경 이벤트"를 메시지로 발행하고, 주문 서비스는 이 큐 이벤트를 구독하여 늘 대기하다가 **자기 자신의 격리된 DB 안에 필요한 상품 정보를 일부 사본으로 복사하여 갱신 보관해 두는 방식**이었습니다.
- **평가**: 주문 서버는 응답 시 자신의 DB만 단독으로 쿼리하면 되므로 조회가 엄청나게 밀리초 단위로 빨라졌고 연결 서버의 생태계에 전혀 제약을 받지 않게 되었습니다. 반면, 데이터 파이프라인 정합성 맞추기 관리 비용과 전제 인프라 구조의 학습 커브가 기하급수적으로 높아지는 Trade-off가 컸습니다.

---

## 💡 회고

"Database per Service" 패턴을 우여곡절 끝에 강제로 적용해 본 후 얻은 가장 뚜렷한 깨달음은, **MSA 체제에서 데이터 통합과 무결성 관리의 편의성은 사실상 무너진다**는 것이었습니다. 분산 트랜잭션의 롤백 처리 부재뿐만 아니라 단순한 다중 도메인 데이터 조회 요건 하나 조차 거대한 아키텍처 통신 설계 인프라 역량을 추가로 요구한다는 점을 배웠습니다.

앞선 깨달음을 토대로 향후 구조를 고려할 때는 항상 "단순하고 편안한 성능의 RDB 쿼리 모델링을 과감히 버릴 만큼, 본 서비스 도메인이 쪼개져야 할 강력한 이유가 존재하는가?"를 먼저 깊이 생각하게 될 것 같습니다.

# OAuth 2.0 보안 강화: Implicit Grant의 취약점과 PKCE(Proof Key for Code Exchange) 도입의 필요성

지금껏 사내 프로젝트나 사이드 프로젝트에 소셜 로그인(구글, 카카오)을 붙이는 일은 늘 즐거웠다. 'Spring Security OAuth2 Client'나 프론트엔드의 'NextAuth' 같은 무적의 라이브러리 덕분이었다. 제공 스펙대로 대시보드에서 `Client ID`와 `Secret Key`만 복사해다 파일에 붙여 넣으면, 내가 한 거라곤 1도 없는데 알아서 리다이렉트 창이 뜨고 내 손에는 완벽한 구글 이메일 티켓(Access Token)이 쥐어졌다.

그러던 어느 날, 라이브러리를 쓸 수 없는 순수 네이티브 앱(iOS) 개발자 동료와 인증을 연동할 일이 생겼다. 그가 나에게 물었다.
**"서버분아, 제가 앱에서 구글 창 띄워서 인가 코드(Authorization Code)라는 텍스트를 하나 받아왔거든요? 그냥 이거 안 받고 다이렉트로 Access Token 받아서 제 로컬 스토리지에 들고 있으면 안 되나요? 로직이 번거로워요."**

그 순간 나는 꿀 먹은 벙어리가 되었다. "음.. 라이브러리가 원래 그렇게 하던데요? 당연히 교환을 두 번 거쳐야 보안에 좋지 않겠어요?"라는 허접한 앵무새 답변밖에 할 수 없었다.
이 부끄러움을 씻어내기 위해 나는 그날 밤, 내가 썼던 라이브러리의 추상화 장막을 전부 걷어내고 OAuth 2.0 프로토콜의 밑바닥을 뜯어보기 시작했다.

---

## 1. 프론트에 Access Token을 직배송하면 벌어지는 끔찍한 일

동료 앱 개발자의 말처럼, 과거 단일 페이지(SPA) 시절에는 귀찮은 백엔드 경유 없이 인가 서버(구글)에서 프론트 브라우저로 냅다 `Access Token`을 발급해버리는 **Implicit Grant(암묵적 승인)** 방식이 흔하게 쓰였다고 한다.

하지만 이 방식은 최악의 치명타를 안고 있었다. 
요청과 응답이 오가는 브라우저 URL 파라미터나 네트워크 로그 창에, 누구나 갖다 대기만 하면 내 정보를 다 털어먹을 수 있는 **권한의 마스터키(Access Token)가 대문짝만하게 노출**되어 버리는 것이다. 카페의 공용 와이파이를 쓰거나 악의적인 크롬 확장 프로그램 하나만 깔려있어도 내 토큰은 세계 방방곡곡으로 복사 생중계될 위험이 있었다.

그래서 모던 OAuth의 표준은 **Authorization Code Grant(인가 코드 승인)** 로 강제 진화했다.
1. 클라이언트(브라우저/앱)는 구글로부터 **"방금 로그인 성공함" 이라는 1회용 짧은 확인증(`Authorization Code` 쪼가리)**만을 받아 내 백엔드 서버에 던져준다.
2. 내 백엔드 서버는 이 쪼가리를 들고, "우리 서버 백엔드 소스코드 깊숙한 곳에만 숨겨둔 1급 비밀 번호(`Client Secret`)"를 덧붙여서 구글 서버의 뒷문(Backchannel)을 두드린다.
3. 구글은 "아, 비밀번호가 맞으니 해커가 가로챈 코드가 아니라 진짜 너네 서버가 맞구나!"라며 **진짜 Access Token을 내 백엔드 서버에게 몰래 건네준다.**

결국, 브라우저라는 위험한 앞단에는 진짜 마스터키(Token)가 1초도 머물지 않도록 서버 간 깊은 곳에서만 주고받기 위한 피눈물 나는 보안 설계였던 것이다.

## 2. 그런데 백엔드 서버가 없는 "앱 단독" 인증이라면? (PKCE의 탄생)

여기까지 공부하고 나서야 나는 iOS 동료에게 "아, 그래야 토큰이 네트워크에 안 털리니까요!"라고 당당히 설명해 줄 수 있었다. 
하지만 한 가지 거대한 모순에 봉착했다. "만약 우리 백엔드 서버를 거치지 않고, 저 iOS 앱 자체가 구글 인가 서버랑 직접 대화해서 토큰을 교환받아야 하는 상황"이라면 어떨까? 

구글 뒷문을 두드리려면 무조건 `Client Secret` 암호가 필요한데, 이 암호를 iOS 앱 코드 안에 하드코딩해서 배포해 버리면? 해커들이 앱 패키지(APK/IPA)를 디컴파일해서 서버 비밀 번호를 그대로 털어가 버릴 게 뻔했다.
그래서 모바일 앱 클라이언트 단독 교환 시나리오에서는 비밀 번호를 없애버려야 했고, 그 빈자리를 노린 해커가 다른 유저의 `Authorization Code`를 가로채 자기가 토큰으로 교환해버리는 취약점이 생겨났다.

이 지옥도를 막아내기 위해 탄생한 궁극의 수학적 방어탑이 바로 **PKCE(Proof Key for Code Exchange, 픽시)** 라는 녀석이었다. 
이 픽시 기술은 암호학의 걸작이었다. 매 로그인 시도마다 무작위 비밀 번호를 즉석에서 만들어낸다는 아이디어였다.

1. **클라이언트(앱)**: 난수 텍스트를 하나 생성한다(`코드_베리파이어`). 그리고 그걸 SHA-256이라는 함수로 암호화한다(`코드_챌린지`).
2. 클라이언트는 처음에 구글 창을 띄울 때 "나중에 이 암호(`챌린지`) 보여줄 테니까 기억해!" 하며 구글에게 미리 던져둔다.
3. 구글에게서 `Authorization Code`를 무사히 발급받아 다시 앱으로 돌아온다.
4. 이제 클라이언트는 토큰과 교환할 때, 아까 숨겨뒀던 최초의 원본 난수 텍스트(`베리파이어`)를 구글에게 당당히 제시한다.
5. **구글의 결론**: "네가 들고 온 원본을 아까 네가 줬던 수식으로 돌려보니 똑같이 나오네! 오케이 넌 아까 로그인 창을 띄웠던 그놈이 확실해! 토큰 압수 안 하고 넘겨줄게."

중간에서 코드만 달랑 훔친 해커는, 로그인 극초반에 폰 안에서 생성된 '원본 난수 텍스트'를 무슨 수를 써도 알 길이 없었기 때문에 마지막 토큰 교환 문턱에서 매몰차게 쫓겨나는 것이었다.

## 맺으며

1주일 남짓한 삽질의 시간. 남들이 만들어둔 함수 `login(...)` 한 줄을 호출하며 으스대던 내가, 그 한 줄이 감싸고 있던 수많은 백엔드 선배들의 탈취 방어선 피땀(Security)의 역사를 고스란히 뜯어보는 시간이었다.

"우리 서비스, 과연 CSRF엔 안전합니까? PKCE는 적용되어 있나요?"
언젠가 보안팀이나 사수에게서 이런 날카로운 질문이 들어온다면, 나는 이제 어리둥절한 표정을 짓는 대신 자신 있게 아키텍처 다이어그램을 그리며 방어 기제를 설명할 수 있을 것 같다. 기술의 'Why(왜)'를 집요하게 파고들었을 때 느끼는 이 카타르시스가, 더 나은 개발자로 한 계단 오르게 하는 연료가 아닐까.

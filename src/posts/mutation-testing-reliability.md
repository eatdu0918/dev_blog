# 테스트 코드의 신뢰도를 측정하는 방법: 뮤테이션 테스팅(Mutation Testing)의 탐구

![뮤테이션 테스팅의 개념: 변이 생성과 박멸 과정](/public/images/mutation-testing-concept.png)

그동안 테스트 코드를 작성하며 가장 뿌듯했던 순간은 테스트 커버리지 리포트에서 '100%'라는 숫자를 마주할 때였습니다. 내가 짠 모든 코드가 테스트되고 있다는 안도감은 개발자에게 큰 자신감을 주기 때문입니다. 

하지만 어느 날 문득 의문이 생겼습니다. **"단순히 코드가 실행되었다고 해서, 그 테스트가 버그를 잡아낼 수 있다는 뜻일까?"** 

이 의문의 답을 찾아가는 과정에서 '뮤테이션 테스팅(Mutation Testing)'이라는 낯설지만 흥미로운 개념을 발견하게 되었습니다.

---

## 1. 코드 커버리지라는 숫자의 함정

우리가 흔히 사용하는 코드 커버리지는 테스트 과정에서 **'코드의 어떤 부분이 실행되었는가'**만을 측정합니다. 이는 강력한 지표지만, 결정적인 맹점이 있습니다.

예를 들어, 어떤 함수를 호출만 하고 결과를 제대로 검증(`expect`나 `assert`)하지 않는 테스트 코드를 작성하더라도 커버리지는 100%가 될 수 있습니다. 즉, 커버리지는 **실행(Execution)**은 보장하지만 **검증(Verification)**의 품질은 보장하지 못한다는 점을 깨닫게 되었습니다.

---

## 2. 뮤테이션 테스팅: 버그를 주입하여 테스트를 테스트한다

뮤테이션 테스팅은 이 문제를 정반대의 시각으로 접근합니다. **"테스트 코드가 좋은지 확인하고 싶다면, 원본 코드에 일부러 버그를 넣었을 때 테스트가 실패하는지 보면 되지 않을까?"**라는 질문에서 시작합니다.

### 핵심 용어 정리
- **뮤턴트(Mutant)**: 원본 코드에 아주 작은 변화(변이)를 준 복사본입니다. (예: `>`를 `<`로 수정)
- **킬(Killed)**: 테스트 코드가 이 변이를 감지해 실패한 경우입니다. 테스트가 제 역할을 하고 있다는 뜻입니다.
- **생존(Survived)**: 코드가 바뀌었는데도 테스트가 여전히 통과한 상황입니다. 해당 부분에 대한 테스트 검증력이 부족하다는 강력한 신호입니다.

---

## 3. 실전 예제: 경계값의 누락을 잡아내는 과정

간단한 성인 인증 로직을 통해 뮤테이션 테스팅이 어떻게 동작하는지 직접 확인해 보았습니다.

```typescript
// src/examples/mutation-example.ts
export function isAdult(age: number): boolean {
  return age >= 18;
}
```

그리고 이 함수를 테스트하는 코드를 다음과 같이 작성했다고 가정해 봅니다.

```typescript
// src/examples/mutation-example.test.ts
import { describe, it, expect } from 'vitest';
import { isAdult } from './mutation-example';

describe('isAdult()', () => {
  it('18세보다 많은 나이(20)에 대해 true를 반환해야 한다', () => {
    expect(isAdult(20)).toBe(true);
  });

  it('18세보다 적은 나이(15)에 대해 false를 반환해야 한다', () => {
    expect(isAdult(15)).toBe(false);
  });
});
```

위 테스트는 `isAdult` 함수의 모든 라인을 실행하므로 **커버리지 100%**가 나옵니다. 하지만 여기서 뮤테이션 테스팅 도구가 아래와 같이 코드를 변조한다면 어떻게 될까요?

```typescript
// 변조된 코드 (Mutant)
return age > 18; // '>='를 '>'로 바꿈
```

놀랍게도 위 테스트 코드는 이 변조된 코드에서도 **모두 통과(Success)**합니다. 20세는 여전히 `true`이고, 15세는 여전히 `false`이기 때문입니다. 이것이 바로 **뮤턴트가 생존한(Survived)** 상황이며, 우리 테스트가 '18'이라는 경계값을 제대로 검증하지 못하고 있음을 증명합니다.

## 4. StrykerJS를 통한 탐구

JavaScript/TypeScript 생태계에서는 **StrykerJS**라는 도구가 이 과정을 자동화해 줍니다. 

학습하며 StrykerJS를 설정해 보니, 내부적으로 우리 소스 코드를 분석해 수십, 수백 개의 뮤턴트를 생성하고 각각에 대해 테스트를 반복 실행하는 것을 볼 수 있었습니다. 비록 커버리지 측정보다 시간은 훨씬 오래 걸리지만, "내 테스트의 어느 부분이 취약한가"를 시각적으로 보여주는 보고서는 그만큼 가치가 있었습니다.

---

## 💡 마치며: 숫자를 넘어 본질로

뮤테이션 테스팅을 학습하며 느낀 가장 큰 점은 **"테스트도 결국 코드이고, 우리를 속일 수 있다"**는 사실이었습니다. 

그동안 리포트의 100%라는 숫자에 매몰되어 '진정으로 버그를 막아내고 있는가'라는 본질을 잠시 잊고 있었던 것 같습니다. 모든 프로젝트에 뮤테이션 테스팅을 도입하기에는 비용이 들 수 있겠지만, 적어도 경계값이나 복잡한 비즈니스 로직만큼은 이 관점을 빌려와 더 꼼꼼하게 검증해야겠다는 다짐을 하게 되었습니다.

다음에는 실제 개인 프로젝트의 핵심 모듈에 StrykerJS를 직접 적용해 보고, 얼마나 많은 '생존 뮤턴트'가 발견되는지 충격적인(?) 결과를 마주해 볼 계획입니다.

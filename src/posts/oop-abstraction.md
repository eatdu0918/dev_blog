# 추상화(Abstraction): 복잡함을 걷어내고 본질에 집중하기

스마트폰에서 '사진 촬영' 버튼을 누를 때 우리는 렌즈의 셔터가 어떻게 열리는지, 센서가 빛을 어떻게 전기 신호로 바꾸는지 고민하지 않습니다. 오직 '촬영'이라는 버튼 하나에 집중합니다. 리모컨의 채널 버튼을 누를 때도 내부 회로의 동작 대신 '채널 변경'이라는 결과에만 집중합니다. 소프트웨어 공학에서의 **추상화(Abstraction)**도 이와 같은 원리입니다.

불필요한 세부 구현 사항은 뒤로 숨기고, 사용자(혹은 다른 모듈)가 알아야 할 핵심 기능만을 추출해내는 과정입니다.

![추상화 시각화](/public/images/oop-abstraction.png)

## 무엇을(What) 할 것인가에 집중하기

추상화의 핵심은 "**어떻게(How)** 구현했는가"가 아니라, "**무엇(What)**을 할 수 있는가"를 정의하는 데 있습니다. 

만약 알림을 보내는 기능을 만든다면, 이메일로 보낼지 SMS로 보낼지부터 고민하는 것은 '구현'의 영역입니다. '알림을 보낸다'라는 공통된 행위 자체에 집중하여 규격을 만드는 것이 '추상화'입니다.

```typescript
// 추상화 단계: 알림이라는 행위의 규격을 정의
interface NotificationService {
  send(recipient: string, message: string): Promise<boolean>;
}
```

이 인터페이스를 사용하는 비즈니스 로직은 실제 알림이 어떻게 전송되는지 알 필요가 없습니다. 그저 `send()` 메서드를 호출하면 알림이 간다는 사실만 알면 됩니다.

## 실전 예제: 데이터 저장소의 추상화

파일 시스템에 데이터를 저장하다가 나중에 데이터베이스로 바꿔야 하는 상황이 올 수 있습니다. 이때 추상화가 없다면 저장 방식을 바꿀 때마다 모든 비즈니스 로직을 수정해야 합니다. 

추상화를 도입하면 비즈니스 로직은 저장 시스템의 세부 사항으로부터 자유로워집니다.

```typescript
// 저장소의 본질적 기능만 정의함
interface DataStorage {
  save(id: string, data: any): void;
  find(id: string): any;
}

// 비즈니스 로직은 추상화된 인터페이스에만 의존함
class UserProfileService {
  constructor(private storage: DataStorage) {}

  updateProfile(userId: string, profileData: any) {
    // 저장 방식이 로컬 파일인지, 클라우드 DB인지 알 필요가 없음
    this.storage.save(userId, profileData);
  }
}
```

## 학습을 통해 깨달은 점

추상화는 단순히 코드를 숨기는 기술이 아니라, **'생각의 계층'**을 만드는 작업임을 깨달았습니다. 적절한 추상화는 개발자가 한 번에 다뤄야 할 정보의 양을 줄여주어, 더 큰 그림의 설계에 집중할 수 있게 해줍니다.

너무 과한 추상화는 오히려 코드의 가독성을 해칠 수 있기에, 항상 '어떤 정보를 감추고 어떤 정보를 드러낼 것인가'에 대한 균형을 맞추는 것이 중요하다는 점도 깊이 새기게 되었습니다.

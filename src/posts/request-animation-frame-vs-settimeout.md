# 자바스크립트 애니메이션 제어: setTimeout 한계와 requestAnimationFrame

자바스크립트로 직접 스크롤 애니메이션이나 진행 게이지 상승 바를 구현할 때, 자바스크립트 학습 초기에는 가장 익숙했던 `setInterval`이나 `setTimeout` 타이머 함수 방식을 활용하곤 했습니다. 

단순 토이 프로젝트 당시에는 눈에 띄지 않았으나 실 서비스 환경에 올렸을 때, 애니메이션 동작이 미세하게 불규칙하게 떨리거나 페이지가 과한 기기 CPU 자원 점유율을 초래하는 문제가 지속적으로 발생했습니다. 이 이슈를 타파하는 과정에서 브라우저의 렌더링 파이프라인 메커니즘을 새롭게 학습하고, `requestAnimationFrame` API의 본질적인 필요성을 체감했던 과정을 되짚어봅니다.

---

## 🛑 타이머(setTimeout) 로직 기반 애니메이션의 구조적 한계

개발 도구를 통한 프로파일링 로그 등을 점검해 본 결과, 일반적인 타이머 호출 함수는 본질적으로 브라우저가 동작하는 화면 구동 메트릭을 전혀 신경 쓰지 않았습니다.

### 1. 화면 갱신 주기 엇박자로 인한 프레임 손실
브라우저는 통상 초당 60프레임(약 16.6ms에 1번 꼴) 간격으로 모니터에 화면을 도출합니다.
그러나 `setTimeout(func, 10)` 과 같이 짧은 강제 타이밍으로 뷰 단의 위치 갱신 연산을 수행시켰을 때, 브라우저의 기본 화면 그리기 리듬보다 제 자바스크립트 실행 속도가 엇갈려버리며 중간 연산 값이 결국 모니터에 도출(Paint)되지 못하고 증발당하는 일명 프레임 드롭(Frame Drop) 현상이 나타났습니다. 이로 인해 사용자에게 부드러운 화면 대신 묘하게 뚝툭 떨리는 현상을 노출하게 되었습니다.

### 2. 콜 스택 밀림으로 제어력 상실 문제
자바스크립트는 싱글 스레드 언어 환경에 놓여 있으므로, 다른 무거운 백그라운드 작업이나 컴포넌트 렌더가 겹쳤을 때, `10ms` 뒤로 예약했던 함수가 제때 호출되지 못하고 실행이 지연되는 현상은 너무나 잦았습니다. 이는 애니메이션의 균일성을 완전히 해치는 변수였습니다.

### 3. 활성화되지 않은 프로세스에서의 무의미한 연산 루프
가장 최악이었던 점은, 사이트 방문자가 창을 내리거나 탭 화면을 벗어났을 때도 타이머 기반 루핑은 여전히 뷰 변경 연산을 이어갔다는 것입니다. 화면이 안 보여지는 환경에서도 불필요한 배터리, CPU 소모량을 무의미하게 발생시키는 문제 로직이었습니다.

## 🟢 렌더링 최적화의 열쇠, requestAnimationFrame (rAF)

이러한 수동 타이머의 문제 요소를 극복하고자, 애니메이션 및 뷰 변경 연산 모델을 브라우저 렌더링에 내재된 전용 훅 모델인 `requestAnimationFrame` 스펙으로 전환 처리했습니다.

```javascript
let targetDiv = document.getElementById("animBox");
let positionValue = 0;

function moveBox() {
  positionValue += 2; // 프레임 연산에 기반한 이동 로직 적용
  targetDiv.style.left = positionValue + 'px';
  
  if (positionValue < 200) {
    // 억지로 강제 지정한 delay 값이 체외됨.
    requestAnimationFrame(moveBox); 
  }
}

// 최초 발동 구문
requestAnimationFrame(moveBox); 
```

### 1. 주사율에 완벽히 포개어지는 유연한 동기화 성능 
해당 시스템을 호출 구조에 넣자마자, 해당 함수가 다음 브라우저 렌더가 화면을 스크린에 출력하기 바로 직전에 도달해야 발화되도록 통제받게 되었습니다. 타이머 강제성 없이 브라우저 고유 시스템이 실행 배정을 자율 판단하므로, 억지성이 없어지고 이전과는 비교할 수 없는 부드러운 스위핑 결과물이 확보되었습니다.

### 2. 백그라운드 프로세싱의 스마트한 저수준 휴면 제어 확보
서브 탭 이동 시, 강제로 할아버지처럼 회전하던 setInterval과 대조적으로 rAF 시스템은 브라우저 렌더가 우선순위 저하 판정을 내리자마자 뷰 연산 횟수를 급강하 시키거나 비활성 휴면 처리로 돌이켰습니다. 쓸데없는 메모리와 연산을 단번에 절감시킬 수 있게 되어 앱 자원 컨트롤이 정교해졌습니다.

## 💡 최종 회고 정리

과거의 개발 패턴은 콘솔 창에서 로직 상 무결성이 지켜진다면 수치를 부여하여 객체를 강제로 컨트롤해도 된다고 생각하는 우물 안 시각이었습니다. 

하지만 이번의 병목 트러블 해소 구간을 건넌 이후로, 뷰어에 표시판을 달아주는 **시각적 DOM 좌표/크기 재제어 연산에는 항상 브라우저 랜더링 파이프라인의 시스템적 메커니즘 룰(rAF 생태계 등)**과 손을 잡아야만 사용자의 눈과 체감에 걸맞는 쾌적한 UX를 납품할 수 다는 사실을 머릿속 깊은 곳까지 주입할 수 있었습니다.

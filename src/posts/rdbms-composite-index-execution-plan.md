# Table Full Scan의 악몽: 복합 인덱스(Composite Index)와 실행 계획을 배우다

처음 백엔드 API를 짜고 테스트할 땐 데이터가 고작 수십 건이라 쿼리가 느리다는 걸 인지할 겨를조차 없었다. 하지만 사내 어드민 페이지에 회원 결제 내역 조회 기능을 붙이고, 실 데이터 백만 건이 쌓였을 무렵 문제가 터졌다. 
"개발자님, 날짜랑 상태값 넣고 검색 누르면 모래시계만 한참 돌다가 3초 뒤에 떠요."

나는 당황하지 않고 당당하게 답했다. "아, 조회가 느리군요. 인덱스(Index) 걸겠습니다!"
그리고는 `WHERE` 절에 있는 `status`, `created_at`, `member_id` 3개의 컬럼 각각에 무지성으로 단일 인덱스 3개를 쾅쾅 찍어냈다. 
"이제 마법처럼 0.01초 만에 뜨겠지?" 하고 실행해 본 내 뒤통수를 친 건 변함없는 3초의 딜레이와, `EXPLAIN` 명령어 실행 결과표에 뜬 새빨간 글씨 **`type: ALL (Table Full Scan)`** 이었다.

이날 나는 데이터베이스가 인덱스를 어떻게 타고 들어가는지(카디널리티와 복합 순서) 제대로 알지도 못한 채 겉핥기로만 RDBMS를 써왔음을 뼈저리게 반성했다.

---

## 1. 옵티마이저(Optimizer)는 바보가 아니었고, 내가 바보였다

내가 짰던 뼈대 쿼리는 이러했다.
```sql
SELECT * FROM Orders 
WHERE status = 'PAID' 
  AND created_at >= '2026-01-01' 
  AND member_id = 15302;
```

나는 각 컬럼마다 인덱스가 있으니 DB가 세 인덱스를 모두 찰떡같이 써서 교집합을 찾아줄 거라 믿었다.
하지만 레퍼런스를 뒤져본 결과, **MySQL 옵티마이저는 기본적으로 WHERE 조건절을 평가할 때 테이블당 '단 하나의 인덱스'만 선택해서 타는 성질**이 있었다. (Index Merge라는 예외가 있지만 비효율적이다).

결국 옵티마이저는 내 3개의 인덱스 중 그나마 가장 좁아 보이는 것 하나만 대충 골라 탔고, 나머지 2개 조건은 디스크를 직접 긁어오는 무자비한 Random I/O를 통해 데이터 블록을 하나하나 뜯어보며 참/거짓을 검사하고 있었던 것이다.

---

## 2. 복합 인덱스(Composite Index)의 창설, 그리고 무서운 "순서"의 법칙

이 3개의 검색 조건을 한 방에 꿰뚫으려면 3개 컬럼을 하나로 묶은 **'복합 인덱스(Composite Index)'** 하나를 제대로 깔아줘야만 했다. 
그런데 여기서 끝이 아니었다. 컬럼을 묶을 때 **'순서(Order)'** 가 성능의 모든 것을 지배한다는 사실을 배웠다.

전화번호부를 떠올려봤다. `(성별, 나이, 이름)` 순으로 정렬된 책에서 "남성(성별)이고, 김 모 씨(이름)인 사람"을 찾으려면 어떻게 될까? 첫 번째 조건인 남성 페이지까진 빠르게 펴지만, 중간 조건인 '나이'를 건너뛰었으므로 남성 페이지 전체를 처음부터 끝까지 다 맨눈으로 훑어야(Full Scan) '김 모 씨'를 찾을 수 있다. B-Tree 인덱스도 완벽히 똑같이 동작했다.

이 원리를 깨닫고 나서 나는 복합 인덱스를 구성하는 **실무의 3대 황금률**을 내 노트에 적어내려갔다.

1. **상등 조건(`=`)을 무조건 맨 앞에 둔다.**
   `status = 'PAID'`나 `member_id = 15302` 같이 정확히 점(Point)으로 일치하는 조건을 제일 앞단에 박아 트리 탐색 범위를 단숨에 쳐낸다.
2. **카디널리티(유니크한 정도)가 높은 것을 앞에 둔다.**
   분별력이 고작 2개뿐인 '성별'보다는, 수만 개로 쪼개지는 '사용자 ID(`member_id`)'를 인덱스 선봉에 세워야 DB가 헛수고를 덜 한다.
3. **범위 조건(`>=`, `<`, `LIKE`)은 가장 수비 자리에(뒤편에) 둔다.**
   가장 충격적인 공식이었다. `created_at >= '2026-01-01'` 같은 선(Range) 조건이 등장하는 순간, **그 인덱스 컬럼 뒤에 배치된 모든 나머지 컬럼들의 인덱스 타기 기능은 완전히 정지(상실)된다**는 것이다! 이미 범위로 넓게 퍼져버려 트리의 정렬 기준을 잃었기 때문이다.

따라서 내가 새로 짜야 했던 완벽한 고속도로는 `INDEX(member_id, status, created_at)` 이 정답이었다.

---

## 3. 내 생애 최초로 쿼리 성적표(EXPLAIN)를 해석하다

인덱스를 갈아엎고 나서 다시 한번 `EXPLAIN` 키워드를 붙여 쿼리를 날렸다.
이전에는 두려워서 보지 않으려 했던 그 표가 이제는 읽히기 시작했다.

- `type: ALL` (최악의 범죄): 어제 만났던 지옥이다. 인덱스 무시하고 테이블 풀 스캔.
- `type: range` (합격선): 오늘 내가 짜낸 결과. 원하는 범위만 깔끔하게 끊어서 인덱스를 탔다.
- `type: ref` (우등생): PK가 아닌 인덱스 컬럼으로 동등(`=`) 조건을 걸어 기가 막히게 조회해냈다.

이뿐만이 아니었다. `WHERE YEAR(created_at) = 2026` 처럼 인덱스 컬럼을 함수로 감싸거나 가공하는 순간, B-Tree가 원래 값을 잃어버려 또다시 인덱스를 버린다는 사실도 깨달았다. 이를 피하고자 `WHERE created_at >= '2026' AND created_at < '2027'`로 풀어서 짜는 꼼꼼함도 생겼다.

---

## 맺으며

JPA의 `findAll()` 메서드가 뿜어내는 수많은 마법들 속에서 잠시 망각하고 있었던 데이터베이스 I/O의 진정한 두려움. 기능 구현에 급급했던 과거를 넘어, 느려진 쿼리의 범인을 찾기 위해 실행 계획(`EXPLAIN`)을 까보고 B-Tree의 마음을 읽어내는 지금 이 순간의 고민들.
성능 병목을 마주할 때마다 막연한 두려움 대신 "이번엔 어떤 컬럼을 조합해서 고속도로를 깔아볼까?" 하는 묘한 기대감이 생기기 시작했다. 더 깊은 지식을 탐구하는 길목에서, RDBMS는 아직 파먹을 게 너무나 많은 커다란 산맥이다.

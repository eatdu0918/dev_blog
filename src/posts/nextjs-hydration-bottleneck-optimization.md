# 렌더링의 늪에서 헤엄치다: 하이드레이션(Hydration) 병목과 Server Components의 충격

처음 React로 CSR(Client-Side Rendering) 환경을 맛보았을 때, 깜빡임 없이 부드럽게 페이지가 넘어가는 SPA(Single Page Application)의 매력에 푹 빠졌었다. 
하지만 서비스가 커지고 유입량이 늘어나면서 심각한 벽에 부딪혔다. 화이트 스크린에 버퍼링만 빙글빙글 도는 끔찍한 초기 로딩 시간, 그리고 검색 포털에서 우리 서비스가 전혀 검색되지 않는 SEO(검색 엔진 최적화) 문제는 과거의 나를 좌절하게 했다.

그래서 야심 차게 Next.js를 도입했다! "와, 이게 SSR(Server-Side Rendering)이구나! 접속하자마자 서버가 완전한 HTML을 렌더링해서 브라우저로 한 방에 쏴주니까 초기 화면 지연이 완전히 사라졌어!"라며 환호성을 질렀다.
나는 내가 프론트엔드 최적화의 정점에 도달한 줄 알았다. 무거운 자바스크립트가 브라우저에 내려오고, **하이드레이션(Hydration)** 이라는 통곡의 벽이 내 목을 조르기 전까지는.

---

## 1. 하이드레이션(Hydration)이라는 착시 현상, 그리고 불쾌한 골짜기

사건은 모바일에 최적화된 상품 상세 페이지를 작업할 때 벌어졌다.
사용자 폰 화면에는 분명히 화려한 상품 배너와 "장바구니 담기" 버튼이 반짝이며 떠 있었다. (SSR이 HTML을 잘 구워다 줬으니까).
그런데 치명적인 버그 리포트가 쏟아졌다. "화면은 떴는데, 장바구니 버튼을 3번씩 눌러도 한동안 반응이 없어요. 그러다가 갑자기 우수수 담기네요. 화면이 얼어버렸어요."

디버거를 켜보고 난 경악했다. 브라우저의 메인 스레드 연산(CPU)이 터져나가고 있었다.
원리는 이랬다. 서버가 내려준 그 멋진 화면은 그저 뼈대뿐인 마네킹(HTML)이었다. 브라우저는 뒤이어 메가바이트(MB) 단위의 거대한 React 자바스크립트 번들 꾸러미를 다운받아 파싱하고, 그 죽은 껍데기 HTML DOM 요소를 하나하나 찾아내 클릭 이벤트 리스너(onClick)나 가상 돔 생명 주기 생명수를 억지로 불어넣어야 했다. 

**이 과정, 건조한 HTML에 물(JavaScript)을 적시어 생명을 불어넣는 "하이드레이션(Hydration)" 구간 동안 브라우저는 눈만 뜨고 있을 뿐 전신 마비 상태(TTI 지연)에 빠졌던 것이다.**
화면은 보이는데 상호작용은 되지 않는 기괴한 시간. 이른바 악명 높은 프론트엔드의 타임 랙(Uncanny Valley) 현상 앞에, 그저 "Next.js 조언대로 SSR 썼는데 왜 느릴까?"라며 변명만 하던 지난날이 부끄러워졌다.

---

## 2. 생명수를 덜어내기 위한 발악: Code Splitting

이 병목을 부수기 위해 나는 발버둥 쳤다. "초반에 모든 자바스크립트를 한 번에 내리지 말자!"
`next/dynamic`과 `React.lazy`를 동원해 소스 코드를 칼질하기 시작했다. 스크롤을 한참 내려야 보이는 무거운 차트 라이브러리, 사용자가 클릭해야만 뜨는 대규모 모달 창들은 죄다 청크 파일 단위로 쪼개어(Code Splitting) 처음 하이드레이션해야 할 자바스크립트 짐짝에서 덜어냈다. 

분명 효과는 탁월했다. 하지만 서비스 규모가 미친 듯이 커지는 상황에서 매번 페이지마다 눈에 핏대를 세우며 "어디를 잘라내서 게으른 로딩으로 지연시킬까?"를 고민하는 것은 밑 빠진 독에 물 붓기처럼 느껴졌다. 조금 더 근본적인 아키텍처의 혁신에 목마르던 시기였다.

---

## 3. 발상의 전환, Server Components(RSC) 패러다임이 가져온 충격

이 지리멸렬한 소모전의 끝자락에서 Next.js 13 버전(App Router)과 React 18의 **서버 컴포넌트(React Server Components)** 가 등장했다. 그들의 선언은 내 상식을 완전히 뒤흔들어 놓았다.
*"애초에 하이드레이션(JS)이 필요한 곳과, 단순 조회용이어서 자바스크립트가 단 1바이트도 필요 없는 곳을 컴포넌트 단위로 격리해 버립시다!"*

그간 SSR은 한 페이지 전체를 하나로 퉁쳐서 "클라이언트 컴포넌트"로 취급했다면, RSC는 달랐다.
- 단순히 정적인 상품 설명, 블로그 포스트 본문 텍스트는 오직 **서버에서만 렌더링(RSC)** 해서 아무런 JS 번들 없이 순면 100% HTML 결괏값만 브라우저로 내려보냈다.
- "장바구니 담기", "수량 조절"처럼 진짜로 클릭 이벤트나 `useState`가 필요한 아주 얇은 버튼 껍데기 컴포넌트(`"use client"`)에만 선택적으로 뇌졸중 없는 최소한의 하이드레이션을 수행하게 허용했다.

거대한 바위(페이지)를 부수어, 서버가 연산해야 할 모래알과 클라이언트 브라우저가 안고 가야 할 모래알(JS)을 가장 이상적인 크기로 분리시켜버린 혁명. 그 작동 원리를 직접 구현하고 깎아내려 가면서, 프론트엔드 최적화가 더 이상 트릭(꼼수)이 아니라 렌더링 철학싸움으로 진화했음을 소름 돋게 깨달았다.

## 맺으며

돌아보면, "그냥 Create-React-App 대충 띄워놓고 API나 붙이면 프론트 개발자지!" 하며 안일하게 타이핑하던 과거의 착각이 부끄럽게 다가온다.

하이드레이션의 불쾌한 골짜기를 건너고, 어떻게든 메인 스레드에 떨어지는 자바스크립트의 비를 피하고자 아키텍처를 이리 뜯고 저리 뜯던 그 치열했던 고민들. 
결국 프론트엔드 렌더링 역사는 **"아무것도 모르는 불쌍한 클라이언트 폰(브라우저)에게, 최대한 일을 덜 주기 위해 서버와 어떻게 렌더링 책임을 나눠 가질 것인가"** 에 대한 기나긴 줄다리기였다. 이 렌더링 진화도의 본질을 피부로 마주하고 나니, 다음 신규 프로젝트에서는 어떤 기술 스택 구성을 주도해야 할지, 한 명의 설계자(미들)로서 자신감 있는 결정을 내릴 수 있을 것 같다.

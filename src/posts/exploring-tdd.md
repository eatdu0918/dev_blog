# Red-Green-Refactor 사이클의 이해: TDD(테스트 주도 개발)의 기초와 실습

![TDD Cycle Concept](/public/images/tdd-cycle.png)

기존에 작성한 코드를 수정(리팩터링)하고 나서, "이게 정말 예전처럼 잘 돌아갈까?"라는 불안감에 휩싸여 본 적이 누구나 있을 것이다. 나 역시 기능을 추가할 때마다 사이드 이펙트가 두려워 코드 수정을 주저하게 되는 상황을 반복하면서, "어떻게 하면 내 코드에 대한 확신을 가질 수 있을까?"라는 질문을 던지게 되었다.

그 답을 찾기 위해 검색을 이어가던 중 발견한 것이 바로 **TDD(Test-Driven Development, 테스트 주도 개발)**였다. 구현 코드보다 테스트 코드를 먼저 작성한다는 이 파격적인 사고방식은 나에게 적잖은 충격을 주었다. 이번 포스팅에서는 TDD의 핵심인 '실패에서 시작하는 선순환'을 직접 경험하며 느낀 점들을 정리해보려 한다.

---

## 1. TDD의 리듬: Red-Green-Refactor

TDD는 단순히 '테스트를 많이 짜는 것'이 아니었다. 그것은 **Red(실패) -> Green(성공) -> Refactor(개선)**라는 세 가지 단계의 리듬을 반복하는 설계 과정이었다.

1.  **Red (실패)**: 구현하려는 기능의 요구사항을 반영한 '실패하는 테스트 코드'를 먼저 작성한다.
2.  **Green (성공)**: 테스트를 통과시키기 위한 '최소한의 코드'를 작성한다. (멋진 코드가 아니어도 된다!)
3.  **Refactor (개선)**: 테스트가 통과된 상태에서 중복을 제거하고 코드의 가독성을 높이는 등 리팩터링을 진행한다.

이 중 가장 생소했던 것은 **'최소한의 코드'**만 작성하는 Green 단계였다. 상상만 했던 완벽한 설계를 잠시 내려두고, 당장 눈앞의 실패(Red)를 해결하는 데만 집중하는 경험은 오히려 내 머릿속을 맑게 해주었다.

---

## 2. 비밀번호 유효성 검사기로 실습해보기

글로만 읽는 것보다 직접 구현해보는 것이 제맛이다. 간단한 비밀번호 유효성 검사기를 TDD 방식으로 만들어보았다.

### Red: 첫 번째 실패
먼저 비밀번호가 8자 미만일 때 실패해야 한다는 테스트를 작성했다. 아직 `validate` 메서드에는 아무런 로직이 없기에 테스트는 당연히 실패(Red)한다.

```typescript
// test/tdd-example.test.ts
it('비밀번호가 8자 미만이면 실패해야 합니다.', () => {
    expect(validator.validate('short')).toBe(false);
});
```

### Green: 첫 번째 성공
이 테스트를 통과시키기 위한 가장 단순한 코드를 작성했다.

```typescript
// src/examples/tdd-example.ts
validate(password: string): boolean {
    if (password.length < 8) return false;
    return true;
}
```

이런 식으로 숫자 포함 여부, 대문자 포함 여부 등에 대한 테스트를 **하나씩 추가**하고, 그때마다 **최소한의 구현**을 덧붙였다.

---

## 3. TDD가 주는 심리적 안정감

실습을 진행하며 가장 크게 느낀 점은 **'심리적 안정감'**이었다. 

-   **불안감 해소**: 테스트가 나를 지켜주고 있다는 느낌 덕분에 리팩터링 단계에서 대담하게 코드를 고칠 수 있었다.
-   **명확한 요구사항**: 테스트 코드를 먼저 짜는 행위 자체가 요구사항을 코드로 정의하는 과정(Specification)이 되어주었다.
-   **오버 엔지니어링 방지**: 지금 당장 필요한 테스트만 통과시키려 노력하다 보니, 나중에 필요할지 모를 기능을 미리 만드는 불필요한 사치를 줄일 수 있었다.

---

## 💡 마치며: 결국은 생산성을 위한 선택

처음에는 테스트 코드를 먼저 짜는 시간이 아깝게 느껴지기도 했다. 하지만 학습을 마칠 즈음에는, 나중에 발생할 버그를 잡기 위해 고군분투할 시간을 미리 당겨 쓰고 있다는 확신이 들었다.

TDD는 단순히 '테스트 기술'이 아니라 **'일하는 방식의 변화'**였다. 실패하는 테스트를 보며 다음에 할 일을 정의하고, 초록불(Green)을 보며 성취감을 느끼는 이 리듬에 익숙해진다면 훨씬 더 즐겁게 코딩할 수 있을 것 같다.

아직은 아주 간단한 로직에만 적용해본 수준이지만, 앞으로 비동기 로직이나 복잡한 비즈니스 정책을 구현할 때도 이 TDD의 리듬을 잃지 않고 도전해보고 싶다.

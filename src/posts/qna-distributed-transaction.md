## Q1. 마이크로서비스 환경에서 ACID 트랜잭션을 그대로 쓸 수 있나요?

**A.** 거의 못 씁니다. 보통 **BASE(Eventual Consistency)** 로 타협합니다.

이유는 두 가지입니다.
- 여러 서비스가 각자의 DB를 가지므로 한 번의 트랜잭션이 네트워크/타임아웃 위험을 동반.
- Kafka, S3, 외부 SaaS API 등 **트랜잭션을 지원하지 않는 리소스**가 거의 항상 끼임.

실무 기준 우선순위는 **Outbox + Saga 조합**이 압도적이고, 2PC는 거의 선택지에 두지 않습니다.

---

## Q2. 왜 2PC(XA)를 거의 안 쓰나요?

**A.** 마이크로서비스에 잘 맞지 않습니다.

- **Coordinator가 SPOF**: 코디네이터가 죽으면 참여자는 in-doubt 상태로 락을 잡고 무한 대기.
- **블로킹 프로토콜**: prepare~commit 사이 락 점유가 길어 처리량 급감.
- 컨테이너 환경에서 노드 셧다운이 흔해 in-doubt 트랜잭션 복구가 운영 부담.
- 트랜잭션을 지원하지 않는 리소스(Kafka, REST API)가 끼면 어차피 별도 보상 로직이 필요.

결과적으로 Saga가 사실상 표준입니다.

---

## Q3. Saga와 Outbox는 어떻게 다른가요?

**A.** 다른 층위의 문제를 풉니다. 경쟁 관계가 아닙니다.

- **Saga**: **비즈니스 흐름 제어**. 여러 단계 트랜잭션이 실패했을 때 보상 트랜잭션으로 되돌리는 패턴.
- **Outbox**: **인프라 신뢰성**. DB 커밋과 메시지 발행을 하나의 원자 동작처럼 보장.

즉, Outbox는 Saga의 한 스텝이 메시지를 안전하게 보내기 위해 쓰는 도구입니다. 두 패턴을 같이 쓰는 게 정석입니다.

---

## Q4. Saga의 Choreography와 Orchestration 중 무엇을 선택하시나요?

**A.** 단계 수와 보상 흐름의 복잡도로 결정합니다.

- **Choreography**: 각 서비스가 이벤트를 듣고 반응. 결합도 낮음. 단순한 흐름에 적합. 단계가 많으면 흐름 추적이 어려움.
- **Orchestration**: 중앙 오케스트레이터가 상태 머신처럼 지휘. 가시성과 디버깅에 유리. 단일 지점이 생기는 단점.

저는 **3단계 이내 = Choreography, 4단계 이상이거나 보상 흐름이 복잡 = Orchestration**으로 갑니다.

---

## Q5. Outbox는 CDC와 폴링 중 무엇을 쓰시나요?

**A.** 환경에 따라 다릅니다.

- **CDC(Debezium 등)**: 지연 작고 표준적. 다만 운영 부담(Kafka Connect, 스키마 관리).
- **폴링**: 스케줄러로 outbox 테이블 조회 후 발행. 단순하지만 지연이 polling interval만큼 발생.

인프라가 가벼운 환경이면 폴링으로 시작해도 충분히 견고합니다. **컨슈머는 반드시 멱등 처리**가 되어 있어야 중복 발행을 흡수할 수 있습니다.

---

## Q6. 실제 흐름을 한 번 그려 본다면?

**A.** "주문 → 결제 → 재고 차감"을 예로 들면:

1. 주문 서비스: 주문 생성 + outbox row 삽입을 **같은 트랜잭션**으로 커밋.
2. Outbox publisher가 `OrderCreated`를 Kafka로 발행.
3. 결제 서비스가 메시지 소비 → 결제 처리 후 자체 outbox로 `PaymentApproved` 발행.
4. 재고 서비스가 차감, 실패 시 `StockReservationFailed` 이벤트 → 결제 환불 보상 트랜잭션 트리거.
5. 모든 컨슈머는 **메시지 ID 기준 멱등성** 보장.

---

## Q7. 메시지 중복은 어떻게 막나요?

**A.** **컨슈머 측 멱등 키 테이블**이 정석입니다.

- 메시지 ID 또는 비즈니스 키를 UNIQUE 제약 컬럼에 저장.
- 이미 처리된 메시지면 무시 또는 동일 응답 반환.
- 키 + 처리 결과를 같은 트랜잭션에 커밋해야 정확.

"exactly-once delivery"는 사실상 환상에 가깝습니다. **at-least-once + 멱등 처리**가 현실적 답입니다.

---

## Q8. 보상이 어려운 작업(이메일 발송, 외부 결제)은 어떻게 처리하나요?

**A.** **최후에 배치**하거나 **2단계로 분리**합니다.

- 외부 부수효과는 saga의 마지막 단계로 미루기.
- "draft 상태로 만들고 confirm 단계에서만 진짜 발송"되도록 분리. confirm 전에 saga가 실패하면 draft를 폐기하면 그만.
- 정말 보상 불가면 **수동 운영 리커버리** 절차를 사전에 문서화.

타임아웃과 saga 상태는 별도 테이블로 추적해 SRE 모니터링에 노출하는 것이 운영의 핵심입니다.

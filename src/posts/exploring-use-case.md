# 사용자 시나리오를 시스템의 언어로: 유스케이스(Use Case)의 탐색과 구현

![Use Case Concept](/public/images/use-case-concept.png)

소프트웨어를 개발하면서 가장 막막한 순간은 언제일까? 나에게는 "그래서 이 시스템이 정확히 '무엇'을 해야 하는가?"라는 질문에 명확히 답하지 못할 때였다. 기획서에는 수많은 버튼과 화면 구성이 나열되어 있지만, 그 이면에 숨겨진 사용자의 진짜 의도와 시스템이 제공해야 할 핵심 가치는 무엇인지 갈피를 잡기 어려울 때가 많았다.

단순히 "댓글 쓰기 기능이 필요하다"는 말만으로는 부족했다. 누가, 어떤 상황에서, 어떤 단계를 거쳐 댓글을 남기며, 시스템은 그 과정에서 어떤 규칙을 지켜야 하는가? 이 복잡한 실타래를 풀기 위해 검색과 공부를 거듭하다 마주친 고전적이면서도 강력한 도구가 바로 **유스케이스(Use Case)**였다.

아직 유스케이스를 완벽하게 명세화하고 다이어그램을 능숙하게 그려내는 전문가는 아니지만, 이번 탐색을 통해 내가 깨달은 유스케이스의 본질과 그 가치를 기록해보고자 한다.

---

## 1. 보이지 않는 가치를 가시화하기

처음 유스케이스를 접했을 때는 그저 UML 다이어그램 중 하나라고만 생각했다. 하지만 깊이 파헤쳐볼수록 이것이 단순한 그림이 아니라 **'사용자의 목표를 달성하기 위한 상호작용의 시나리오'**라는 점이 흥미로웠다.

### 액터(Actor)와 시스템의 경계
가장 먼저 배운 것은 **액터**의 개념이었다. 시스템 외부에서 시스템과 상호작용하는 모든 존재를 액터로 정의함으로써, "누구를 위한 기능인가?"를 명확히 할 수 있었다. 액터와 시스템 사이에 선을 긋는 과정(System Boundary)은 내 개발 범위가 어디까지인지를 정의하는 중요한 이정표가 되었다.

### 기능(Function)이 아닌 목표(Goal)
"댓글 저장 버튼 클릭"은 기능이지만, "댓글 작성"은 사용자의 목표다. 유스케이스는 사소한 UI 조작이 아닌, 사용자가 시스템을 통해 얻고자 하는 궁극적인 가치에 집중하게 해준다. 이 관점의 변화는 코드의 구조를 설계할 때도 큰 영향을 미쳤다.

---

## 2. 유스케이스를 코드로 형상화하기

유스케이스라는 개념이 설계 단계에서 끝나는 것이 아니라 코드까지 이어질 수 있을지가 궁금했다. 클린 아키텍처나 DDD 관련 아티클을 참고하여, '댓글 작성'이라는 목표를 하나의 유니크한 클래스로 분리해 보는 실험을 진행했다.

```typescript
// src/examples/use-case-example.ts

export interface UseCase<TRequest, TResponse> {
  execute(request: TRequest): Promise<TResponse>;
}

/**
 * '댓글 작성' 유스케이스 구현체
 * 액터(사용자)가 댓글을 작성하려는 목표를 달성하기 위한 시나리오를 담고 있습니다.
 */
export class CreateCommentUseCase implements UseCase<CreateCommentRequest, Comment> {
  constructor(private readonly commentRepository: CommentRepository) {}

  async execute(request: CreateCommentRequest): Promise<Comment> {
    // 1. 유효성 검사 (비즈니스 규칙)
    if (!request.content.trim()) {
      throw new Error('댓글 내용은 비어 있을 수 없습니다.');
    }

    // 2. 도메인 객체 생성 및 저장 (시스템 상태 변경)
    const comment = { ... };
    await this.commentRepository.save(comment);

    // 3. 결과 반환
    return comment;
  }
}
```

이 시도의 가장 큰 수확은 **'비즈니스 로직의 응집'**이었다. 예전에는 컨트롤러나 서비스 레이어 여기저기에 흩어져 있던 규칙들이 `CreateCommentUseCase`라는 하나의 장소로 모였다. 테스트 코드를 작성할 때도 "사용자의 댓글 작성 시나리오가 올바르게 작동하는가?"라는 관점에서 훨씬 직관적으로 접근할 수 있었다.

---

## 💡 마치며: 도구보다 중요한 것은 태도

유스케이스를 탐색하며 깨달은 가장 큰 교훈은, 화려한 다이어그램 작성 기법보다 **"사용자가 이 시스템으로 무엇을 하려 하는가?"**를 끊임없이 묻는 태도가 더 중요하다는 사실이다.

때로는 유스케이스 다이어그램 한 장보다 잘 작성된 시나리오 한 줄이 더 명확한 지침이 될 수 있음을 알게 되었다. 이번 학습은 단순히 요구사항을 코드로 옮기는 '코딩 머신'이 아니라, 사용자의 의도를 시스템의 언어로 번역하는 '설계자'로서의 고민을 시작하게 된 소중한 계기가 되었다.

다음에는 이 유스케이스들을 어떻게 더 체계적으로 관리하고, 특히 오류 발생 시의 '대안 흐름(Alternative Flow)'을 코드에 어떻게 우아하게 녹여낼 수 있을지를 깊이 고민해 볼 계획이다.

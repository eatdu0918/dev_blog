## 핵심 요약

두 축은 **다른 질문**입니다.

- **블로킹 vs 논블로킹**: 호출자가 결과를 **기다리며 멈추는가**? (제어가 즉시 돌아오는가)
- **동기 vs 비동기**: 호출자가 결과를 **누가 챙기는가**? (호출자가 직접 vs 콜백/이벤트로 받음)

흔히 "동기 = 블로킹"으로 단순화되지만 4가지 조합이 모두 가능합니다.

## 4가지 조합

| | 블로킹 | 논블로킹 |
|---|---|---|
| **동기** | 가장 흔한 호출. read()가 데이터 올 때까지 멈춤 | read()가 즉시 반환하지만 호출자가 polling으로 직접 확인 |
| **비동기** | "비동기"인데 결과를 받기 전까지 대기(드물지만 select 같은 다중화) | epoll/io_uring 같이 결과를 콜백/이벤트로 받음 |

핵심 사례:
- **동기 블로킹**: `data = file.read()` — 읽을 때까지 멈춤.
- **동기 논블로킹**: 소켓을 `O_NONBLOCK`으로 설정 후 `read` → 즉시 반환, 안 왔으면 직접 다시 폴링.
- **비동기 논블로킹**: epoll/kqueue/io_uring + 콜백. Node.js, Netty의 모델.

## 왜 이 구분이 중요한가

웹 서버의 동시성 모델 차이로 직결됩니다.

- **Thread-per-request (동기 블로킹)**: 톰캣 기본. 직관적이지만 스레드 수 = 동시 처리 한계.
- **이벤트 루프(비동기 논블로킹)**: Node.js, Netty. 스레드 1~수개로 수만 커넥션. 단 한 콜백이 CPU를 잡으면 전체 멈춤.
- **코루틴 / virtual thread**: 코드는 블로킹처럼 짜고 런타임이 비블로킹으로 변환. Go, Kotlin, Java 21+. **동기 블로킹의 가독성 + 비동기의 효율** 절충.

## I/O 관점의 모델 (Linux)

- **select / poll**: 모든 fd를 매번 스캔, O(n).
- **epoll**: 변경된 fd만 리포트, O(1). 대규모 커넥션에 적합.
- **io_uring**: 커널-유저 공유 링 버퍼로 시스템콜 비용 감소, 진정한 비동기 I/O.

## 자바스크립트의 비동기

- **이벤트 루프 + 마이크로태스크 큐**: Promise는 마이크로태스크 — `setTimeout(0)`보다 먼저 실행.
- `async/await`는 **Promise의 syntactic sugar**. 코드는 동기처럼 보이지만 실제로는 비동기 비블로킹.
- await 지점에서 함수가 일시 정지 → 이벤트 루프는 다른 일을 함.

## 자주 헷갈리는 디테일

- "비동기는 무조건 빠르다" → CPU 바운드 작업에는 의미 없음. I/O 대기 시간을 다른 일로 채울 때 가치.
- "Node.js는 싱글 스레드라 동시성이 약하다" → I/O 동시성은 강함. CPU 작업은 worker_threads로 분리 필요.
- 코루틴은 OS 입장에서 스레드 한 개 안에서 일어나는 협력 스케줄링. **블로킹 시스템콜을 하면** 코루틴 전체가 멈출 수 있음(런타임이 잘 처리하지 않으면).

## 실무 선택 기준

- **단순한 사내 시스템, RPS 낮음**: thread-per-request로 충분. 디버깅도 쉬움.
- **고성능 게이트웨이/프록시**: 이벤트 루프(Netty, Nginx).
- **많은 동시성 + 가독성 양립**: virtual thread / 코루틴.
- **CPU 바운드 워커**: 멀티프로세스 + 큐.

## 면접 follow-up

- "Promise.all과 Promise.allSettled의 차이?" → all은 하나라도 실패 시 reject. allSettled는 전부 결과 반환.
- "Java의 CompletableFuture와 reactive(Mono/Flux)의 차이?" → Future는 단일 값, Reactive는 스트림 + 백프레셔.
- "이벤트 루프에서 무거운 동기 함수가 들어오면?" → 모든 콜백이 지연. CPU 작업은 별도 스레드/프로세스로 오프로드.

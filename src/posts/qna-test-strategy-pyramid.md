## 핵심 요약

테스트 관련 면접 질문은 거의 항상 "**테스트 종류 구분 + 어떤 비율로 운영하는지 + 모킹은 어떻게 하는지**"로 모입니다.

- **테스트 피라미드**: 빠른 단위 多 → 통합 中 → 느린 E2E 少.
- **트로피 모델(Kent C. Dodds)**: 프론트엔드는 통합 테스트 비중을 더 높임.
- 정답 비율은 없음. **결함이 어디서 자주 새는가**로 결정.

## 테스트의 종류

### 단위 테스트(Unit Test)

- 클래스/함수 1개 단위. 외부 의존성 모킹 또는 인메모리.
- 빠름(ms). 수천 개 가능.
- 강점: 회귀 방지, 리팩토링 안전망.
- 약점: 단위는 통과해도 **연결되면 깨질 수 있음**.

### 통합 테스트(Integration Test)

- 여러 컴포넌트가 협력하는 단위. DB, 외부 API, 메시지 큐 포함.
- Testcontainers, in-memory H2, WireMock 등으로 실제에 가깝게.
- 백엔드는 **Repository + Service + DB**까지 묶는 슬라이스 테스트가 가성비 최고.

### E2E 테스트(End-to-End)

- 실제 브라우저/모바일에서 사용자 시나리오 전체.
- Playwright, Cypress, Selenium.
- 느림(초~분). flaky하기 쉬움.
- **핵심 사용자 경로 5~20개**만. 모든 케이스 커버 시도는 함정.

## 비율: 피라미드 vs 트로피

- **피라미드**: 단위 70 / 통합 20 / E2E 10. 백엔드/일반 앱.
- **트로피**: 정적분석 + 단위 20 / 통합 50 / E2E + 컴포넌트 10. 프론트엔드는 **DOM과 상호작용하는 통합**이 진짜 가치.

수치 자체보다 **"단위에서 잡히는 회귀가 많은가, 통합에서 새는가"**를 보고 조정.

## 모킹 전략

- **자기 코드 모킹 금지에 가까움**: 내가 짠 클래스를 모킹하면 구현 결합 테스트가 됨.
- **외부 시스템(서드파티 API, 시간, 랜덤, 결제)**은 모킹 또는 fake로 격리.
- **DB 모킹은 보통 안티패턴**. 실 DB(또는 컨테이너) 사용 권장 — SQL 행동까지 검증해야.
- **시간/UUID**는 인터페이스로 추상화 후 테스트에서 고정값 주입.

## 좋은 테스트의 속성 (FIRST)

- **F**ast: 빠르게.
- **I**ndependent: 순서 무관, 격리.
- **R**epeatable: 환경과 무관하게 같은 결과.
- **S**elf-validating: 통과/실패가 자동.
- **T**imely: 코드 작성 시점에 함께.

## 테스트가 신뢰를 잃는 순간

- **flaky 테스트**: 가끔 실패. 원인은 시간, 네트워크, 순서 의존, 공유 상태. 즉시 격리 또는 수정.
- **구현에 결합된 테스트**: 함수 이름만 바꿔도 깨짐. 행위 기반(behavior-based) 검증으로 전환.
- **항상 통과하는 테스트**: assert가 약하거나 실제 분기를 안 탐. mutation testing으로 검증.

## TDD를 꼭 해야 하나

면접에서 자주 나오는 질문. 솔직한 답:

- TDD는 도구지 신앙이 아님.
- 도메인이 잘 보이지 않을 때 **설계 발견 도구**로 유용.
- UI/실험성 코드는 비효율적일 수 있음.
- 핵심은 "**테스트가 있는 상태로 머지된다**". 작성 시점은 팀이 결정.

## 자주 헷갈리는 디테일

- 커버리지 80% = 품질 보장 X. 커버리지는 "**안 본 곳**" 찾는 도구.
- 단위 테스트가 빠르다고 비율을 무조건 높이는 건 함정. 통합에서 새는 결함이 많은 워크로드는 통합 비중을 키워야.
- 스냅샷 테스트는 가볍지만 **변경의 의미를 잃기 쉬움**. 화이트리스트.

## 면접 follow-up

- "단위 테스트와 통합 테스트의 경계?" → 외부 I/O(DB, 네트워크, 파일)가 들어가면 통합. 정의는 팀마다 합의 필요.
- "flaky 테스트를 어떻게 줄였나?" → 시간 추상화, 명시적 wait 대신 조건 wait, 데이터 격리(트랜잭션 롤백/스키마 분리).
- "테스트가 너무 느리면?" → 병렬화, 의존성 슬라이스 분리, 컨테이너 재사용, 영향 받은 테스트만 실행(test impact analysis).

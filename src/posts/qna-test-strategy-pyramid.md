## Q1. 단위/통합/E2E 테스트의 차이를 설명해 주세요.

**A.** 검증 범위가 다릅니다.

- **단위 테스트**: 클래스/함수 1개 단위. 외부 의존성 모킹/인메모리. 빠름(ms).
- **통합 테스트**: 여러 컴포넌트가 협력. DB, 외부 API 포함. Testcontainers, WireMock 활용.
- **E2E**: 실제 브라우저/모바일에서 사용자 시나리오 전체. Playwright, Cypress. 느림(초~분).

뒤로 갈수록 **현실에 가깝지만 느리고 flaky해집니다**.

---

## Q2. 어떤 비율로 가져가시나요?

**A.** 두 가지 모델이 대표적입니다.

- **테스트 피라미드**: 단위 70 / 통합 20 / E2E 10. 백엔드/일반 앱.
- **트로피 모델**(Kent C. Dodds): 정적분석 + 단위 20 / **통합 50** / E2E + 컴포넌트 10. 프론트엔드.

정답 비율은 없습니다. **결함이 어디서 자주 새는가**로 조정합니다. 단위에서 잡히는 회귀가 많으면 단위를 늘리고, 통합에서 새면 통합을 늘립니다.

## Q3. 모킹은 어떻게 결정하시나요?

**A.** 4가지 원칙입니다.

- **자기 코드는 모킹 금지에 가까움**: 내가 짠 클래스를 모킹하면 **구현 결합 테스트**가 됨.
- **외부 시스템(서드파티 API, 결제, SMS)** 은 모킹 또는 fake.
- **DB는 모킹 안티패턴**: Testcontainers나 실제 DB. SQL 행동까지 검증해야 함.
- **시간/UUID/랜덤**: 인터페이스로 추상화 후 테스트에서 고정값 주입.

"무엇을 모킹할지"가 아니라 "**무엇을 진짜로 테스트할지**" 관점이 더 정확합니다.

## Q4. 좋은 테스트의 속성은 무엇인가요?

**A.** **FIRST 원칙**입니다.

- **F**ast: 빠르게.
- **I**ndependent: 순서 무관, 격리.
- **R**epeatable: 환경 독립적.
- **S**elf-validating: 통과/실패 자동.
- **T**imely: 코드 작성 시점에 함께.

이 5가지가 모두 깨지면 테스트가 짐이 됩니다.

---

## Q5. flaky 테스트를 어떻게 줄이시나요?

**A.** **원인 식별이 출발점**입니다.

흔한 원인:
- **시간 의존**: 시계 추상화 + 고정 시간 주입.
- **명시적 wait**: `sleep` 대신 **조건 wait**(`waitFor`).
- **순서 의존**: 테스트 격리, 트랜잭션 롤백.
- **공유 상태**: 테스트 간 데이터 격리(스키마 분리, 트랜잭션 롤백).
- **네트워크**: 외부 의존을 fake로 대체.

flaky 테스트를 그대로 두면 신뢰가 무너지므로 **즉시 격리 또는 수정**해야 합니다.

## Q6. 구현에 결합된 테스트가 뭔가요?

**A.** **함수 이름이나 내부 구조를 바꾸면 깨지는 테스트**입니다.

```javascript
// ❌ 구현 결합
expect(service.calculatePrice).toHaveBeenCalledWith(...);

// ✅ 행위 기반
expect(result.totalAmount).toBe(150);
```

리팩토링 시 테스트가 같이 깨지면 **"테스트가 사양인지 구현 사본인지"** 의심해야 합니다. 행위 기반(behavior-based)으로 짜면 내부 변경에 강합니다.

## Q7. 테스트 커버리지를 얼마로 가져가시나요?

**A.** **목표 자체보다 안 본 곳을 찾는 도구**로 봅니다.

- 커버리지 80% = 품질 보장 X. assert가 약하거나 분기를 안 타도 줄은 실행됨.
- "어디가 테스트 없는지" 가시화하는 게 본질.
- **mutation testing**(Stryker, PIT)으로 진짜 검증력을 측정 가능.

핵심 도메인(결제, 인증)은 95%+, 단순 CRUD는 70%대도 충분 — 영역마다 합리적 수준이 다릅니다.

## Q8. TDD를 꼭 해야 하나요?

**A.** **도구이지 신앙이 아닙니다**.

가치 있는 경우:
- 도메인이 잘 보이지 않음 → **설계 발견 도구**.
- 명세가 명확한 알고리즘 구현.

비효율적일 수 있는 경우:
- UI 탐색적 개발.
- 실험성/스파이크 코드.

핵심은 **"테스트가 있는 상태로 머지된다"** 입니다. 작성 시점은 팀이 결정합니다.

---

## Q9. 단위 테스트와 통합 테스트의 경계가 모호하면 어떻게 하나요?

**A.** **외부 I/O 유무**로 잡으면 단순합니다.

- DB, 네트워크, 파일시스템이 들어가면 **통합**.
- 메모리에서만 끝나면 **단위**.

다만 정의는 팀마다 합의가 우선이고, 핵심은 **"같은 단어를 같은 의미로 쓰는가"** 입니다. 한 팀에서 "단위"라고 부르는 게 다른 팀의 "통합"일 수 있습니다.

## Q10. 테스트가 너무 느려졌을 때 어떻게 줄이시나요?

**A.** 4가지 도구가 있습니다.

1. **병렬화**: 테스트를 여러 워커로.
2. **의존성 슬라이스**: 필요한 빈만 로딩(Spring `@WebMvcTest`).
3. **컨테이너 재사용**: Testcontainers `withReuse(true)`.
4. **Test impact analysis**: 변경된 코드에 영향받는 테스트만 실행. Bazel, Nx, Gradle test selection.

CI 시간이 10분을 넘기 시작하면 개발 속도가 눈에 띄게 떨어지므로 적극 투자할 만한 영역입니다.

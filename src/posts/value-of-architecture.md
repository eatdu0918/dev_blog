# 기능 구현의 외주화

개발을 처음 시작했을 때는 프로젝트를 완성하고 기능이 정상적으로 동작하게 만드는 것만으로도 충분히 만족스러웠다. "사용자가 입력한 정보가 화면에 잘 나오고, DB에 잘 저장되면 된 것 아닌가?"라는 생각이 지배적이었다. 요구사항이 들어오면 바로 키보드를 잡고 로직을 한 땀 한 땀 작성해 나가는 과정이 즐거웠다.

하지만 인공지능 에이전트와 함께 협업하는 개발 환경으로 넘어오면서, 나의 역할에 대한 고민은 더 깊어졌다. 에이전트는 내가 한 시간 동안 작성할 로직을 단 몇 초 만에, 심지어 더 정확하게 구현해낸다. 이제 단순한 '기능 구현' 능력만으로는 생산성의 차별화를 만들기 어려운 시대가 된 것이다.

## 1. 에이전트가 로직을 짜고, 인간은 구조를 설계한다

최근 진행한 프로젝트에서 단순한 회원가입 기능에 '이메일 인증'과 '마케팅 동의 확인'이라는 새로운 요구사항이 추가되었을 때, 나는 직접 코드를 타이핑하는 대신 에이전트에게 구현을 맡겼다. 에이전트는 복잡한 벨리데이션 로직과 외부 API 연동 코드를 순식간에 작성했다.

하지만 에이전트가 짠 코드가 아무리 정확하더라도, 그것이 전체 프로젝트의 흐름과 맞지 않거나 확장이 불가능한 구조라면 결국 부채가 된다. Agent가 생산한 결과물을 개발자가 우리 프로젝트에 접합한지 검증하는 것이 부채를 해소할 수 있는 방법이 된다고 생각한다. **에이전트에게 어떤 '구조(Architecture)' 위에서 코드를 작성하라고 명령할 것인지, 그리고 생성된 결과물이 그 설계에 부합하는지 '검증(Validation)'하는 단계**가 생산성을 극대화하는 핵심 전략임을 깨달았다.

---

## 2. 아키텍처: 코드가 머무를 '방'을 정해주는 일

이 문제를 해결하기 위해 고민하던 중, 소프트웨어 아키텍처에 대한 이론적인 공부를 시작했다. 핵심은 **'관심사의 분리(Separation of Concerns)'**였다. 단순히 로직을 나열하는 것이 아니라, 에이전트가 구현할 개별 로직들이 자리 잡을 수 있는 탄탄한 설계를 먼저 만드는 것이었다.

### 구조적인 코드의 예시

아래는 내가 설계한 구조 위에서 에이전트가 구현한 회원가입 로직의 예시다. 나는 에이전트에게 "UserRepository와 EmailService를 주입받는 UseCase 계층을 만들어줘"라고 명확한 구조적 가이드라인을 제시했다.

```typescript
/* test/architecture-demo.ts */
// 유스케이스 계층: 비즈니스 흐름 제어 (Infrastructure에 의존하지 않음)
export class RegisterUserUseCase {
  constructor(
    private userRepository: UserRepository,
    private emailService: EmailService
  ) {}

  async execute(email: string, name: string): Promise<UserEntity> {
    const user = new UserEntity(email, name);

    // 중복 확인, 저장, 메일 발송이라는 각기 다른 책임을 분리된 인터페이스를 통해 수행
    const existing = await this.userRepository.findByEmail(email);
    if (existing) throw new Error('Already exists');

    await this.userRepository.save(user);
    await this.emailService.sendWelcome(user.email);

    return user;
  }
}
```

이렇게 계층을 분리하고 보니, 에이전트가 생성한 코드를 검증하고 테스트하는 과정이 훨씬 수월해졌다.

## 3. 이론적 배경이 주는 확신: 생산성의 새로운 기준

에이전트가 로직을 담당하게 되면서, 개발자에게 필요한 역량은 '코드를 치는 법'에서 '코드를 설계하고 검증하는 법'으로 이동하고 있다. 이를 위해서는 자바스크립트의 엔진 동작 원리, 객체지향 설계 원칙, 혹은 테스트 주도 개발(TDD)과 같은 이론적 배경이 더더욱 중요해진다.

이론적 기반이 탄탄할 때 비로소 에이전트에게 "이 부분은 SOLID 원칙 중 단일 책임 원칙을 어기고 있으니 다시 설계해줘"라고 명확하게 피드백을 줄 수 있고, 생성된 코드의 잠재적인 결함을 빠르게 파악하여 리스크를 줄일 수 있기 때문이다.

---

## 결론: 설계와 검증의 힘

이제 개발자의 생산성은 "얼마나 많은 코드를 짜는가"가 아니라, "에이전트라는 강력한 도구를 활용해 얼마나 견고한 시스템을 설계하고 검증하는가"에 달려 있다는 사실을 확인했다. 

기능 구현이라는 벽돌 쌓기는 에이전트에게 맡기고, 나는 이 집이 무너지지 않도록 지탱하는 설계도를 정교하게 다듬고 완성된 결과물이 설계도대로 만들어졌는지 꼼꼼히 검토하는 '검증자'로서의 역할에 집중해야 한다. 앞으로도 원리와 구조에 대한 탐구를 통해, 변화하는 환경 속에서 더 높은 가치를 만드는 개발자가 되고 싶다.

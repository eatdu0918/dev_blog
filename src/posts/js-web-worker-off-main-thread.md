# 브라우저 메인 스레드 병목 해결: Web Worker를 활용한 연산 분리와 OMTA 구조 도입 사례

자바스크립의 특징을 논할 때 가장 먼저 앵무새처럼 되뇌이는 주문. "자바스크립트는 싱글 스레드(Single-Threaded)라서 어쩌고, 비동기 처리가 저쩌고." 이 말을 수십 번 외치고 다녔지만, 내가 진짜 이 ‘싱글 스레드’라는 단어의 공포를 체감한 적이 있었나? 

모든 참사는 사내 어드민 페이지에 "30만 줄짜리 엑셀 JSON 배열을 브라우저에 파싱해서 복잡하게 교차 맵핑(Sorting)하는 기능"을 구현하던 오후에 발생했다.
서버 부하를 줄이겠다고 호기롭게 "그 정도 파싱은 클라이언트 브라우저에서 다 처리 가능합니다!"라고 큰소리치고 로직을 짰다. 그리고 테스트로 30만 건의 데이터를 집어넣은 순간, 내 맥북 브라우저는 비명을 지르고 화면은 시체처럼 굳어버렸다. 

스피너(로딩 애니메이션)조차 멈췄고, 버튼을 수십 번 눌러도 아무 반응이 없었다. 급기야 브라우저는 "오류: 응답 없는 페이지입니다. 중지하시겠습니까?"라는 끔찍한 알럿 창을 얼굴에 토해냈다.
나는 그제야 Event Loop 큐(Queue)에 데이터를 비동기로 던져봤자 아무 소용없다는 진리를, 이 무거운 데이터 분쇄 작업이 스택에 올라탄 4초 내내 **"스크립트를 해석하고 화면(UI)에 페인트를 칠하는 단 하나의 메인 CPU(스레드)가 숨통을 부여잡고 점유"** 당해 버렸다는 뼈아픈 현실을 뒤늦게 깨달았다.

## 1. 빛의 속도 뒤의 그림자, 동시성에 대한 무지함 해결

백엔드 동료에게 "아, 브라우저가 버티질 못하니 그냥 서버에서 다시 페이징 해서 내려주심 안 될까요?"라고 읍소할까 고민하던 중, 내 오기를 건드린 건 어느 컨퍼런스 영상에서 지나치듯 들은 키워드였다.
*"무거운 연산을 멀티 스레드로 빼버리면 그만이지 않습니까."*

그렇다, 내겐 비장의 무기 **'웹 워커(Web Worker)'** 가 있었다.
웹 워커는 메인 스레드(UI)의 화면 칠하기 작업과 완전히 물리적으로 독립된 다른 백그라운드 코어 프로세서에서 자바스크립트를 몰래 돌려주는 엔진이다.

### 2. 고립된 밀실과 우편배달부(postMessage)의 도입

처음 워커를 띄우며 마주한 제약은 참 까다로웠다. 워커 엔진 내부는 철저히 고립된 밀실이었다. 
- 그곳에선 화면 객체인 `window`나 `document.getElementById` 따위의 DOM 접근이 일절 금지되어 있었다. 오로지 피도 눈물도 없는 '순수 데이터(JSON, 문자열, 버퍼)의 압그레이드 노역장'일 뿐이었다.
- 게다가 공용 자원(Shared Memory)을 쓸 수 없어, 철저하게 양 끝단에서 데이터를 택배 포장(`postMessage`)해서 던져주고 콜백으로 다시 덩어리를 받아채는 메시지 패싱 방식을 고수했다.

하지만 내가 그 구조를 코드로 녹여낸 순간의 희열은 엄청났다!

```javascript
/* [메인 컴포넌트 진영 - 평화롭게 UI만 그리는 곳] */

const massiveData = [...어마어마한 30만 건 모음집];
const worker = new Worker("heavySorter.js"); // 외주 공장장 고용!

// 1. 반장님에게 복잡한 정리 정돈 작업을 몽땅 던져버림
worker.postMessage(massiveData); 

// (메인 스레드는 데이터를 던지자마자 이 라인을 쌩 지나가며, 
//  멈춤 없이 화면에 빙글빙글 스피너 애니메이션을 부드럽게 유지한다!!)

// 3. 저 멀리 백그라운드 공역장에서 3초 뒤에 완료된 결과물을 던져주면 탁 받는다.
worker.onmessage = (event) => {
    const sortedResult = event.data;
    console.log("계산이 1도 끊김 없이 무사히 완료! 화면에 뿌리자.");
};
```

```javascript
/* [heavySorter.js - 빛이 닿지 않는 어둠의 백그라운드 공단] */

self.onmessage = function(e) {
    const rawData = e.data;
    
    // 이 끔찍하고 거대한 N^2 정렬 작업이 CPU를 90% 이상 갉아먹으며 폭주해도,
    // 메인(UI) 쪽의 부드러운 스크롤은 전혀 눈 하나 깜짝하지 않는다.
    const result = heavyDeepMappingAndSort(rawData); 
    
    // 외주 작업 끝! 사장님한테 결과물 택배로 발송
    self.postMessage(result);
};
```

## 3. OMTA(Off-Main-Thread Architecture)의 깊은 바다를 향해

이 삽질 덕분에 무사히 사내 어드민 페이지는 화면 얼어붙음 ZERO라는 찬사와 함께 배포되었다. 그리고 나는 메인 스레드만을 미친 듯이 채찍질하던 편협한 시야에서 벗어나, 이제 대규모 프론트엔드 환경에서 논의되는 **OMTA(Off-Main-Thread Architecture)** 라는 아키텍처 사상까지 기웃거리고 있다.

메시지 패싱 비용(직렬화/복사)을 획기적으로 없애기 위해 물리적 메모리를 잘라쓰는 `SharedArrayBuffer` 나, 무거운 차트 및 지도 그래픽을 렌더링하기 위한 `OffscreenCanvas` 등. 브라우저가 싱글 스레드의 감옥에서 벗어나기 위해 치열하게 발전시켜 놓은 탈출구들을 비로소 맛보았다.

## 맺으며

예전엔 프론트엔드 성능 최적화라 하면 그저 무거운 이미지를 webp로 줄이고, React `useMemo`나 `useCallback` 옵션을 달달 외워 빈틈없이 도배하는 것(Micro-optimization)인 줄로만 알았다.

하지만 진정한 퍼포먼스의 도약은 "애당초 화면을 담당하는 메인 스레드에서 이런 무자비한 계산을 돌려 병목을 사서 만든 구조적 오만함"을 도려내는 결단력에 있었다. 로딩 바가 굳어버린 침묵의 크래시 현상 앞에서 애꿎은 비동기 코드만 만지작거리던 내 모습이 부끄럽게 느껴진다. 
지금 당신의 프론트가 헉헉거리며 유저 경험의 허리를 끊고 있다면, 그 지옥문을 봉쇄할 저 뒷방의 구원자, 웹 워커를 진지하게 마주할 때다.

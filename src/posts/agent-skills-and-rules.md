# 에이전트의 능력과 질서를 잡아주는 Skills와 Rules 설계하기

최근 AI 에이전트와 페어 프로그래밍을 하거나 업무를 자동화하면서 가장 크게 느낀 갈증은 '예측 가능성'이었다. 에이전트에게 매번 "답변은 한국어로 해줘", "코드 스타일은 이렇게 유지해줘"라고 반복해서 말하는 것은 비효율적일 뿐만 아니라, 에이전트가 문맥이 길어지면 이를 망각하는 경우도 잦았다.

이러한 문제를 해결하기 위해 에이전트의 시스템을 **질서(Rules)**와 **능력(Skills)**이라는 두 가지 축으로 구조화하며 얻은 깨달음을 정리해본다.

## 질서(Rules): 행동의 제약 조건 설정하기

에이전트가 지켜야 할 '최소한의 가이드'나 '스타일 가이드'를 설정하는 과정이다. 이를 프롬프트에 매번 포함시키는 대신, 별도의 규칙 파일로 관리하니 에이전트의 행동이 훨씬 일관되게 변했다.

예를 들어, 우리 프로젝트의 커밋 컨벤션이나 코드 작성 규칙을 `rules/` 디렉토리에 정의해두면, 에이전트는 작업을 시작하기 전 이 규칙들을 먼저 읽고 자신의 행동을 제어한다. 이는 마치 개발 팀의 코딩 컨벤션 문서를 공유하는 것과 같다. 규칙이 명확할수록 에이전트는 방황하지 않고 정해진 궤도 안에서 최적의 결과물을 만들어낸다.

---

## 능력(Skills): 확장 가능한 전문 기능 부여하기

모든 것을 에이전트의 일반적인 지능(LLM)에만 맡기기보다, 특정 작업을 위한 전용 도구와 스크립트를 제공하는 것이 훨씬 강력하다. 이를 'Skills'라고 부르는데, 복잡한 로직을 수행하는 쉘 스크립트나 외부 API 호출 등을 캡슐화한 단위다.

에이전트는 필요할 때 이 Skill을 호출하여 정교한 결과물을 만들어낸다. 직접 모든 코드를 짜게 하는 것보다, 이미 검증된 도구를 쥐여주는 것이 작업의 성공률을 비약적으로 높여주었다. 이는 에이전트가 단순히 '말하는 존재'에서 '행동하는 존재'로 진화하는 핵심 단계이기도 하다.

---

## 실전 예제: 시스템 구조화하기

에이전트가 어떻게 규정된 규칙 내에서 자신의 능력을 발휘하는지 간단한 코드로 시뮬레이션해 보았다. 규칙을 위반했을 때 예외를 발생시키고, 정해진 질서(Rules)를 사용하는 방식이다.

```md
# 커밋 가이드라인 (Commit Guidelines)

이 가이드라인은 프로젝트의 이력을 명확하게 관리하고, 의미 있는 단위로 작업을 나누어 기록하기 위한 규칙입니다.

## 1. 커밋 메시지 컨벤션
- **언어**: 커밋 메시지는 **한국어**로 작성합니다.
- **형식**: `태그: 요약` 형식을 사용합니다.
- **태그**:
    - `feat`: 새로운 기능 추가
    - `fix`: 버그 수정
    - `docs`: 문서 수정 (README, 블로그 포스팅 등)
    - `style`: 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음)
    - `refactor`: 코드 리팩토링
    - `test`: 테스트 코드 추가 및 수정
    - `chore`: 빌드 업무, 패키지 매니저 설정 등 (프로덕션 코드 변경 없음)
- **톤앤매너**: 명확하고 간결한 문장으로 작성하며, "~함", "~함 추가"와 같이 명사형 종결 어미를 권장합니다.

## 2. 작업 단위 그룹화 원칙
- **원자성(Atomicity)**: 하나의 커밋은 하나의 논리적인 변화만 담아야 합니다.
- **관련성**: 연관된 파일들(예: 기능 코드 + 해당 기능의 테스트 코드)은 하나의 작업 단위로 묶습니다.
- **독립성**: 각 커밋은 그 자체로 의미가 있어야 하며, 빌드가 깨지지 않는 상태를 유지해야 합니다.
- **대규모 변경 분할**: 한 번에 너무 많은 파일을 수정했다면, 이를 논리적인 단계(예: 데이터 모델 변경 -> 비즈니스 로직 적용 -> UI 반영)로 나누어 제안합니다.
```

이러한 상호작용은 에이전트가 `rules/` 디렉토리와 `skills/` 디렉토리 내의 파일들을 동적으로 참고함으로써 실현된다. 에이전트는 작업을 명령받으면 가장 먼저 규칙 파일을 로드하여 현재 작업이 허용된 범위 내에 있는지(Constraints) 확인하고, 정의된 스킬 파일을 읽어 어떤 도구와 사고방식을 사용해야 할지(Capabilities) 결정한다.

이 구조를 통해 에이전트는 '아키텍처 관점의 사고'라는 자신의 능력을 정확히 발휘할 수 있게 된다. 규칙은 에이전트가 넘지 말아야 할 선을 정해주고, 기술은 에이전트가 휘두를 수 있는 무기가 되는 셈이다.

## 결론 및 회고

에이전트를 단순히 질문을 던지는 대상이 아니라, 하나의 완성된 시스템으로 바라보기 시작하면서 협업의 질이 달라졌다. 강력한 지능보다 중요한 것은 그 지능이 발휘될 수 있는 '질서'와 '도구'라는 점을 이번 기회에 다시 한번 배울 수 있었다.

처음에는 에이전트가 알아서 잘 해줄 것이라 기대했지만, 결국 좋은 에이전트를 만드는 것은 좋은 환경을 설계하는 개발자의 몫이라는 사실을 깨닫게 되었다. 앞으로 더 복잡한 시스템을 구축하더라도 이 'Rules'와 'Skills'의 분리 원칙은 내가 에이전트와 협업하는 가장 견고한 기반이 될 것 같다.

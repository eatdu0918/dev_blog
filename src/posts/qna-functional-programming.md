## Q1. 함수형 프로그래밍의 핵심 개념을 짧게 설명해 주세요.

**A.** **데이터를 변경하지 않고 변환하는 패러다임**입니다. 5가지 개념이 핵심입니다.

1. **순수 함수**: 같은 입력 → 같은 출력 + 부수효과 없음.
2. **불변성**: 데이터를 바꾸지 않고 새 데이터 생성.
3. **일급 함수**: 함수가 값처럼 전달/반환됨.
4. **고차 함수**: 함수를 인자로 받거나 반환.
5. **선언형**: 무엇을 할지(what)에 집중, 어떻게(how)는 추상.

---

## Q2. 순수 함수의 장점이 뭔가요?

**A.** 4가지가 큽니다.

```javascript
// 순수
const add = (a, b) => a + b;

// 불순 — 외부 상태에 의존
let counter = 0;
const tick = () => ++counter;
```

- **테스트 쉬움**: 독립 실행 가능.
- **캐싱(메모이제이션) 가능**: 같은 입력에 같은 출력이 보장되니 결과 캐싱 안전.
- **병렬 안전**: 공유 상태 없음.
- **추론 쉬움**: 함수 안만 보면 됨.

## Q3. 불변성을 지키는 게 왜 중요한가요?

**A.** **변경 추적**과 **참조 비교**가 가능해지기 때문입니다.

React에서 상태 업데이트는 항상 새 객체로 합니다.
```javascript
setUser({ ...user, name: 'kim' }); // 새 객체
```

이러면 React는 참조 비교(`===`)만으로 변경을 감지할 수 있어 메모이제이션이 효율적입니다. mutation을 허용하면 deep equal 비교가 필요해 매우 비싸집니다.

라이브러리: **Immer**(proxy로 mutation처럼 짜고 내부에서 불변 객체 생성), **Immutable.js**.

## Q4. 고차 함수와 함수 합성은 어떻게 연결되나요?

**A.** 고차 함수는 **함수를 인자/반환**하는 함수, 함수 합성은 **여러 함수를 연결**하는 패턴입니다.

```javascript
const pipe = (...fns) => x => fns.reduce((v, f) => f(v), x);

const process = pipe(
  s => s.trim(),
  s => s.toLowerCase(),
  s => s.replace(/-/g, ''),
);
```

OOP의 상속/합성 대신 **함수 합성**으로 동작을 조립합니다. `map`, `filter`, `reduce`가 일상적인 고차 함수입니다.

## Q5. 명령형과 함수형 코드의 차이를 보여줄 수 있나요?

**A.** "어떻게"가 보이는가, "무엇을"이 보이는가의 차이입니다.

```javascript
// 명령형 — 단계가 보임
let total = 0;
for (let i = 0; i < items.length; i++) {
  if (items[i].active) total += items[i].price;
}

// 함수형 — 의도가 보임
const total = items
  .filter(i => i.active)
  .reduce((sum, i) => sum + i.price, 0);
```

함수형 코드가 항상 더 짧지는 않지만, 의도가 더 직관적입니다.

## Q6. 모나드라는 단어를 자주 듣는데 무엇인가요?

**A.** **값을 컨텍스트로 감싸 합성을 안전하게 만드는 추상**입니다. 이름을 외울 필요는 없습니다.

흔한 예시:
- **Maybe/Optional**: null 처리. `Optional.of(x).map(...)`로 null이면 자동 스킵.
- **Either/Result**: 성공/실패. Rust의 `Result`.
- **Promise**: 비동기. `then` chain이 사실상 모나드 패턴.

핵심 사고: **null/예외/비동기를 chain으로 합성**.

## Q7. 부수효과가 필요한 IO/렌더링은 함수형에서 어떻게 처리하나요?

**A.** **부수효과를 가장자리(edge)로 밀고 핵심 로직은 순수 유지**가 정석입니다.

- **React**: 컴포넌트는 함수, `useEffect`로 부수효과 격리.
- **Haskell**: IO 모나드로 효과를 타입에 명시.
- **Redux**: reducer는 순수, 부수효과는 미들웨어(thunk, saga).

핵심 로직이 순수하면 테스트와 추론이 압도적으로 쉬워집니다.

## Q8. OOP와 FP 중 어느 것이 더 좋다고 생각하시나요?

**A.** **도구의 선택**입니다. 한쪽이 더 우월하지 않습니다.

- **OOP**: 도메인 모델링, 행동을 가진 엔티티 표현에 자연스러움.
- **FP**: 데이터 변환, 파이프라인, 동시성에 자연스러움.

현실 코드는 거의 항상 혼용입니다. Java도 람다/Stream으로 함수형 도입, JS도 클래스/this로 OOP 가능. **문제에 맞는 도구**를 선택하는 게 본질입니다.

## Q9. 불변성의 성능 비용은 어떻게 해결하나요?

**A.** 새 객체를 만드는 비용은 분명 있지만, **구조 공유(persistent data structure)** 로 완화합니다.

- 변경되지 않은 부분은 새 객체와 공유, 변경된 노드만 새로 생성.
- Immutable.js, Immer 모두 이 기법을 사용.
- 큰 데이터셋에서 deep clone 대비 수십~수백 배 빠름.

map/filter 체이닝의 중간 배열 생성이 부담될 만큼 큰 데이터는 **transducer 패턴**이나 **단일 reduce로 결합**해 한 번에 처리합니다.

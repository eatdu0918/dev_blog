# 복잡한 도메인을 코드로 풀어내는 첫걸음: 도메인 주도 설계(DDD)와의 조우

![DDD Architecture Concept](/public/images/ddd-concept.png)

소프트웨어를 개발하다 보면 누구나 한 번쯤 마주하는 순간이 있다. 처음에는 간단했던 기능들이 하나둘 추가되면서, 어느새 비즈니스 로직이 데이터베이스 접근 코드와 뒤섞이고, UI 로직이 서비스 레이어까지 침범하며 거대한 '스파게티 코드' 덩어리가 되어가는 과정 말이다. 

나 역시 최근 진행 중인 프로젝트의 규모가 커짐에 따라 비슷한 갈증을 느꼈다. "어떻게 하면 비즈니스의 복잡함을 코드에 더 유연하고 유지보수하기 쉽게 담아낼 수 있을까?" 이 질문에 대한 답을 찾기 위해 검색을 거듭하다 마주친 것이 바로 **도메인 주도 설계(Domain-Driven Design, 이하 DDD)**였다.

아직 실무에 이 거대한 철학을 온전히 녹여본 경험은 없지만, 관련 아티클과 가이드들을 탐독하며 머릿속에 정리된 DDD의 핵심 조각들을 하나씩 풀어보고자 한다.

---

## 1. 전술보다 중요한 전략: 언어와 경계

DDD를 처음 접했을 때 가장 놀라웠던 지점은, 이 설계 방식이 단순히 '코드 잘 짜는 법'이 아니라는 사실이었다. DDD는 기술보다 **'비즈니스 그 자체(Domain)'**를 코드와 일치시키는 데 더 큰 방점을 찍고 있었다.

### 유비쿼터스 언어 (Ubiquitous Language)
가장 흥미로웠던 개념은 개발자와 기획자, 도메인 전문가가 서로 같은 단어를 사용해야 한다는 **보편적 언어**의 중요성이었다. 기획서에는 '승인'이라고 되어 있는데 코드에는 `updateStatus(2)`라고 되어 있다면, 거기서부터 오해와 버그가 시작된다는 지적은 뼈아프게 다가왔다.

### 바운디드 컨텍스트 (Bounded Context)
모든 문제를 해결하는 하나의 거대한 모델을 만드는 것이 아니라, 각기 다른 역할과 의미를 가진 영역(문맥)을 나누고 그 경계를 명확히 하는 것이 핵심이었다. 똑같은 '상품'이라 하더라도 주문 영역에서의 의미와 배송 영역에서의 의미가 다를 수 있음을 인정하고 경계를 긋는 방식은, 모놀리식 아키텍처에서 마이크로서비스로 넘어가는 징검다리 같은 느낌을 주었다.

---

## 2. 도메인을 코드로 형상화하는 전술적 도구들

이런 전략적 설계가 코드라는 구체적인 모습으로 나타날 때, 몇 가지 핵심적인 패턴들이 등장한다. 나는 이 패턴들을 이해하는 것이 DDD를 시작하는 열쇠라고 판단했다.

- **엔티티 (Entity)**: 고유한 식별자를 가지며, 상태가 변해도 정체성이 유지되는 객체 (예: 주문, 회원)
- **밸류 오브젝트 (Value Object, VO)**: 식별자 없이 그 속성값들이 같으면 같은 것으로 간주되는 불변 객체 (예: 주소, 금액, 전화번호)
- **애그리거트 (Aggregate)**: 서로 관련된 엔티티와 VO의 묶음. 이 묶음의 대장 격인 '애그리거트 루트'를 통해서만 데이터의 일관성이 유지된다.

---

## 3. 탐색한 개념을 코드로 모사해보기

글로만 읽는 것보다 직접 코드로 옮겨보는 것이 이해에 훨씬 도움이 될 것 같아, 주문(Order) 도메인을 주제로 아주 간략한 DDD 스타일의 코드를 작성해 보았다.

```typescript
// src/examples/ddd-order-example.ts 중 핵심 로직 발췌

export class Money {
  constructor(public readonly amount: number, public readonly currency: string = 'KRW') {
    if (amount < 0) throw new Error('금액은 0보다 작을 수 없습니다.');
  }

  add(other: Money): Money {
    if (this.currency !== other.currency) throw new Error('통화가 일치하지 않습니다.');
    return new Money(this.amount + other.amount, this.currency);
  }
}

export class Order {
  private _status: OrderStatus = OrderStatus.PENDING;

  constructor(
    public readonly id: string,
    private _totalPrice: Money,
    private _shippingAddress: Address
  ) {}

  // 도메인 비즈니스 로직을 엔티티 내부로 캡슐화
  ship(): void {
    if (this._status !== OrderStatus.PENDING) {
      throw new Error('대기 중인 주문만 배송할 수 있습니다.');
    }
    this._status = OrderStatus.SHIPPED;
  }
}
```

이 코드를 작성하며 흥미로웠던 점은 `Money`라는 단순한 데이터(숫자)를 **객체(VO)**로 바라보기 시작하니, 금액 합산 시 통화(Currency) 체크나 음수 방지 로직이 도메인 객체 안으로 자연스럽게 녹아들어 갔다는 점이다. 또한 `Order` 상태 변경 로직이 외부 서비스 레이어가 아닌 `Order` 엔티티 내부에 위치함에 따라, 객체 스스로가 자신의 비즈니스 규칙을 책임지게 된 모습이 인상적이었다.

---

## 💡 마치며: 보이지 않던 설계의 지도를 그리다

지금까지의 나는 시스템을 설계할 때 '데이터베이스 테이블을 어떻게 짤까?'를 가장 먼저 고민했다. 하지만 DDD라는 지도를 손에 쥐고 보니, 중요한 것은 테이블이 아니라 **'도메인의 규칙과 말'**을 어떻게 코드로 보호할 것인가에 있음을 어렴풋이 깨닫게 되었다.

물론 DDD는 학습 곡선이 매우 높고, 소규모 프로젝트에는 오히려 과할 수 있다는 경고도 많이 접했다. 하지만 도구의 원리를 알면 더 적절한 순간에 꺼내 쓸 수 있듯이, 이번 탐색은 나에게 '복잡함을 다루는 새로운 시각'을 선물해 주었다. 

아직은 멀리서 숲의 윤곽을 본 수준이지만, 언젠가 실제 프로젝트에서 이 정교한 설계의 맛을 제대로 느껴보고 싶다는 각오와 함께 오늘의 회고를 마친다.

# Next.js 블로그 아키텍처 구축과 마크다운 렌더링 확장기

개인 기술 블로그를 직접 설계하면서, 단순한 글쓰기 플랫폼을 넘어 내가 겪은 기술적 실험을 직접 구동해 볼 수 있는 공간으로 만들고자 했다. 이를 위해 **Next.js 16.1 (App Router)** 환경 위에서 별도의 데이터베이스 없이 **Markdown** 파일을 근간으로 하는 파일 기반 CMS(File-based CMS) 아키텍처를 채택했다.

하지만 평범한 텍스트 변환기를 넘어, 포스팅 내부에서 코드를 직접 실행하고 다이어그램을 렌더링하는 방향으로 파이프라인 구조를 확장하면서 겪은 고민의 과정을 기록한다.

---

## 1. 정적 블로그의 기본 기술 스택

블로그의 핵심 동력은 마크다운 파일을 HTML로 안전하고 빠르게 변환하는 것이었다.

*   **Framework**: Next.js 16.1 (App Router)
*   **Language**: TypeScript
*   **Markup Processing (통합 파이프라인)**:
    *   `remark`: 순수 Markdown 문법을 AST(추상 구문 트리)로 파싱하여 HTML 구조로 변환
    *   `unified`: 플러그인 생태계를 결합하여 커스텀 마크업 노드를 삽입하고 제어할 수 있는 기반 제공

단순 변환 단계에서는 문제가 없었지만, 코드 하이라이팅을 넘어서 브라우저 샌드박스 런타임 환경(`@codesandbox/sandpack-react`)과 동적 아키텍처 다이어그램 렌더러(`mermaid`)를 연동하려 하자, 기존의 일방향 텍스트 파싱 구조로는 한계에 부딪혔다.

---

## 2. 하이브리드 아키텍처 구조의 확장 도입

문제를 해결하기 위해, 단순히 문자열을 HTML로 치환하는 것을 넘어 특수한 마크업 블록을 감지해 리액트 컴포넌트로 마운트 시키는 **하이브리드 렌더링** 구조를 새롭게 설계했다.

### 파서 계층의 진화 (`parser.ts`)

플러그인 파이프라인 중간에 개입하여 커스텀 블록을 식별하는 전처리 단계를 추가했다.
1.  **Sandpack 블록 감지**: `:::sandpack` 그룹 텍스트를 감지하여, 내부에 작성된 다중 파일의 소스 코드와 실행 모드(Editor/Test) 메타데이터를 정형화된 JSON 객체로 추출해 냈다.
2.  **Mermaid 블록 감지**: ` ```mermaid ` 코드블록을 일반 텍스트가 아닌, 시각화 컴포넌트가 주입될 플레이스홀더 DOM 노드로 치환하여 변환기의 예외 규칙으로 등록했다.

### 렌더러 계층의 분리 (`components/`)

파서가 남겨둔 뼈대(플레이스홀더) 위에 실제 인터랙티브 UI를 덧씌우는 책임을 클라이언트 컴포넌트 단으로 이관했다.
*   **SandpackRenderer**: 포스팅 렌더링 시 HTML 내의 커스텀 데이터 속성을 역참조하여 미리 분리해 둔 소스 코드를 `SandpackWrapper` 프레임워크에 주입해 즉시 실행 가능한 가상 IDE를 렌더링했다.
*   **MermaidRenderer**: 플레이스홀더 텍스트를 `mermaid.js` 코어 엔진으로 넘겨 동적인 SVG 아키텍처 다이어그램으로 클라이언트 상에서 안전하게 그려냈다.

## 💡 최종 회고

단순한 개인 기록용 게시판을 넘어 "경험의 재생산"이 가능한 인터랙티브 블로그 생태계를 구축하는 과정에서, 파서(데이터 가공)와 렌더러(시각화 및 동작)의 관심사를 명확히 분리하는(Separation of Concerns) 아키텍처 설계의 중요성을 깊이 체감했다. 

이러한 모듈식 파이프라인 설계 덕분에 기존 하드 코딩 방식의 스파게티 로직을 피할 수 있었으며, 추후 새로운 3D 뷰어나 인터랙티브 차트 컴포넌트를 추가해야 할 때도 기존 파서 로직을 훼손하지 않고 유연하게 확장할 수 있는 탄탄한 뼈대를 완성하게 되었다.

# N+1 문제의 무서움과 극복: Fetch Join, EntityGraph, 그리고 BatchSize

처음 JPA(Java Persistence API) 코드를 짰던 날, SQL 없이 자바 객체만으로 데이터베이스를 휘젓는 모습에 감탄을 금치 못했다. 로컬 환경에서는 테스트용 데이터 10건, 20건이 전부인지라 페이지네이션이든 목록 조회든 눈 깜짝할 새에 화면이 떴다.

하지만 프로젝트가 커지고, 페이크 데이터 수백만 건을 밀어 넣은 후 부하 테스트를 진행하던 날 서버는 그대로 뻗어버렸다. 어드민 페이지에서 회원 목록 하나 조회했을 뿐인데 터미널에는 `Hibernate: select ...` 로그가 수천 줄 쏟아지며 스크롤을 미친 듯이 올리고 있었다. 바로 악명 높은 **N+1 문제**와의 뼈아픈 첫 만남이었다. 

더 나은 개발자가 되기 위해서는 "돌아가기만 하는 코드"가 아니라 이 N+1 폭격을 우아하게 진압해야 한다고 굳게 마음먹고, 해결 방법들을 하나씩 파고들기 시작했다.

---

## 1. N+1 문제의 원인, 그리고 어리석었던 첫 대처

현상은 다음과 같았다. 회원을 1,000명 불러오는 메인 쿼리(`findAll`) 1번이 실행된 후, 그 1,000명의 소속 팀 이름을 출력하기 위한 루프 안에서 지연 로딩 때문에 팀을 찔러보는 1,000번의 추가 쿼리가 터지고 있었다. (1 + N 번)

이 현상을 처음 마주했을 때 난 아주 순진하고도 치명적인 꼼수를 부렸다.
"어차피 팀 정보 무조건 같이 쓰는데, 그냥 매핑 어노테이션을 `@ManyToOne(fetch = FetchType.EAGER)`로 바꾸면 다 같이 가져오지 않나?"

바꾸고 며칠 뒤, 더 끔찍한 사태가 벌어졌다. 메인 쿼리만 필요한 상황에서도 EAGER 속성은 "아 맞다, 팀도 다 가져와야 해!" 라며 사방에서 쓰지도 않을 조인(Join)과 후속 쿼리들을 사정없이 폭발시키고 있었던 것이다. **모든 연관 관계의 Fetch 전략은 반드시 `FetchType.LAZY`(지연 로딩)로 고정해야 한다는 실무의 황금률**을 피눈물과 함께 배우게 된 순간이었다.

---

## 2. 문제를 직면하다: Fetch Join과 조작된 페이징

LAZY를 포기하지 않으면서 N+1을 해결하는 정석, 그 두 번째 관문은 바로 **Fetch Join**이었다. 특정 비즈니스 로직 단위 안에서 "어차피 이 쿼리는 무조건 회원과 소속 팀 데이터가 같이 필요하다"고 선언해 주는 것이다.

```java
// 프록시 말고 진짜 객체를 조인해서 한 방 쿼리로 긁어와주세요.
@Query("SELECT m FROM Member m JOIN FETCH m.team")
List<Member> findAllWithTeam();
```

또는 `@EntityGraph(attributePaths = {"team"})` 어노테이션을 활용해 쿼리 코드를 지루하게 쓰지 않고 훌륭히 조인해냈다. 
여기까지만 해도 나는 내가 N+1 문제를 마스터한 줄 알았다.

그러나 이번엔 1:N(`@OneToMany`), 즉 **컬렉션 패치 조인**의 늪이 나를 덮쳤다.
한 팀(Team)이 가진 멤버(Members) 컬렉션들을 Fetch Join하려고 하자 데이터가 SQL 상에서 곱하기(Cartesian Product) 연산으로 뻥튀기되었다. 

더 소름 돋았던 것은 페이징(`Pageable`) 처리를 같이 넘길 때 등장한 빨간 글씨의 메모리 경고문이었다.
`HHH000104: firstResult/maxResults specified with collection fetch; applying in memory!`

JPA는 1대다 조인에서 쿼리단에 `LIMIT`을 걸어버리면 데이터가 잘려 나갈까 두려워, 일단 무식하게 DB의 데이터를 전부 다 JVM의 램(RAM)으로 퍼 올린 뒤, 자바 애플리케이션 단에서 페이징을 처리하고 있었다. OOM(명백한 메모리 폭발) 직전의 극단적 비효율이었다. "Fetch Join이 만능이 아니구나"라는 걸 뼈저리게 실감했다.

---

## 3. 타협과 깨달음: @BatchSize의 마법

"N+1도 막아야 하고, 컬렉션 페이징 OOM도 막아야 한다"는 딜레마 속에서 선배들의 코드 베이스를 따라가다 드디어 은탄환(Silver Bullet)을 찾았다. 바로 **Batch Size** 설정이다.

`application.yml`에 다음과 같은 한 줄을 끼워 넣었다.

```yaml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 1000
```

이 옵션 하나가 가져온 기적은 놀라웠다.
"지연 로딩 때문에 N+1 쿼리가 터질 상황이 오면, 하나씩 단건으로 날리지 말고 어차피 긁어올 대상들 ID를 `WHERE id IN (1, 2, ... 1000)` 처럼 한 방 묶음으로 처리해서 한 번만 가져와라" 라는 지시였다.
컬렉션을 Fetch Join 할 필요가 없으니 페이징(`LIMIT`)은 데이터 뻥튀기 없이 평화롭게 동작했고, 최악의 1,000건짜리 쿼리 폭격은 단 1번의 IN 쿼리 병합으로 말끔히 방어해냈다.

---

## 맺으며

N+1 문제는 "단순한 버그"나 "ORM의 단점"이 아니었다. 객체의 연관 관계와 관계형 데이터베이스의 집합 패러다임이 부딪히면서 생기는 지극히 당연한 구조적 충돌이었다. 

예전엔 에러가 터지면 구글링해서 복사-붙여넣기 하느라 급급했다면, 이제는 쿼리 로그를 한 줄 한 줄 쳐다보며 DB와 메모리의 고통을 상상하게 된다. LAZY를 유지하고, `Fetch Join`의 한계를 `Batch Size`로 부드럽게 넘나들 줄 아는 감각. 이 치열했던 인프라 최적화 고민의 경험이, 더 단단한 백엔드 개발자로 나를 성장시킬 것이라 믿는다.

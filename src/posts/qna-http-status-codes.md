## Q1. HTTP 상태 코드의 분류와 가장 자주 쓰는 코드를 설명해 주세요.

**A.** 5개의 군으로 나뉩니다. 1xx 정보, 2xx 성공, 3xx 리다이렉션, 4xx 클라이언트 오류, 5xx 서버 오류.

실무에서 자주 쓰는 핵심만 추리면:
- **200 OK**: 일반 성공.
- **201 Created**: 리소스 생성 + `Location` 헤더로 새 URL 안내.
- **204 No Content**: 성공 + 본문 없음(DELETE 응답에 자주).
- **301 / 308**: 영구 이동. 308은 메서드/본문 보존.
- **302 / 307**: 임시 이동. 307이 메서드 보존.
- **304 Not Modified**: 캐시 유효(ETag와 짝).
- **400, 401, 403, 404, 409, 422, 429**가 가장 자주 등장하는 4xx.
- **500, 502, 503, 504**가 5xx의 핵심.

## Q2. 401과 403의 차이는 뭔가요?

**A.** 이름이 헷갈리지만 의미가 다릅니다.

- **401 Unauthorized**: 사실은 "**미인증**"의 뜻. 로그인이 필요하거나 토큰이 잘못된 경우. `WWW-Authenticate` 헤더로 어떤 인증 방식이 필요한지 알립니다.
- **403 Forbidden**: 인증은 됐지만 **권한이 없음**. 다른 계정이 필요하거나 접근 자체가 금지된 리소스.

보안 정책에 따라 권한 없는 리소스의 존재를 숨기려고 **403 대신 404로 응답**하는 경우도 많습니다.

## Q3. 502와 504는 어떻게 구분하나요?

**A.** 두 코드 모두 게이트웨이/프록시(Nginx, ALB 등)가 응답할 때 발생합니다.

- **502 Bad Gateway**: 업스트림이 응답을 보냈는데 **잘못된 응답**(예: 커넥션 reset, 잘못된 HTTP 형식).
- **504 Gateway Timeout**: 업스트림이 **응답 자체를 못 줌**(시간 초과).

요약: "응답이 왔는데 이상함 = 502, 응답이 안 옴 = 504".

## Q4. POST로 리소스 생성 시 어떤 상태 코드와 헤더를 반환하나요?

**A.** **201 Created** + 응답 헤더 `Location: /users/123`로 새로 생성된 리소스의 URL을 안내합니다. 본문에 생성된 자원을 같이 담는 것이 클라이언트 단순화에 좋습니다.

생성이 비동기라면 **202 Accepted** + 폴링용 상태 URL을 반환하는 패턴도 자주 씁니다.

## Q5. 검증 실패는 400과 422 중 무엇으로 응답해야 하나요?

**A.** 정답은 없지만 통념은:
- **400 Bad Request**: 형식 자체가 잘못(JSON 파싱 실패, 필수 헤더 누락).
- **422 Unprocessable Entity**: 형식은 맞지만 **의미적으로 잘못**(이메일 형식 위반, 필수 필드 누락, 값 범위 초과).

팀 컨벤션 일관성이 더 중요합니다. RFC 7807 `application/problem+json`으로 본문에 에러 코드/필드를 명세하면 클라이언트가 분기 처리하기 좋습니다.

---

## Q6. 200 응답 본문에 `{"success": false}` 형태로 에러를 표현해도 되나요?

**A.** 안티패턴에 가깝습니다.

- **모니터링/알림**이 상태 코드 기반인 경우가 많아 5xx/4xx가 묻혀 버립니다.
- **HTTP 캐시**가 200을 캐시할 수 있어 잘못된 응답이 전파됩니다.
- 클라이언트의 표준 에러 처리(fetch의 `response.ok` 등)가 무력화됩니다.

상태 코드와 본문은 일치시켜야 합니다. GraphQL이 200 + errors 배열을 쓰는 건 **다른 모델이라 의도된 설계**일 뿐, REST에 그대로 가져오면 안 됩니다.

## Q7. Rate limit에 걸렸을 때 어떻게 응답해야 하나요?

**A.** **429 Too Many Requests** + `Retry-After` 헤더(초 또는 HTTP-date)로 언제 재시도하면 되는지 알립니다. 추가로:

- `X-RateLimit-Limit / Remaining / Reset` 헤더로 한도 정보 노출.
- 본문에 에러 코드/메시지.
- 클라이언트는 **exponential backoff + jitter**로 재시도.

5xx와 달리 429는 **클라이언트의 행동 변화**를 유도해야 하는 응답입니다.

## Q8. 3xx 중 301과 302의 SEO/캐시 차이는요?

**A.** 검색엔진과 브라우저의 동작이 다릅니다.

- **301 Moved Permanently**: 영구 이동. 검색엔진이 인덱스를 새 URL로 업데이트, 브라우저가 적극 캐시.
- **302 Found**: 임시 이동. 인덱스 유지, 캐시 보수적.

URL 영구 변경은 301, A/B 테스트나 임시 우회는 302 또는 307. 메서드 보존이 필요하면 307/308.

## 핵심 요약

면접에서 "401 vs 403 차이?", "200 vs 204?", "PUT 응답 코드?"가 자주 나옴. 단순 암기보다 **언제 어떤 의미인지** 설명할 수 있어야 합니다.

## 분류

| 군 | 의미 |
|---|---|
| 1xx | 정보(거의 안 씀) |
| 2xx | 성공 |
| 3xx | 리다이렉션 |
| 4xx | 클라이언트 오류 |
| 5xx | 서버 오류 |

## 자주 쓰는 2xx

- **200 OK**: 일반 성공 + 응답 본문.
- **201 Created**: 리소스 생성 성공. `Location` 헤더에 새 리소스 URL.
- **202 Accepted**: 요청 받았지만 비동기 처리. 폴링 URL 동봉.
- **204 No Content**: 성공이지만 본문 없음. DELETE 응답에 적합.

## 자주 헷갈리는 3xx

- **301 Moved Permanently**: 영구 이동. 검색엔진 인덱스 업데이트.
- **302 Found / 303 See Other / 307 Temporary**: 임시 이동.
- **304 Not Modified**: 캐시 유효. ETag/If-None-Match와 짝.
- **308 Permanent Redirect**: 301과 비슷하지만 메서드/본문 보존.

301 vs 302: SEO와 캐시 영향 다름. 301은 브라우저가 적극 캐시.

307 vs 302: 307은 **메서드 보존**(POST→POST). 302는 구현이 GET으로 바꾸는 경우 多.

## 4xx 핵심

- **400 Bad Request**: 요청 형식이 잘못. 검증 실패 메시지.
- **401 Unauthorized**: 인증 필요. 사실은 "**미인증**"의 의미. `WWW-Authenticate` 헤더로 인증 방식 알림.
- **403 Forbidden**: 인증은 됐지만 **권한 없음**.
- **404 Not Found**: 리소스 없음.
- **405 Method Not Allowed**: 경로는 있지만 메서드가 다름. `Allow` 헤더로 가능한 메서드 알림.
- **409 Conflict**: 현재 상태와 충돌(중복 키, 동시 수정).
- **410 Gone**: 영구 삭제. 404보다 강한 신호.
- **422 Unprocessable Entity**: 형식은 맞지만 의미적으로 처리 불가(검증 실패에 흔히 사용).
- **429 Too Many Requests**: rate limit 초과. `Retry-After` 헤더.

### 401 vs 403
- 401: "당신이 누군지 모름" → 로그인 필요.
- 403: "당신을 알지만 권한 없음" → 다른 계정 필요/접근 거부.

### 404 vs 403
- 보안 관점: 권한 없는 리소스의 존재를 숨기려고 **404로 통일**하는 정책도 흔함.

## 5xx 핵심

- **500 Internal Server Error**: 일반 서버 오류.
- **502 Bad Gateway**: 게이트웨이가 업스트림에서 잘못된 응답.
- **503 Service Unavailable**: 일시 불가(점검/과부하). `Retry-After`.
- **504 Gateway Timeout**: 업스트림 타임아웃.

502 vs 504: 응답이 왔지만 이상함(502) vs 응답이 안 옴(504).

## 메서드별 응답

- `GET`: 200 + 본문.
- `POST` 생성: 201 + Location.
- `POST` 액션(검색/명령): 200.
- `PUT` 갱신: 200(본문 반환) 또는 204(본문 없음).
- `PATCH`: 200 또는 204.
- `DELETE`: 204(보통).

## REST 설계의 헷갈림

- "리소스가 없으면 200 + empty array vs 404?" → 컬렉션 조회는 보통 200 + []. 단일 리소스는 404.
- "검증 실패는 400 vs 422?" → 형식 오류 = 400, 의미 오류(필수 필드 누락 등) = 422. 팀 컨벤션이 더 중요.
- "403을 401로 응답?" → 의미 혼동. 분리해야 클라이언트가 적절히 대응.

## 자주 헷갈리는 디테일

- 200 응답 본문에 `{"success": false}` 패턴은 안티패턴 가까움. 상태 코드와 본문이 일치해야 클라이언트/모니터링이 일관.
- GraphQL은 거의 모든 응답이 200 + 본문에 errors 배열. REST와 다른 모델.
- 5xx인데 클라이언트가 자동 재시도하면 트래픽 폭발 가능 → 멱등성 + backoff.

## 면접 follow-up

- "비동기 작업의 진행 상황 표현?" → 202 + Location으로 폴링 URL. 또는 webhook.
- "rate limit 응답에 무엇을 포함?" → 429 + `Retry-After` + 한도 헤더(`X-RateLimit-*`).
- "PUT vs PATCH 응답 본문?" → 일관성 있게 결정. 항상 갱신된 자원을 반환하면 클라이언트 단순화.

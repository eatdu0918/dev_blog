# 복잡한 데이터 모델을 위한 해법: CQRS 패턴의 기본 개념과 적용 시점

애플리케이션이 성장하면서 가장 흔히 겪는 통증 중 하나는 **"조회 모델이 너무 복잡해지는 것"**이다. 하나의 도메인 모델이 저장(CUD)될 때도 쓰이고, 복잡한 통계나 검색(R) 화면을 보여줄 때도 쓰이다 보니, 양측의 요구사항이 충돌하며 코드가 비대해지고 유지보수가 불가능한 수준에 이르게 된다.

이 문제를 해결하기 위해 **명령(Command)**과 **조회(Query)**의 책임을 분리하는 **CQRS(Command Query Responsibility Segregation)** 패턴을 학습해 보았다.

---

## 🧐 왜 책임을 분리해야 할까?

전통적인 시스템에서는 하나의 데이터 모델(Entity)을 가지고 CRUD를 모두 처리한다. 하지만 서비스가 커지면 다음과 같은 불균형이 발생한다.

1. **읽기 vs 쓰기의 비율**: 대부분의 서비스는 읽기 요청이 쓰기 요청보다 압도적으로 많다.
2. **요구사항의 상충**: 쓰기 작업은 엄격한 유효성 검사와 데이터 무결성이 중요하지만, 읽기 작업은 사용자에게 보여줄 데이터의 빠른 가공과 성능이 최우선이다.
3. **데이터 모델의 왜곡**: 화면에 보여 줄 복잡한 정보들을 끼워 넣다 보니, 정작 비즈니스 핵심 로직을 담아야 할 도메인 모델이 'UI용 데이터 껍데기'로 변질된다.

---

## 🏗️ CQRS: 분리하여 정복하기

CQRS는 단순하게 말하면 **"상태를 변경하는 메서드(Command)와 상태를 반환하는 메서드(Query)를 분리하라"**는 원칙이다.

![CQRS 아키텍처 다이어그램](/public/images/cqrs-architecture.png)

### 1. 명령(Command) 모델
- 시스템의 상태를 변경하는 작업(CUD)을 담당한다.
- 데이터 무결성과 도메인 비즈니스 로직을 보호하는 데 집중한다.
- 성공 여부만 반환하거나 최소한의 정보만 응답한다.

### 2. 조회(Query) 모델
- 데이터를 그대로 읽어와서 화면에 최적화된 형태(DTO)로 제공하는 작업(R)을 담당한다.
- 비즈니스 로직 없이 성능 최적화(인덱싱, 캐싱 등)에 집중한다.
- 필요하다면 조회 전용 데이터베이스를 별도로 구축하기도 한다.

---

## 🚀 한 걸음 더: 이벤트 기반 조영(Projections)

CQRS의 진가는 명령과 조회의 저장소가 분리될 때 발휘된다. 쓰기 작업이 일어날 때 이벤트를 발행하고, 이를 구독하여 조회 전용 DB를 업데이트(Projection)하는 방식이다.

- **장점**: 읽기 전용 DB는 정규화를 무시하고 오직 조회 성능에 최적화된 구조로 만들 수 있다.
- **트레이드오프**: 데이터가 동기화되기까지 짧은 지연 시간이 발생하는 **최종적 일관성(Eventual Consistency)**을 받아들여야 한다.

---

## 💡 회고: "언제" 쓸 것인가가 더 중요하다

CQRS는 강력한 만큼 시스템의 복잡도를 급격히 높인다. 단순히 데이터가 조금 많다고 해서 도입하는 것은 "닭 잡는 데 소 잡는 칼을 쓰는 격"이 될 수 있음을 배웠다.

- **도입이 필요한 곳**: 도메인이 매우 복잡하여 조회 로직이 핵심 비즈니스 로직을 오염시킬 때, 혹은 읽기 성능의 확장이 극도로 필요할 때.
- **피해야 할 곳**: 단순한 CRUD 위주의 서비스라면 기존의 통합 모델 방식이 훨씬 효율적이고 직관적이다.

모든 기술이 그렇듯, CQRS 또한 "정답"이라기보다 **"내 서비스의 복잡도를 어떤 비용으로 관리할 것인가"**에 대한 선택지 중 하나라는 점을 깨달았다. 아키텍처는 결국 문제를 해결하기 위한 도구일 뿐, 기술 자체의 화려함에 매몰되지 말아야 함을 다시 한번 되새긴다.

## 핵심 답변

**"모듈화된 모놀리식(Modular Monolith)으로 시작한다"**가 제 디폴트입니다. MSA는 분명한 이유가 생긴 다음에 쪼갭니다. "MSA가 트렌드라서"는 합리적 사유가 아닙니다.

## 왜 모놀리식이 디폴트인가

신규 프로젝트의 가장 큰 적은 **불확실성**입니다. 도메인 경계가 아직 명확하지 않은 상태에서 서비스로 쪼개면 다음 비용이 즉시 발생합니다.

- 분산 트랜잭션: Saga, Outbox, 멱등성을 모든 흐름에 깔아야 합니다.
- 네트워크 실패 처리: timeout, retry, circuit breaker를 모든 호출에 적용.
- 운영 복잡도: 서비스 N개 만큼 CI/CD, 모니터링, 알람 설정이 필요.
- 디버깅 비용: 분산 트레이싱 없이는 한 트랜잭션을 따라가기 어렵습니다.

이 비용은 도메인을 잘 모를 때 가장 비싸게 청구됩니다. 도메인 경계를 잘못 그어서 나중에 합쳐야 하는 일이 가장 흔한 실패 패턴이기 때문입니다.

## MSA가 정당화되는 신호

다음 신호 중 둘 이상이 보일 때 분리를 진지하게 고민합니다.

- **팀이 5~6팀 이상**으로 커져 동일 코드베이스 변경이 빈번하게 충돌.
- **서비스의 부하 프로파일이 명확히 다를 때**(이미지 처리는 CPU, 채팅은 메모리, 결제는 일관성 우선). 한 덩어리로 스케일하면 자원이 낭비됩니다.
- **배포 주기가 너무 다를 때**. 매일 배포하는 기능과 분기 단위로 배포하는 기능을 한 빌드에 묶기 어려워질 때.
- **기술 스택의 다양성이 정당화되는 영역**(추천 시스템은 Python, 결제는 JVM 등)이 생겼을 때.
- **도메인 경계가 안정화**되어 변경되어도 한 서비스 안에서 끝남.

## 모놀리식에서 잘 쪼개려면

처음부터 잘 쪼갤 준비를 해둡니다.

- **모듈 경계를 패키지/모듈로 강제**: Spring이라면 multi-module Gradle, 모듈 간 의존은 인터페이스만 노출.
- **DB 스키마도 모듈별로 분리**: 같은 DB여도 schema 또는 prefix로 나누고, 다른 모듈 테이블은 직접 조회 금지.
- **이벤트 기반 통신을 미리 도입**: 모듈 간 호출을 메서드 호출 대신 이벤트(Spring `ApplicationEventPublisher` 등)로 두면, 추후 메시지 큐로 교체하기 쉽습니다.
- **트랜잭션 경계를 모듈 안으로**: 두 모듈에 걸친 단일 트랜잭션을 만들지 않습니다.

## "지금 회사 상황에서 분리해야 한다면"이라는 후속 질문

저는 **Strangler Fig 패턴**으로 답합니다.

1. 빅뱅 분리 금지. 한 도메인씩 떼어냅니다.
2. 새 서비스를 만들고 기존 모놀리식 앞에 라우팅 레이어를 둡니다.
3. 기능 단위로 트래픽을 점진 이전, 양쪽이 같은 데이터를 보면 dual-write 또는 CDC로 동기화.
4. 트래픽 100% 이전 후 모놀리식의 해당 코드를 제거.

## 면접 follow-up

- "DB는 어떻게 쪼개나요?" → 우선 schema, 그다음 instance, 그다음 다른 엔진. 동기화는 CDC + Outbox.
- "공유 라이브러리는?" → 도메인 의존이 들어가면 결합이 다시 생기므로, 공통 인프라(로깅, 트레이싱)만 공유 라이브러리로. 도메인 코드는 공유 금지.
- "MSA에서 가장 어려운 점은?" → 데이터 일관성과 분산 트레이싱. 운영 도구에 들이는 시간이 코드보다 큽니다.

# 반복 업무에서 해방되고 싶다는 간절함이 나를 n8n으로 이끌었다

## 도입: "이걸 언제까지 손으로 하고 있어야 할까?"에 대한 고민

개인 프로젝트를 운영하다 보면 불가피하게 발생하는 단순 반복 업무들이 있습니다. 예를 들어, 특정 서비스에서 발생한 데이터를 수집해 요약하고, 이를 팀 메신저나 이메일로 발송하는 일 같은 것들입니다. 처음에는 '나중에 자동화 프로그램을 짜야지'라고 가볍게 생각했지만, 막상 코드를 한 줄씩 적으려고 하니 서버 인프라 구축부터 각 서비스의 API 연동까지 생각보다 큰 공수가 들어간다는 사실에 매번 뒤로 미루게 되었습니다.

그러다 문득 **"내가 모든 걸 처음부터 개발하지 않고도, 기존의 서비스들을 블록처럼 조립해서 자동화할 수는 없을까?"**라는 호기심이 생겼습니다. 그렇게 찾아낸 도구가 바로 n8n이었습니다.

## 본론 1: n8n, 노드(Node) 위에서 흐르는 데이터의 철학

n8n을 처음 접하고 대시보드를 열었을 때, 가장 인상 깊었던 것은 '노드(Node)'라고 불리는 단위들이었습니다. 개발자로서 평소 익숙하게 사용하던 함수나 객체들이 시각적인 인터페이스로 구현되어 있는 듯한 느낌을 받았습니다.

n8n의 핵심은 크게 세 가지로 요약할 수 있었습니다.

1.  **노드(Node)**: 특정 기능을 수행하는 최소 실행 단위입니다. Slack에 메시지를 보내거나, HTTP 요청을 날리거나, JSON 데이터를 변환하는 등 각각의 노드가 하나의 명확한 역할을 수행합니다.
2.  **트리거(Trigger)**: 워크플로우를 시작하는 '방아쇠' 역할을 합니다. 정해진 시간(Cron)마다 실행하거나, 외부에서 Webhook이 들어올 때 실행되도록 설정할 수 있습니다.
3.  **워크플로우(Workflow)**: 노드와 노드를 선으로 연결해 설계한 전체적인 업무 흐름입니다.

특히 n8n이 매력적이었던 이유는 **셀프 호스팅(Self-hosting)**이 가능하다는 점이었습니다. 보안상의 이유로 외부 클라우드 서비스에 데이터를 맡기는 것이 불안할 수 있는데, 내가 직접 서버를 띄워 운영할 수 있다는 사실은 심리적인 안정감을 주었습니다.

## 본론 2: JSON, 노드 사이의 혈관

n8n에서의 데이터 흐름을 깊게 파고들다 보니, 결국 모든 것은 **JSON**으로 귀결된다는 것을 깨달았습니다. 한 노드에서 처리가 끝나면 그 결과물은 JSON 형태로 다음 노드에게 전달됩니다.

이 과정을 직접 TypeScript 코드로 구현해보면 어떤 느낌일지 궁금해졌습니다. "데이터가 흐르는 통로를 설계한다"는 개념을 아주 단순화한 워크플로우 엔진 시뮬레이션을 만들어 보았습니다.

```typescript
// src/examples/n8n-workflow-concept.ts 에서 발췌

export class SimpleWorkflowEngine {
  private nodes: Map<string, WorkflowNode> = new Map();
  private connections: Connection[] = [];

  // ...중략

  async execute(startNodeId: string, initialData: any): Promise<any> {
    let currentNodeId: string | undefined = startNodeId;
    let currentData = initialData;

    while (currentNodeId) {
      const node = this.nodes.get(currentNodeId);
      if (!node) break;

      // 노드 실행 후 결과 데이터를 다음 노드로 전달
      currentData = await node.handler(currentData);

      const nextConnection = this.connections.find(c => c.from === currentNodeId);
      currentNodeId = nextConnection?.to;
    }

    return currentData;
  }
}
```

이 예제처럼 각 노드는 데이터를 입력받아 가공한 뒤 약속된 형태로 반환합니다. n8n의 내부 구현은 훨씬 복잡하고 정교하겠지만, 노드 간의 데이터 흐름이라는 전형적인 추상화 모델을 이해하는 데 큰 도움이 되었습니다.

## 마무리: 효율을 고민하는 개발자로의 한 걸음

n8n을 공부하며 얻은 것은 단순히 새로운 도구 하나를 다룰 줄 알게 된 것만이 아니었습니다. **'내가 직접 모든 코드를 짜는 것'만이 정답은 아니며, 때로는 잘 만들어진 추상화 도구를 활용해 시스템의 전체 효율을 높이는 안목이 중요하다**는 사실을 배운 것입니다.

그동안 "개발자라면 무조건 코딩으로 해결해야지"라는 고정관념에 갇혀 있었던 것 같습니다. 하지만 n8n을 통해 더 적은 공수로 더 큰 가치를 만들어낼 수 있는 방법을 알게 되었습니다. 앞으로도 단순한 반복 업무에 매몰되기보다는, 어떻게 하면 더 스마트하게 시스템을 자동화할 수 있을지를 고민하며 한 단계 더 성장해 나가고 싶습니다.

## Q1. TCP와 UDP의 차이를 한 줄로 설명해 주세요.

**A.**
- **TCP**: 연결 지향. **신뢰성/순서 보장**, 흐름·혼잡 제어. 헤더 20B+, 핸드셰이크 비용.
- **UDP**: 비연결, 데이터그램. **빠르고 단순**, 신뢰성은 애플리케이션이 직접 구현.

웹/메일/파일 = TCP, 실시간/멀티캐스트/DNS = UDP가 전통. HTTP/3의 QUIC는 **UDP 위에서 신뢰성을 재구성**해 둘의 장점을 합쳤습니다.

---

## Q2. TCP가 보장하는 것들을 정리해 주세요.

**A.** 4가지입니다.

- **신뢰성**: 시퀀스 번호 + ACK + 재전송.
- **순서 보장**: 수신 측이 시퀀스 번호로 재정렬.
- **흐름 제어**: 수신 윈도우(rwnd)로 너무 빠른 송신 방지.
- **혼잡 제어**: 네트워크 상태에 따라 송신 속도 조절(CUBIC, BBR 등).

대신 **HOL blocking**(한 패킷 손실이 뒤 패킷 처리를 막음)은 TCP의 본질적 한계로 남습니다.

---

## Q3. 3-way handshake는 왜 필요한가요?

**A.** **양쪽이 서로의 시퀀스 번호를 동기화**하기 위해서입니다.

```
Client                Server
  -- SYN -----------> (seq=x)
  <-- SYN+ACK ------- (seq=y, ack=x+1)
  -- ACK -----------> (ack=y+1)
```

- 두 번 만으로는 한쪽 시퀀스만 알게 되어 부족.
- TCP Fast Open으로 첫 핸드셰이크에 데이터 동봉도 가능(특수 옵션).
- TLS까지 결합하면 추가 RTT가 더해져 HTTP/3가 0-RTT를 추구하는 이유가 됩니다.

---

## Q4. 4-way handshake에서 왜 ACK과 FIN이 분리되나요?

**A.** **양방향 독립 종료**라서, 서버가 보낼 데이터를 마저 보낼 시간이 필요하기 때문입니다.

```
Client                Server
  -- FIN -----------> 
  <-- ACK -----------     (즉시 ACK)
  <-- FIN -----------     (서버 보낼 데이터 끝나면)
  -- ACK ----------->
```

만약 ACK과 FIN을 한 번에 보내면 서버가 아직 보낼 데이터가 있는데 닫힘. 분리 덕분에 graceful shutdown이 가능합니다.

---

## Q5. TIME_WAIT는 무엇이고 왜 문제가 되나요?

**A.** **능동 종료한 쪽이 마지막 ACK 분실 대비 일정 시간(2*MSL ≈ 60s) 대기**하는 상태입니다.

문제 시나리오:
- 짧은 시간에 **대량 커넥션을 능동 종료**(테스트, API 게이트웨이).
- 포트가 묶여 새 커넥션 생성 불가.
- 서버 재시작 시 포트 재사용 못 함.

해결:
- `SO_REUSEADDR` 옵션.
- HTTP keep-alive로 **커넥션 재사용**.
- TCP 옵션 튜닝(`net.ipv4.tcp_tw_reuse`).

## Q6. 흐름 제어와 혼잡 제어는 어떻게 다른가요?

**A.** **누구를 보호하느냐**가 다릅니다.

- **흐름 제어**: 수신측 버퍼가 넘치지 않게. **rwnd**(receive window).
- **혼잡 제어**: 네트워크가 막히지 않게. **cwnd**(congestion window) + slow start + AIMD.

송신 속도는 `min(rwnd, cwnd)`로 결정됩니다. 흐름은 양 끝점, 혼잡은 네트워크 상태가 기준입니다.

---

## Q7. UDP에서 신뢰성이 필요하면 어떻게 구현하나요?

**A.** **앱이 직접** 시퀀스/ACK/재전송/혼잡 제어를 구현합니다. **QUIC가 정석 모델**.

QUIC가 UDP 위에 구현한 것:
- 자체 시퀀스 번호 + ACK + 재전송.
- 혼잡 제어(BBR, CUBIC).
- TLS 1.3 통합 암호화.
- 스트림 다중화(TCP HOL blocking 회피).

UDP의 단순함을 유지하면서 TCP의 신뢰성을 얻는 방식입니다.

---

## Q8. TCP keep-alive와 HTTP keep-alive는 같은 건가요?

**A.** **다른 계층**입니다.

- **TCP keep-alive**: OS가 일정 시간 idle 시 **살아있는지 probe**. 죽은 커넥션 정리.
- **HTTP keep-alive**: 한 TCP 커넥션에서 **여러 HTTP 요청 재사용**. HTTP/1.1 기본.

이름이 같아 면접에서 헷갈리기 쉽습니다.

---

## Q9. TCP는 왜 메시지 경계가 없나요?

**A.** **스트림 프로토콜**이기 때문입니다.

```
보낸 쪽: send("hello"), send("world")
받은 쪽: read() → "helloworld" 또는 "hellow", "orld" 등
```

read 시점에 어디까지가 한 메시지인지 보장 안 됨. 애플리케이션이 **길이 프레이밍**(앞에 길이 4바이트) 또는 **구분자**(개행)를 직접 정의해야 합니다.

UDP는 데이터그램 단위라 메시지 경계가 보존됩니다.

---

## Q10. Nagle 알고리즘은 무엇이고 언제 끄나요?

**A.** **작은 패킷을 모아서 보내** 네트워크 효율을 높이는 알고리즘입니다.

- 작은 send마다 패킷이 나가지 않고 일정 양 모이거나 ACK가 오면 발송.
- 효율은 좋지만 **지연 추가**.

지연 민감 앱(텔넷, 게임, 실시간 채팅)에서는 `TCP_NODELAY` 옵션으로 비활성화. HTTP는 보통 큰 응답이라 영향이 작지만, 작은 요청이 잦은 RPC에서는 켜두면 응답이 느려질 수 있습니다.

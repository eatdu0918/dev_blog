## 핵심 요약

- **TCP**: 연결 지향, **신뢰성/순서 보장**, 흐름·혼잡 제어. 헤더 20+B, 핸드셰이크 비용.
- **UDP**: 비연결, 데이터그램, **빠르고 단순**. 신뢰성이 필요하면 애플리케이션이 직접 구현.

웹/메일/파일 전송은 TCP, 실시간/멀티캐스트/DNS는 UDP가 전통적 선택. **HTTP/3의 QUIC는 UDP 위에서 신뢰성을 재구성**해 둘의 장점을 합쳤습니다.

## TCP의 책임

- **신뢰성**: 시퀀스 번호 + ACK + 재전송.
- **순서 보장**: 수신측이 시퀀스 번호로 재정렬.
- **흐름 제어**: 수신 윈도우(rwnd)로 너무 빠른 송신 방지.
- **혼잡 제어**: 네트워크 상태에 따라 송신 속도 조절(CUBIC, BBR 등).
- **HOL blocking**: 한 패킷 손실이 뒤 패킷 처리를 막음 — TCP의 본질적 한계.

## UDP의 책임

거의 없음. 송신만 함.

- 신뢰성/순서/흐름·혼잡 제어 모두 **애플리케이션 책임**.
- 헤더 8B, 오버헤드 적음.
- 멀티캐스트/브로드캐스트 가능.
- 실시간 음성/영상에서 **재전송하느니 누락**이 더 나은 시나리오에 적합.

## 3-way handshake (연결)

```
Client                Server
  | -- SYN ----------> |   (seq=x)
  | <-- SYN+ACK ------ |   (seq=y, ack=x+1)
  | -- ACK ----------> |   (ack=y+1)
```

- 양쪽 시퀀스 번호 동기화.
- TCP Fast Open으로 첫 핸드셰이크에 데이터 동봉도 가능(특수 옵션).
- TLS와 결합하면 TCP 3-way + TLS handshake로 추가 RTT.

## 4-way handshake (종료)

```
Client                Server
  | -- FIN ----------> |
  | <-- ACK ---------- |
  | <-- FIN ---------- |   (서버가 보낼 데이터 다 보낸 후)
  | -- ACK ----------> |
```

- 양방향 독립 종료. 서버가 보낼 데이터를 마저 보낼 시간이 필요해서 ACK과 FIN이 분리됨.
- **TIME_WAIT**: 능동적 종료한 쪽이 마지막 ACK 분실 대비 일정 시간(보통 2*MSL ≈ 60s) 대기. 서버 재시작 시 포트 재사용 문제의 원인.
- 해결: `SO_REUSEADDR`, 짧은 TCP 옵션 튜닝, keep-alive로 연결 재사용.

## 흐름 제어 vs 혼잡 제어

자주 혼동.

- **흐름 제어**: 수신측 버퍼가 넘치지 않게(rwnd).
- **혼잡 제어**: 네트워크가 막히지 않게(cwnd, slow start, AIMD).

## 어떤 프로토콜을 쓸지

- **TCP**: HTTP/1.1, HTTP/2, gRPC, DB 커넥션, 메일.
- **UDP**: DNS, NTP, 실시간 게임, VoIP, 영상 스트리밍.
- **QUIC(UDP 기반)**: HTTP/3. TCP의 HOL blocking 회피 + 0-RTT 재방문.

## TCP keep-alive vs HTTP keep-alive

이름은 같지만 다른 계층.

- **TCP keep-alive**: OS가 일정 시간 idle 시 살아있는지 probe.
- **HTTP keep-alive**: 한 TCP 커넥션에서 여러 HTTP 요청 재사용. HTTP/1.1 기본.

## 자주 헷갈리는 디테일

- UDP라고 무조건 빠른 건 아님. 신뢰성을 직접 구현하면 결국 비슷한 비용 발생(QUIC가 그 예).
- TCP는 **스트림** 프로토콜. 메시지 경계 없음 → 애플리케이션이 길이 프레이밍 필수.
- 패킷 == 세그먼트 == 프레임이 아님. 계층마다 단위가 다름(IP=패킷, TCP=세그먼트, 이더넷=프레임).

## 면접 follow-up

- "TIME_WAIT가 운영 환경에서 문제 되는 시나리오?" → 짧은 시간에 대량 커넥션을 능동 종료하는 클라이언트(테스트, API 게이트웨이). 커넥션 재사용으로 회피.
- "Nagle 알고리즘?" → 작은 패킷을 모아서 보냄. 지연 민감 앱(텔넷, 게임)에서는 `TCP_NODELAY`로 비활성.
- "UDP에서 신뢰성을 어떻게 구현하나?" → 시퀀스 번호 + ACK + 재전송 + 혼잡 제어를 앱이 직접. QUIC가 정석 모델.

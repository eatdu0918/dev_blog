## 핵심 요약

- **HTTP/1.1**: 텍스트 프로토콜, 커넥션당 요청 1개(파이프라이닝은 사실상 사장). HOL blocking이 TCP 위가 아닌 **HTTP 레벨**에서 발생.
- **HTTP/2**: 바이너리 프레이밍 + **멀티플렉싱**으로 한 TCP 커넥션에서 여러 스트림 동시 처리. 다만 **TCP HOL blocking은 그대로** 남음.
- **HTTP/3**: TCP를 버리고 **QUIC(UDP 기반)** 위에서 동작. 스트림이 독립적이라 패킷 손실이 다른 스트림에 영향을 안 줍니다.

## HTTP/1.1의 한계

- 한 커넥션에서 한 번에 하나의 요청만. 그래서 브라우저는 도메인당 6개 정도의 커넥션을 병렬로 엽니다.
- 헤더가 매 요청마다 평문으로 반복 전송. 쿠키가 크면 오버헤드가 큼.
- 회피책: 도메인 샤딩, 이미지 스프라이트, CSS 인라이닝 — 모두 HTTP/2에서는 **안티패턴**이 됩니다.

## HTTP/2의 개선

- **멀티플렉싱**: 한 커넥션에서 여러 스트림이 인터리브. 6개 커넥션 → 1개로 충분.
- **HPACK 헤더 압축**: 동일 헤더 반복을 인덱스로 참조.
- **서버 푸시**: 사실상 폐기됨(Chrome도 제거). 캐시 무효화 어려움 + 효과 미미.
- **우선순위(priority)**: 의도대로 동작 안 하는 구현이 많아 실효성 낮음.

여전히 남는 문제: TCP는 패킷 순서를 보장하므로 **하나의 패킷이 손실되면 그 뒤 모든 스트림이 멈춥니다**. 와이파이/모바일 환경에서 체감됩니다.

## HTTP/3 = QUIC + HTTP

- **UDP 위에서 자체적으로 신뢰성/혼잡제어/암호화** 구현. TCP+TLS 핸드셰이크 2-3 RTT를 **1 RTT 또는 0-RTT**(재방문)로 단축.
- 스트림이 트랜스포트 계층에서 독립. 한 스트림의 패킷 손실이 다른 스트림을 막지 않음.
- **Connection ID** 기반이라 IP 변경(와이파이→LTE)에도 커넥션 유지 = 모바일 친화적.
- 트레이드오프: UDP를 차단/QoS 낮추는 방화벽이 있어 **fallback to HTTP/2** 경로가 필요.

## 어떻게 선택하나

대부분의 경우 **CDN/리버스 프록시(Cloudflare, Fastly, Nginx)에 맡기고 자동 협상**하면 됩니다. 클라이언트와 서버가 ALPN으로 가장 높은 공통 버전을 선택.

- 정적 자산이 많고 모바일 트래픽이 큼 → HTTP/3 활성화 우선.
- gRPC 서버 간 통신 → HTTP/2(트레일러 헤더 사용).
- 레거시 인프라/방화벽 제약 → HTTP/1.1 보장은 필수.

## TLS 핸드셰이크 한 줄

- TLS 1.2: 2 RTT.
- TLS 1.3: 1 RTT, 재방문 시 0-RTT(replay 위험으로 GET/idempotent에만).
- HTTP/3는 QUIC가 TLS 1.3를 내장 → 별도 TLS 핸드셰이크가 없음.

## 자주 헷갈리는 디테일

- "HTTP/2면 도메인 샤딩하면 더 빠르지 않나?" → 오히려 느려집니다. 한 커넥션의 압축/우선순위 효과를 날립니다.
- "HTTP/3 쓰면 무조건 빨라지나?" → RTT가 짧고 손실이 거의 없는 데이터센터 내부에선 차이 미미. 모바일/장거리에서 의미 있음.
- "HTTPS = HTTP/2/3?" → 별개 개념. HTTPS는 TLS 적용 여부, 버전은 ALPN 협상 결과.

## 면접 follow-up

- "HOL blocking이 HTTP/2에서도 남는 이유?" → TCP가 순서 보장하므로 패킷 손실 시 전체 스트림이 대기.
- "0-RTT의 보안 위험?" → Replay attack. 멱등하지 않은 요청에 쓰면 위험.
- "QUIC가 UDP 위에서 신뢰성을 어떻게 만들죠?" → 자체 시퀀스/ACK/재전송, 혼잡제어(BBR/CUBIC), 암호화는 TLS 1.3 통합.

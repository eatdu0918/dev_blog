# 메모리 관리의 기본 원리: 스택, 힙 그리고 가비지 컬렉션

![메모리 관리 리치어빌리티 그래프](/public/images/memory-management-reachability.png)

그동안 자바스크립트로 개발을 하며 메모리에 대해 깊이 고민해 본 적이 거의 없었다. 변수를 선언하고 객체를 만들면 엔진이 알아서 자리를 잡아주고, 다 쓰면 알아서 치워줄 것이라는 막연한 믿음이 있었기 때문이다. 하지만 최근 성능 최적화에 관심을 가지게 되면서, 내가 누리고 있는 이 '편리함'이 사실은 수십 년에 걸친 치열한 고민의 결과물이라는 사실을 깨닫게 되었다. 

가비지 컬렉터(Garbage Collector)는 어떤 원리로 메모리를 관리하는 것일까? 단순히 \"메모리가 부족하다\"는 현상을 넘어서, 시스템이 메모리를 바라보는 시각을 이해하고 싶어 공부한 내용을 기록해 보았다.

---

## 1. 메모리의 두 공간: 스택(Stack)과 힙(Heap)

공부를 시작하며 가장 먼저 마주한 것은 메모리가 데이터를 관리하는 두 가지 공간이었다. 처음에는 그저 "변수가 저장되는 곳"이라고만 생각했으나, 그 특성을 파헤쳐 보니 각자의 역할이 명확했다.

- **스택(Stack)**: 정적이고 규칙적인 공간이다. 함수가 호출될 때마다 생겼다 사라지는 지역 변수나 매개변수들이 이곳에 머문다. LIFO(Last-In-First-Out) 방식으로 일사분란하게 움직이며 접근 속도가 매우 빠르다는 점이 인상적이었다. 하지만 런타임에 크기가 변하는 데이터를 담기에는 그릇이 너무 고정적이라는 한계도 보였다.
- **힙(Heap)**: 반면 힙은 훨씬 더 자유롭고 동적인 거대한 운동장 같았다. 우리가 흔히 만드는 객체나 배열처럼 크기를 가늠할 수 없는 데이터들이 이곳에 자리를 잡는다. 자유로운 만큼 관리가 까다롭고, 이 공간을 어떻게 효율적으로 청소하느냐가 런타임 성능의 핵심 지표가 된다는 점을 알게 되었다.

---

## 2. 관리 방식의 변화를 따라가며 얻은 깨달음

메모리 관리의 역사는 개발자의 실수를 줄이고 창의성에 집중할 수 있도록 돕는 방향으로 흘러왔음을 알 수 있었다.

### 직접 하는 관리의 무게
문서상으로 접한 수동 할당 방식은 개발자에게 절대적인 권한을 부여했다. 하지만 동시에 '직접 빌려오고 직접 갚아야 한다'는 규칙은 나에게는 너무 무겁게 느껴졌다. 조금만 방심해도 메모리가 줄줄 새거나 프로그램이 죽어버리는 위험을 안고 코드를 짜야 했던 시절의 고충을 간접적으로나마 실감할 수 있었다.

### 참조 카운팅(Reference Counting)의 명과 암
다음에 마주한 방식은 참조 횟수를 세는 논리였다. 누군가 나를 바라보면 숫자를 올리고, 관심을 끊으면 숫자를 내리는 방식은 꽤 직관적이었다. 하지만 **순환 참조(Circular Reference)**라는 함정을 보는 순간 소름이 돋았다. 서로가 서로를 붙잡고 있다는 이유만으로 아무도 쓰지 않는 데이터가 영원히 메모리를 점유할 수 있다는 사실은, 단순한 숫자 세기만으로는 완벽한 해답이 될 수 없음을 가르쳐 주었다.

### 도달 가능성(Reachability)이라는 새로운 시각
오늘날 V8이 택한 **Mark-and-Sweep** 방식은 나에게 새로운 통찰을 주었다. 단순히 "몇 명이 나를 보는가"가 아니라 **"뿌리로부터 나에게 도달할 수 있는가"**를 묻는 방식이었다. 
- **Mark**: 루트(Global Object 등)에서 출발해 손에 손을 잡고 연결된 모든 객체를 찾아낸다.
- **Sweep**: 그 손을 놓쳐버린, 즉 뿌리로부터 고립된 섬들을 일괄적으로 수거한다.
이 알고리즘 덕분에 앞선 순환 참조 문제도 자연스럽게 해결된다는 점이 무척이나 경이롭게 느껴졌다. 아무리 서로를 강하게 쥐고 있어도, 뿌리로부터의 선이 끊겼다면 가차 없이 정리 대상이 된다는 단순하고 강력한 논리였다.

## 3. 코드로 시뮬레이션해 본 참조 구조

이해한 내용을 바탕으로 순환 참조 상황이 발생했을 때, 도달 가능성이 어떻게 끊어지는지 TypeScript로 표현해 보았다.

```typescript
// src/examples/memory-demo.ts

export class MemoryNode {
    public partner: MemoryNode | null = null;
    constructor(public name: string) {}
}

/**
 * 순환 참조 상황에서 도달 가능성이 끊어지는 논리를 시뮬레이션합니다.
 */
export function simulateCircularReference() {
    let nodeA: MemoryNode | null = new MemoryNode("Node A");
    let nodeB: MemoryNode | null = new MemoryNode("Node B");

    // 서로를 참조하는 고리 형성
    nodeA.partner = nodeB;
    nodeB.partner = nodeA;

    // 참조 카운팅 방식이라면 여기서 문제가 되지만, 
    // 루트 변수(nodeA, nodeB)와의 연결을 끊어버리면...
    nodeA = null;
    nodeB = null;

    // 현대의 엔진은 이제 이 두 객체에 도달할 방법이 없음을 감지하고 
    // 다음 청소 시간에 이들을 수거해갈 것이다.
    return "Disconnected from Root";
}
```

실제 가비지 컬렉션의 타이밍을 제어할 수는 없지만, 테스트 코드를 통해 내가 설계한 참조 구조가 의도대로 끊어지는지 논리적으로 확인해 볼 수 있었다.

```typescript
// test/memory-management.test.ts
import { describe, it, expect } from 'vitest';
import { MemoryNode, simulateCircularReference } from '../src/examples/memory-demo';

describe('메모리 관리 원리 학습', () => {
    it('순환 참조가 있어도 루트와의 연결이 끊기면 수거 대상이 됨을 논리적으로 이해한다', () => {
        const result = simulateCircularReference();
        expect(result).toBe("Disconnected from Root");
    });
    
    it('유효한 참조가 남아있는 동안에는 메모리가 유지되어야 함을 확인한다', () => {
        const root = new MemoryNode("Root");
        const leaf = new MemoryNode("Leaf");
        root.partner = leaf;
        
        // root가 살아있는 한 leaf로 가는 길은 확보되어 있다.
        expect(root.partner?.name).toBe("Leaf");
    });
});
```

## 💡 마치며: 메모리 도달 가능성의 이해

가비지 컬렉터가 도달 가능성(Reachability)을 기준으로 메모리를 관리한다는 점을 배우며, '논리적 누수'의 무서움을 다시 한번 실감했습니다. 특히 전역 변수나 해제되지 않은 이벤트 리스너처럼, 실질적으로는 쓰이지 않지만 루트와의 연결이 남아있어 메모리를 점유하는 패턴을 더 주의 깊게 살펴야겠다는 다짐을 하게 되었습니다.

편리한 자동 메모리 관리 환경에 익숙해져 있었지만, 결국 효율적인 자원 사용은 개발자가 객체의 생명주기를 얼마나 명확히 인지하고 있느냐에 달려 있음을 알게 되었습니다. 앞으로도 참조의 연결과 해제를 명확히 의식하며 코드를 작성하는 습관을 들여보려 합니다.

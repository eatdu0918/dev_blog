## Q1. 주요 정렬 알고리즘의 복잡도와 안정성을 정리해 주세요.

**A.**

| 알고리즘 | 평균 | 최악 | 메모리 | 안정 | 비고 |
|---|---|---|---|---|---|
| Bubble | O(n²) | O(n²) | O(1) | ✅ | 학습용 |
| Insertion | O(n²) | O(n²) | O(1) | ✅ | 거의 정렬된 데이터에 빠름 |
| Merge | O(n log n) | O(n log n) | O(n) | ✅ | 외부 정렬 가능 |
| Quick | O(n log n) | O(n²) | O(log n) | ❌ | 실무 평균 가장 빠름 |
| Heap | O(n log n) | O(n log n) | O(1) | ❌ | 우선순위 큐 |
| Tim | O(n log n) | O(n log n) | O(n) | ✅ | Java/Python 표준 |

## Q2. Quick Sort의 최악이 O(n²)가 되는 조건은?

**A.** **이미 정렬된 입력 + 첫/끝 피벗**입니다.

피벗이 항상 가장 작거나 큰 값이면 분할이 한쪽으로만 진행되어 트리 높이가 n이 됩니다 → O(n²).

회피:
- **무작위 피벗** 선택.
- **Median-of-three**: 첫/중간/끝 중 중간값.
- **Intro Sort**: 깊이 임계를 넘으면 Heap Sort로 전환.

## Q3. Merge Sort와 Quick Sort 중 무엇을 선호하시나요?

**A.** 상황에 따라 다릅니다.

- **Merge**: 안정 정렬 + 보장된 O(n log n) + 외부 정렬 가능. 단, **추가 메모리 O(n)**.
- **Quick**: in-place + 캐시 친화적이라 평균 가장 빠름. 단, 최악 O(n²).

순서가 중요한 데이터(다중 키 정렬)는 Merge, 일반적인 in-memory 정렬은 Quick(또는 Intro/Tim)입니다.

## Q4. 안정 정렬이 무엇이고 왜 중요한가요?

**A.** **같은 키를 가진 원소들의 상대 순서가 보존**되는 정렬입니다.

```
[("kim", 30), ("lee", 25), ("kim", 28)]  // name으로 정렬
안정: [("kim", 30), ("kim", 28), ("lee", 25)]  // 원래 순서 유지
불안정: [("kim", 28), ("kim", 30), ("lee", 25)]  // 임의 순서
```

다중 키 정렬에서 결정적입니다. "부서 정렬 후 이름 정렬"하면 부서 안에서 이름 순서가 유지되어야 합니다. Java/Python 표준은 안정, C++ `std::sort`는 불안정(필요하면 `std::stable_sort`).

## Q5. 비교 기반 정렬의 O(n log n) 한계는 왜 그런가요?

**A.** **정보이론 하한** 때문입니다.

n개를 정렬하는 가능한 순열 = n!. 비교 한 번으로 얻는 정보는 1bit. n!개 중 하나를 결정하려면 log₂(n!) ≈ n log n 비교가 필요.

이 한계를 넘으려면 **비교 외 정보**를 써야 합니다(키 범위, 분포 등) → Counting/Radix/Bucket Sort.

---

## Q6. O(n) 정렬은 어떤 조건에서 가능한가요?

**A.** **키 범위가 제한적이거나 분포가 균등할 때**입니다.

- **Counting Sort**: 키 빈도 카운팅. 정수 + 작은 범위(예: 0~1000).
- **Radix Sort**: 자릿수별 안정 정렬. 정수/문자열에 효율.
- **Bucket Sort**: 균등 분포 가정 + 각 버킷 내 정렬.

일반적인 데이터에는 부적합하지만 특수 도메인에서는 매우 빠릅니다.

## Q7. 언어별 표준 정렬은 무엇을 쓰나요?

**A.**
- **Java**: 원시 타입 = Dual-Pivot Quicksort, 객체 = Tim Sort.
- **Python**: `sorted` / `list.sort` = Tim Sort.
- **JavaScript**: ES2019+ `Array.prototype.sort` 안정성 보장. V8은 Tim Sort.
- **C++**: `std::sort` = Intro Sort.

대부분 **하이브리드 정렬**입니다. 작은 배열은 Insertion, 일반은 Quick/Merge, 최악 회피는 Heap.

## Q8. 10억 개 데이터를 정렬해야 한다면?

**A.** 메모리에 안 들어가니 **외부 정렬 또는 분산 정렬**이 필요합니다.

- **외부 정렬(External Merge Sort)**: 청크로 나눠 메모리에서 정렬 후 디스크에 저장 → 병합. Unix `sort`도 이 방식.
- **분산 정렬(MapReduce sort)**: 여러 노드가 부분 정렬 후 병합.

핵심은 **메모리 사용을 데이터 크기와 무관하게 일정 수준으로 유지**하는 알고리즘 선택입니다.

## Q9. Top-K 문제는 어떻게 푸나요?

**A.** 정렬 전체를 할 필요가 없습니다. 두 방법.

- **Min-heap (크기 K)**: O(n log k). K개 원소만 힙에 유지. 큰 데이터에 효과적.
- **Quickselect**: 평균 O(n). 피벗 분할로 K번째 원소만 찾고 그 앞을 가져옴. 메모리 O(1).

K가 작으면 힙, K가 n에 가까우면 그냥 정렬이 더 단순합니다.

## Q10. 안정 정렬을 불안정 정렬로 만드는 트릭이 있나요?

**A.** **키에 인덱스를 결합**하면 됩니다.

```
원본: [a, b, c, d]
키 + 인덱스: [(a, 0), (b, 1), (c, 2), (d, 3)]
```

같은 키여도 인덱스가 다르므로 비교가 결정적이 되어 결과가 안정적입니다. 면접에서 가끔 묻는 트릭이고 실무에서도 분산 정렬에서 가끔 사용합니다.

# Java 제어자 활용 전략: static, final, abstract를 이용한 객체 설계의 제약과 명확성 확보

![Java 제어자 개념도](/public/images/java-modifiers-static-final-abstract.png)

코드를 작성하다 보면 단순히 '기능이 동작하게' 만드는 것을 넘어, '이 코드는 이렇게 사용되어야만 한다'는 제약을 걸고 싶을 때가 있다. 누군가 무분별하게 값을 바꾸지 못하게 하거나, 모든 객체가 공유해야 하는 데이터를 정의하거나, 혹은 반드시 구현해야 하는 밑그림만 그려두고 싶을 때 말이다.

최근 프로젝트에서 상태 관리 로직을 설계하며 이러한 '제약의 필요성'을 절실히 느꼈고, Java가 제공하는 대표적인 제어자들인 `static`, `final`, `abstract`가 단순히 문법적인 장치를 넘어 설계자의 의도를 담는 강력한 언어라는 사실을 다시금 깨닫게 되었다.

---

## 1. static: "개인이 아닌 공동의 자산"

`static`은 멤버(변수, 메서드)가 특정 객체(Instance)가 아닌 **클래스 자체에 귀속**되도록 만든다. 

### 깨달음의 순간
유틸리티 함수를 만들 때마다 매번 객체를 생성(`new`)해야 하는 번거로움에 의문을 가졌던 적이 있다. "도대체 왜 이 함수는 상태도 없는데 매번 메모리를 잡아먹으며 생성되어야 할까?" 

`static`은 이 고민을 해결해 주었다. 클래스 로더가 클래스를 읽어들이는 시점에 메모리(Method Area)에 딱 한 번 할당되어 모든 객체가 공유한다. 마치 아파트 단지의 개별 세대 안에 있는 정수가 아니라, 단지 전체가 공유하는 '중앙 분수대'와 같은 존재였다.

```java
class Config {
    static String serverUrl = "https://api.example.com"; // 모든 인스턴스가 공유
}
```

---

## 2. final: "변치 않는 약속의 도장"

`final`은 **'마지막(Last)'**이라는 의미 그대로, 한 번 결정되면 더 이상 변경할 수 없음을 선언한다.

- **변수**: 초기화 이후 값을 바꿀 수 없다 (상수).
- **메서드**: 하위 클래스에서 재정의(Overriding)할 수 없다.
- **클래스**: 더 이상 상속(Inheritance)할 수 없다.

협업을 하다 보면 내가 의도치 않게 다른 사람의 핵심 설정값을 바꿔버리거나, 확장해서는 안 되는 중요한 로직을 상속받아 고쳐버리는 사고가 발생하곤 한다. `final`은 "이 부분은 내 설계의 핵심이니 더 이상 건드리지 마세요"라고 찍어두는 안전한 인장과 같았다. 이 제약이 주는 심리적 안정감은 생각보다 컸다.

---

## 3. abstract: "미완의 미학, 설계의 밑그림"

`abstract`는 **실체는 없지만 반드시 존재해야 하는 개념**을 정의할 때 사용한다.

추상 메서드는 구현부(`{}`) 없이 이름만 존재하며, 이 메서드를 가진 클래스는 반드시 추상 클래스가 된다. 이는 "구체적인 구현은 자식들에게 맡길 테니, 최소한 이 이름과 형식을 갖춘 기능은 반드시 만들어라"라는 강제적인 약속이다.

### 설계적 관점에서의 발견
처음에는 왜 굳이 '비어 있는 메서드'를 만드는지 이해하지 못했다. 하지만 복잡한 시스템을 설계하다 보니, 상위 수준에서는 흐름만 제어하고 상세 동작은 하위에서 결정하게 만드는 '프레임워크' 적인 도구가 반드시 필요함을 알게 되었다. `abstract`는 명확한 가이드라인을 제시함으로써 코드의 통일성을 지켜주는 든든한 설계도 역할을 해주었다.

## 4. 코드로 증명하기 (TypeScript simulation)

Java의 이러한 제어자 철학은 현대적인 JavaScript/TypeScript에서도 고스란히 이어진다. `static`, `readonly`(final 역할), `abstract`를 활용하여 Java의 설계 방식을 시뮬레이션해 보았다.

```typescript
// test/java-modifiers.test.ts 중 일부 발췌

abstract class BaseService {
  // 1. static: 클래스 수준의 공유 데이터
  static VERSION = "1.0.0";

  // 2. final 역할을 하는 readonly (생성 이후 수정 불가)
  readonly serviceName: string;

  constructor(name: string) {
    this.serviceName = name;
  }

  // 3. abstract: 자식 클래스에서 반드시 구현해야 함
  abstract execute(): string;
}

class ConcreteService extends BaseService {
  execute() {
    return `${this.serviceName} executed (v${BaseService.VERSION})`;
  }
}
```

이 테스트 코드를 작성하며, 언어는 달라도 '안전하고 명확한 설계를 지향하는 마음'은 모두 같다는 점을 느꼈다.

## 💡 마치며: 제약이 자유를 만든다

입문 시절의 나에게 제어자는 오직 컴파일 오류를 뱉어내는 귀찮은 제약 조건일 뿐이었다. 하지만 지금의 나는 **제약이 명확할수록 개발자가 더 자유로워질 수 있다**는 역설적인 진리를 배워가고 있다.

`static`으로 메모리 효율을 고민하고, `final`로 데이터의 무결성을 지키며, `abstract`로 유연한 확장성을 설계하는 과정. 이 삼총사를 적절히 배치하는 것이야말로 진흙탕 같은 코드 속에서 질서를 세우는 설계자의 첫걸음임을 깨닫는다. 앞으로도 단순히 '동작함'에 안주하지 않고, 이 제어자들을 활용해 '의도가 읽히는 코드'를 짜기 위해 노력할 것이다.

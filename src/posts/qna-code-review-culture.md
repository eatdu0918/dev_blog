## Q1. 코드 리뷰의 목적이 무엇이라고 생각하시나요?

**A.** "버그 찾기"라고만 답하면 깊이가 얕아 보입니다. 결함 검출은 의외로 작은 비중이고, 대부분의 버그는 테스트와 프로덕션이 잡습니다.

좀 더 좋은 답:
- **지식 공유**: 한 사람만 아는 코드 영역이 생기지 않게.
- **설계 정합성**: 이 변경이 모듈 책임/경계와 맞는가.
- **장기 유지보수성**: 6개월 뒤에 이해되는가.
- **저자의 학습**: 더 나은 패턴을 함께 익힘.

## Q2. 리뷰할 때 무엇부터 보시나요?

**A.** 위에서 아래로 우선순위가 명확합니다.

1. **요구사항 vs 구현 일치**: PR/티켓과 실제 변경이 같은 문제를 푸는가.
2. **설계/아키텍처**: 책임 분리, 의존 방향, 추상화 레벨. 여기서 틀어지면 라인 코멘트는 의미 없음.
3. **정확성**: 엣지 케이스, 동시성, 에러 처리, 트랜잭션 경계.
4. **테스트**: 회귀가 잠겨 있는가.
5. **가독성/명명**.
6. **스타일/포매팅**: 자동화로 처리 — 사람 시간을 쓰지 않음.

## Q3. 코드 리뷰가 잘 안 굴러갈 때 어떤 신호가 보이나요?

**A.** 주로 4가지로 드러납니다.

- 리뷰 대기로 PR이 며칠 정체 → 영업일 24시간 내 1차 응답 SLA + 리뷰어 로테이션.
- 형식적 승인 후 사고 빈발 → 머지 게이트 강화(테스트/커버리지/E2E), 시니어 동행 리뷰.
- PR이 수백~수천 줄 → 작업 분리 가이드, 리팩토링 PR과 기능 PR 분리.
- 리뷰 톤이 공격적 → "Why we choose this" 형식, 결정권자는 저자라는 합의.

## Q4. PR 사이즈가 너무 커지는 문제는 어떻게 해결하나요?

**A.** 200~400줄 이하가 리뷰 품질이 유지되는 경험적 한계선입니다. 그 이상은 리뷰어가 표면만 보게 됩니다.

전략:
- **사전 설계 합의**(RFC/ADR)로 방향을 PR 전에 정해 두기.
- **draft PR**로 큰 변경의 방향을 일찍 공유.
- 리팩토링 PR을 기능 PR보다 먼저 머지해 기능 변경을 작게.
- 큰 작업은 **여러 단계 PR**로 쪼개고 피처 플래그로 미완성 보호.

## Q5. 리뷰에 자동화는 어디까지 적용하시나요?

**A.** 사람 시간을 가장 많이 잡아먹는 부분부터 자동화합니다.

- **포매터/린터/타입체커**: 스타일 코멘트 0.
- **PR 템플릿**: 동기, 변경 요약, 테스트 방법, 리스크 명시.
- **CI 게이트**: 테스트, 커버리지 임계, 보안 스캔, 빌드 사이즈.
- **Conventional Commits + 자동 changelog**.

자동화가 끝나면 사람 리뷰는 **설계와 정확성**에 집중할 수 있습니다.

## Q6. 리뷰 코멘트는 어떻게 다는 게 좋다고 생각하세요?

**A.** 4가지 톤을 의식적으로 구분합니다.

1. **Blocking**: 머지 막는 결함. 이유 명시.
2. **Non-blocking suggestion** (`nit:`): 더 나은 방안 제안, 결정은 저자.
3. **Question**: "이 분기는 어떤 상황을 위한 거에요?" — 의도 확인.
4. **Praise**: "이 추상화 좋네요" — 강화. 비판만 있는 리뷰는 문화를 망칩니다.

코드를 평가하는 것이지 사람을 평가하는 게 아니라는 톤이 기본입니다.

## Q7. 주니어가 시니어의 PR을 리뷰할 때 어려움은 어떻게 풀어야 하나요?

**A.** 직급이 아니라 책임이 리뷰의 단위입니다.

- 의견을 단정형 대신 **질문 형식**으로: "왜 이 방식인지 알고 싶어요."
- 시니어의 의도를 먼저 파악한 뒤 의견을 더하기.
- 모르면 **묻는 것 자체가 가치**(나중에 다른 멤버도 같은 의문을 가짐).

리뷰가 양방향 학습이 되는 팀이 가장 건강합니다.

---

## Q8. 보안/성능 이슈는 리뷰에서 어떻게 놓치지 않나요?

**A.** 사람의 주의력에만 의존하면 반드시 새어 나갑니다.

- **체크리스트**: 인증/권한, 입력 검증, SQL 파라미터화 등.
- **자동 스캔**: SAST, 의존성 취약점 검사를 CI에 포함.
- **보안/성능 라벨 PR**: 별도 전문 리뷰어 지정.
- **성능 회귀 테스트**: 벤치마크 자동화 + 임계 초과 시 알림.

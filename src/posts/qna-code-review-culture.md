## 핵심 요약

코드 리뷰는 면접에서 "협업 능력 + 코드 품질에 대한 관점"을 동시에 보는 단골 질문입니다. **리뷰의 목적**과 **무엇을 우선순위로 보는지**, **리뷰가 막힐 때 무엇을 바꿨는지**를 묻습니다.

## 리뷰의 목적은 결함 찾기가 아니다

면접에서 흔히 나오는 답: "버그 찾기, 컨벤션 검사."

더 좋은 답:
- **지식 공유**: 코드베이스에 대한 컨텍스트가 한 사람에게만 있는 상황 회피.
- **설계 정합성**: 이 변경이 모듈 책임/경계와 맞는가.
- **장기 유지보수성**: 6개월 뒤의 나/팀이 이해할 수 있는가.
- **결함 검출**: 의외로 비중이 낮음. 대부분의 버그는 테스트/프로덕션이 잡음.

## 리뷰에서 우선순위로 보는 것

위에서 아래로 순서대로 보는 것이 효율적입니다.

1. **요구사항 vs 구현 일치**: PR 설명/티켓과 실제 변경이 같은 문제를 푸는가.
2. **설계/아키텍처**: 책임 분리, 의존 방향, 추상화 레벨. 여기서 틀어지면 라인 코멘트는 무의미.
3. **정확성**: 엣지 케이스, 동시성, 에러 처리, 트랜잭션 경계.
4. **테스트**: 변경된 동작이 회귀 방지 테스트로 잠겨 있는가.
5. **가독성/명명**: 6개월 뒤에도 이해되는가.
6. **스타일/포매팅**: 자동화로 처리 — 사람 시간을 쓰지 않음.

## 안티패턴

- **수백 줄 PR + 하루 뒤 리뷰**: 리뷰어가 표면만 봄. 200~400줄 이하로 쪼개기.
- **"승인했는데 동작은 안 봤어요"**: 리뷰어 책임 회피. 최소한 의도된 시나리오 한 번은 머리로 시뮬레이션.
- **취향 논쟁(bikeshedding)**: 컨벤션은 자동화, 취향은 ADR로 결정 후 종결.
- **저자 방어형 응답**: 리뷰는 코드를 보는 것이지 사람을 평가하는 것이 아님을 PR 템플릿/문화로 명시.

## 리뷰 비효율의 신호와 처방

- **리뷰 대기로 PR이 며칠씩 정체** → SLA 도입(예: 영업일 24시간 내 1차 응답), 리뷰어 로테이션, 핫시트(당번) 제도.
- **승인이 형식적이고 사고가 자주 남** → 머지 게이트 강화(테스트 커버리지/E2E), 페어 리뷰, 시니어 동행 리뷰.
- **PR이 너무 큼** → 작업 단위 분리 가이드, "리팩토링 PR과 기능 PR 분리" 룰.
- **리뷰 톤이 공격적** → "Why we choose this" 형식 제안 + 결정권자(저자) 명시.

## 자동화로 사람의 시간을 아낌

- **포매터/린터/타입체커**: 스타일 코멘트 0.
- **PR 템플릿**: 동기, 변경 요약, 테스트 방법, 리스크.
- **CI 게이트**: 테스트, 커버리지 임계, 보안 스캔.
- **Conventional Commits + 자동 changelog**: 히스토리 가독성.

## 좋은 리뷰 코멘트의 4가지 톤

1. **Blocking**: "이건 머지하면 안 됨, 이유는…" — 명확한 결함.
2. **Non-blocking suggestion**: "이렇게 하면 더 나을 수 있는데, 결정은 저자" — `nit:`.
3. **Question**: "이 분기가 어떤 상황을 위한 거에요?" — 의도 확인.
4. **Praise**: "이 추상화 좋네요" — 강화. 리뷰가 비판만 있으면 문화가 망가짐.

## 자주 헷갈리는 디테일

- 리뷰는 **저자의 학습**도 목적. "내가 다시 짤게" 식의 hijack은 효과가 떨어짐.
- "두 명 승인" 같은 룰만으로는 품질 보장 안 됨. **누가 무엇을 책임지고 보는지** 명시(설계/보안/DB).
- 리뷰는 **실시간 페어 프로그래밍을 대체할 수 없음**. 큰 설계는 PR 전에 합의.

## 면접 follow-up

- "리뷰가 길어지는 PR을 어떻게 처리하나?" → 작업 분리, 사전 설계 합의(ADR/RFC), drafting PR로 방향 미리 검증.
- "주니어가 시니어의 PR 리뷰할 때 어려움?" → 질문 형식, 의도 파악 후 의견. 리뷰는 직급이 아니라 책임.
- "리뷰에서 보안/성능 이슈를 어떻게 놓치지 않나?" → 체크리스트, 보안 라벨 PR은 별도 리뷰어, 성능 회귀 테스트 자동화.

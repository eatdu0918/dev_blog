## Q1. 선착순 쿠폰 발급에서 가장 먼저 생각하는 것은 무엇인가요?

**A.** 두 가지 결정으로 나눕니다. **어디서 직렬화할지(DB/앱/인프라)** 와 **어떻게 직렬화할지(락/큐/카운터)**.

정확도 우선이라면 DB 락, 처리량이 우선이라면 Redis 카운터, 폭증 트래픽이라면 큐로 직렬화. 한 가지 정답은 없고 트래픽 규모와 비즈니스 허용치로 갈립니다.

---

## Q2. 비관적 락과 낙관적 락 중 무엇을 선택하시나요?

**A.** **충돌 빈도**로 결정합니다.

- **비관적 락(`SELECT ... FOR UPDATE`)**: 충돌이 잦거나 정확도가 최우선일 때. 단순하고 강력하지만 락이 길어지면 처리량 폭락 + DB 단일 병목.
- **낙관적 락(`@Version`)**: 충돌이 드물 때. 재시도 비용이 작아야 의미.

선착순은 충돌이 거의 100%라 낙관적 락은 부적합합니다. 재시도 폭주로 오히려 망가집니다.

---

## Q3. Redis 분산 락은 언제 쓰고 어떤 함정이 있나요?

**A.** 여러 인스턴스가 같은 자원을 다툴 때 좋은 선택입니다. Redisson이나 SETNX + TTL 패턴.

함정:
- **락 해제 실패**: TTL 안전장치 필수.
- **클럭 드리프트**: 노드 간 시계 차이.
- **Redis SPOF**: 단일 노드 다운 시 락 시스템 마비. Redlock은 안전성 논쟁이 있음.
- **fencing token**: TTL 만료로 다른 워커가 들어왔을 때 이전 워커의 작업이 충돌하지 않게 토큰으로 검증.

---

## Q4. 트래픽이 매우 클 때(초당 1만 건 이상)는 어떻게 설계하나요?

**A.** **Redis 카운터 + DB 정산** 또는 **Kafka 직렬화**가 정석입니다.

- 재고를 Redis에 두고 `DECR` 후 음수면 거절. DB는 사후 영속화.
- 응답 속도와 처리량이 압도적이지만 Redis 장애 시 fallback 정책이 필요합니다.
- 더 큰 규모라면 큐에 넣고 단일 컨슈머가 처리 → 본질적으로 직렬화 보장. 다만 사용자 응답이 비동기가 되어 UX 설계도 같이 가야 합니다("발급 처리 중" 상태).

---

## Q5. 트래픽 규모별로 어떤 도구를 선택하시는지 정리해 주세요.

**A.** 보통 이렇게 나눕니다.

- **분당 수십~수백 건**: 비관적 락. 단순한 게 최고.
- **초당 수백~수천 건**: Redis 카운터 + DB 정산.
- **초당 1만 건 이상**: 큐로 직렬화 + 사전 분배(미리 발급할 토큰 풀 준비).

---

## Q6. 같은 사용자의 더블 클릭/재시도는 어떻게 처리하나요?

**A.** **멱등성 키(Idempotency-Key)** 로 처리합니다.

- 클라이언트가 요청마다 고유 키(UUID)를 헤더에 동봉.
- 서버는 키 + 요청 본문 해시를 캐시/DB에 저장.
- 같은 키로 재요청이 오면 저장된 응답을 그대로 반환.

DB 레벨에서는 `UNIQUE` 제약 + `ON CONFLICT DO NOTHING`이 추가 안전망입니다.

---

## Q7. Hot row 문제는 어떻게 회피하나요?

**A.** 같은 row에 락이 몰리면 DB가 병목이 됩니다.

- **카운터 분할**: 재고를 100개 슬롯으로 쪼개고 무작위 슬롯에 `DECR`. 사용자 입장에서는 동일.
- **샤딩**: 사용자 ID 기반으로 다른 노드에 분배.
- **사전 분배**: 발급 가능한 토큰 풀을 미리 만들어 분산.

---

## Q8. 정확도와 성능 사이에서 어디까지 양보할 수 있나요?

**A.** 비즈니스가 결정합니다. 쿠폰처럼 1~2개 오차가 운영 정책으로 흡수 가능한 케이스가 많습니다.

이 가정이 처리량 설계를 좌우합니다. 절대 오차 0이 필요하면 단일 DB 직렬화로 갈 수밖에 없고, 약간의 오차를 허용하면 Redis/큐 기반 고처리량 설계가 가능합니다. 면접에서는 **"비즈니스 허용치를 먼저 묻겠다"** 가 좋은 답입니다.

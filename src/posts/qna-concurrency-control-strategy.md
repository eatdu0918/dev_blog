## 핵심 요약

같은 자원을 두고 여러 요청이 다투는 문제는 **(1) 어디서 직렬화할지**와 **(2) 어떻게 직렬화할지**의 결정입니다. DB 레벨, 애플리케이션 레벨, 인프라 레벨 모두 도구가 있고, 한 가지 정답은 없습니다.

## 선택지 비교

### 1. 비관적 락 (`SELECT ... FOR UPDATE`)
- 가장 단순하고 강력. 정확도가 최우선이면 무난.
- 약점: 락이 길어지면 처리량이 급감하고, **단일 DB로 트래픽이 몰립니다.** 1만 RPS급 선착순은 DB가 못 버팁니다.

### 2. 낙관적 락 (`@Version`)
- 충돌이 드물 때 적합. 충돌 시 재시도 비용이 작아야 의미가 있습니다.
- 선착순처럼 **충돌이 거의 100%인 시나리오**에서는 재시도 폭주로 오히려 성능이 망가집니다.

### 3. Redis 기반 분산 락 (Redisson, SETNX)
- TTL 안전장치를 둘 수 있어 운영 친화적.
- 함정: **락 전파 / 해제 실패 / 클럭 드리프트**를 고려해야 합니다. Redis 단일 노드는 SPOF, Redlock은 논쟁이 있습니다.

### 4. Redis INCR/DECR로 카운터 자체를 옮기기
- 선착순에 가장 잘 맞는 패턴. 재고 카운터를 Redis에 두고 `DECR` 후 음수면 거절.
- DB는 사후 정산. 응답 속도와 처리량이 압도적입니다.

### 5. Kafka로 직렬화
- 요청을 큐에 넣고 단일 컨슈머가 처리. 본질적으로 직렬화가 보장.
- 사용자 응답을 비동기로 만들어야 해서 UX 설계가 같이 가야 합니다(예: "발급 처리 중" 상태).

## 실제 트래픽 규모로 갈라치기

저는 보통 이렇게 나눕니다.

- **분당 수십~수백 건**: 비관적 락이면 충분. 단순한 게 최고.
- **초당 수백~수천 건**: Redis 카운터 + DB 정산. INCR 후 큐에 넣고 컨슈머가 영속화.
- **초당 1만 건 이상**: 큐로 직렬화 + 미리 발급할 토큰 풀을 준비해두는 식의 사전 분배 설계까지 같이 검토.

## 자주 놓치는 디테일

- **멱등성 키**: 같은 사용자가 더블 클릭하거나 네트워크 재시도로 중복 요청을 보낼 때, 요청 ID 기반으로 한 번만 처리되도록 막아야 합니다.
- **리스크: lost update**: `read-modify-write`가 트랜잭션 밖에 있으면 DB 락만으로는 부족합니다. 같은 트랜잭션 안에서 잠가야 합니다.
- **Hot row 문제**: 같은 row에 락이 몰리면 InnoDB의 buffer pool에서도 대기열이 길어집니다. row를 쪼개거나(예: 100개 슬롯으로 분할 후 무작위 슬롯에 INCR) 카운터 분산을 검토해야 합니다.
- **장애 복구**: Redis가 죽었을 때 DB로 fallback할 수 있게 설계해두면 운영이 편합니다. 단, 재고 일관성을 잠시 포기할지 사전에 합의가 필요합니다.

## 면접 follow-up

- "정확도 vs 성능, 어디까지 양보 가능한가요?" → 비즈니스에 따라 다르지만, 쿠폰은 보통 **재고 1~2개 오차는 운영 정책으로 흡수**합니다. 그 가정이 처리량 설계를 바꿉니다.
- "분산 락이 풀리지 않았을 때 어떻게 하나요?" → TTL 기반 자동 해제 + `lockToken`(fencing token)을 같이 두어, 만료 후 다른 워커가 들어왔을 때 이전 워커의 작업이 충돌하지 않도록.
- "큐에 적체되면?" → 백프레셔, 사용자에게 노출되는 대기열 위치 UX, 컨슈머 오토 스케일링.

# 직렬화(Serialization)와 역직렬화(Deserialization) 실무 적용기

개발 과정에서 애플리케이션 내부에 존재하는 도메인 객체를 프론트엔드로 전달하거나 데이터베이스, 캐시 서버에 저장하려 할 때 반드시 거쳐야 하는 변환 과정이 있습니다. 바로 **직렬화(Serialization)**와 **역직렬화(Deserialization)**입니다.

개념적으로만 알고 있던 이 변환 과정에서 제가 직접 마주했던 실무적인 고민과 이슈들을 회고해 봅니다.

---

## 📦 직렬화가 필요한 이유

메모리 공간 위에 생성된 객체는 현재 실행 중인 프로세스 환경(가상 메모리 주소 등)에 강하게 묶여 있습니다. 이 데이터를 네트워크를 통해 전송하거나 디스크에 보관하려면, 어떤 환경에서든 동일하게 해석될 수 있는 1차원의 바이트 스트림이나 텍스트 포맷으로 변환해야 합니다. 
- **직렬화**: 객체 상태를 전송/저장 가능한 포맷(JSON, 바이너리 등)으로 변환하는 작업.
- **역직렬화**: 수신한 포맷 데이터를 다시 애플리케이션의 메모리 객체로 복원하는 작업.

---

## 🚚 포맷 선택: JSON vs 바이너리

초기 웹 서버와 클라이언트 간의 REST API를 구축할 때는 가장 대중적인 **JSON** 포맷을 채택했습니다. 사람이 읽을 수 있어 디버깅이 편하고, JavaScript 기반 프론트엔드 환경(`JSON.stringify`, `JSON.parse`)에서 기본적으로 잘 지원되기 때문입니다.

그러나 서비스가 발전하며 내부 마이크로서비스 간의 통신 로직을 구현하게 되었을 때, JSON의 텍스트 파싱 비용과 데이터 크기가 시스템의 병목으로 작용함을 확인했습니다. 이를 해결하기 위해 백엔드 간 통신에서는 Google의 **Protocol Buffers (gRPC)**와 같은 바이너리 직렬화 방식을 도입해 보았습니다. 읽기는 어려워졌지만 데이터 크기가 현저하게 줄어들고 직렬화/역직렬화 속도가 대폭 개선되는 이점을 얻을 수 있었습니다.

---

## ⚠️ 직렬화 과정에서 겪은 장애와 주의점

### 1. 민감성 데이터의 무차별 직렬화
과거 회원 데이터를 JSON으로 응답하는 API를 구현하면서, User 도메인 객체를 그대로 직렬화 프레임워크에 던져버리는 실수를 한 적이 있습니다. 이로 인해 암호화되었더라도 비밀번호 해시값이나 주민등록상세 번호 등의 필드가 클라이언트로 모두 노출되었습니다. 이후로는 응답용 **DTO(Data Transfer Object)**를 철저하게 분리하고, 직렬화 무시 애노테이션(예: `@JsonIgnore`)을 활용하여 노출을 통제하는 습관을 들이게 되었습니다.

### 2. 버전 호환성 문제
캐시 저장소(Redis)에 직렬화하여 담아둔 객체가 있었습니다. 추후 도메인 클래스에 새로운 필드를 몇 개 추가하고 배포를 진행했는데, 기존에 캐싱되어 있던 일주일 전의 바이트 데이터를 현재의 클래스 구조로 역직렬화 시도하다가 매핑 에러가 발생하며 서버가 다운되었습니다. 
객체의 구조가 변경되었을 때 과거 데이터와의 하위 호환성을 유지하기 위한 전략(Java의 `serialVersionUID` 고정, 혹은 필드 버전 관리)이 반드시 동반되어야 함을 배운 뼈아픈 경험이었습니다.

---

## 💡 회고

직렬화와 역직렬화는 프레임워크가 알아서 처리해 주는 '블랙박스'처럼 여겨지기 쉽습니다. 하지만 그 이면에 존재하는 포맷별 성능 차이나, 데이터 필드 노출 보안, 이전 버전과의 호환성 문제 등을 고려하지 않고 수동적으로 사용하면 큰 시스템 장애로 이어질 수 있음을 프로젝트들을 거치며 체감하고 있습니다.

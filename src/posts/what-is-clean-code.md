# 클린 코드(Clean Code)란? 읽기 좋은 튼튼한 코드 작성하기

개발자로 일하다 보면 **"클린 코드(Clean Code)"**라는 말을 자주 접하게 됩니다. 세계적인 소프트웨어 엔지니어 '로버트 C. 마틴(Uncle Bob)'의 저서 제목이기도 한 클린 코드는, 단순히 동작하는 것을 넘어 유지보수하기 좋고 타인이 읽기 좋은 코드를 의미합니다.

그렇다면 '나쁜 코드'는 무엇이고, 왜 우리는 일정을 쪼개서라도 클린 코드를 작성해야 할까요? 이번 포스팅에서는 클린 코드의 핵심 개념과 당장 실천할 수 있는 몇 가지 기본 원칙을 살펴보겠습니다.

---

## 1. 클린 코드(Clean Code)란 무엇인가?

클린 코드는 한마디로 **"다른 사람이 읽고 이해하기 쉽게 작성된 코드"**입니다. 여기서 '다른 사람'은 6개월 뒤의 나 자신도 포함됩니다.

> *"코드는 1번 작성되고 10번 읽힌다. 따라서 코드를 읽기 쉽게 만드는 것이 더 중요하다."*

작동만 하는 쓰레기 코드는 결국 '기술 부채(Technical Debt)'가 되어 미래의 개발 속도를 갉아먹습니다. 반면 클린 코드는 의도가 명확하고, 중복이 없으며, 테스트가 가능하고 오류 처리가 잘 되어 있는 상태를 말합니다.

---

## 2. 나쁜 코드(Bad Code)가 주는 치명적인 영향

- **생산성 하락**: 얽히고설킨 '스파게티 코드'는 작은 기능을 추가하거나 버그를 고칠 때마다 예상치 못한 곳에서 에러를 냅니다. 개발자들은 코드를 읽고 이해하는 데 엄청난 시간을 낭비하게 됩니다.
- **깨진 유리창 법칙**: "어차피 더러운 코드인데 대충 수정하자."라는 마인드가 생겨 전체 시스템의 품질이 급격히 떨어집니다.
- **재설계(Rewrite)의 함정**: 코드가 도저히 손쓸 수 없는 지경에 이르면 기존 시스템을 포기하고 처음부터 다시 개발(재구축)하자는 논의가 나옵니다. 이는 엄청난 리소스 낭비를 초래합니다.

---

## 3. 당장 실천할 수 있는 클린 코드의 핵심 원칙 5가지 💡

로버트 C. 마틴은 수많은 원칙을 제시했지만, 그중에서도 가장 기본이 되고 당장 적용해 볼 수 있는 원칙들을 추려보았습니다.

### ① 의도가 분명한 이름 짓기 (Meaningful Names)
클린 코드의 첫걸음은 변수, 함수, 클래스에 '의미 있는 이름'을 지어주는 것입니다. 축약어나 애매한 이름은 피하세요.

```javascript
// 나쁜 예: t가 뭘 의미하는지 알 수 없음
const t = 10;
const aCnd = true;

// 좋은 예: 의도가 명확함
const elapsedTimeInSeconds = 10;
const isAccountActive = true;
```

### ② 함수는 작게, 단 한 가지 일만 하라 (Do One Thing)
함수는 그 길이가 짧을수록 좋고, 오직 **'하나의 기능'**만 수행해야 합니다. 예를 들어 회원 가입을 처리하는 함수가 있다면 데이터베이스 저장 처리만 해야지, 비밀번호 암호화 로직이나 이메일 발송 로직까지 다 가지고 있어서는 안 됩니다.

### ③ 주석은 나쁜 코드를 변명할 뿐이다 (Comments)
코드 자체가 자신의 역할을 명확하게 설명하고 있다면, 주석은 필요 없습니다. 최고의 주석은 '코드 그 자체'입니다. 코드가 너무 복잡해서 어쩔 수 없이 다는 주석보다는, 변수명이나 함수명, 구조를 리팩토링하여 의미를 명확하게 드러내는 것이 좋습니다.
물론, 왜(Why) 이런 방식이나 알고리즘을 선택했는지 이유를 설명하는 주석은 환영입니다.

### ④ 포맷팅(Formatting) 맞추기
팀 내에서 정한 코딩 컨벤션(들여쓰기, 줄바꿈 위치, 빈 줄 등)을 철저히 지켜야 합니다. 일관된 스타일은 코드를 읽는 속도를 비약적으로 높여줍니다. 최근에는 `Prettier`나 `ESLint` 같은 도구를 활용해 이 과정을 자동화하는 추세입니다.

### ⑤ 오류 처리도 한 가지 작업이다 (Error Handling)
비즈니스 로직과 에러 처리 로직이 섞여 있으면 코드의 흐름을 파악하기 매우 어렵습니다. `try-catch` 블록을 활용하여 정상 흐름과 예외 처리 흐름을 분리하고 코드를 깔끔하게 유지하세요.

```javascript
// 나쁜 예: 로직과 에러 처리가 뒤엉켜 있음
function deletePage(page) {
    const error = deleteRegistry(page.name);
    if (!error) {
        // 성공 로직
    } else {
        console.log("Delete failed:", error);
    }
}

// 좋은 예: 예외 처리를 활용하여 깔끔하게 분리
function deletePage(page) {
    try {
        deleteRegistry(page.name);
    } catch (e) {
        console.log("Delete failed:", e.message);
    }
}
```

## 4. 마치며 (보이스카우트 규칙) ⛺

클린 코드에는 **"처음 왔을 때보다 캠핑장을 더 깨끗하게 치우고 떠나라"**는 '보이스카우트 규칙'이 적용됩니다. 한꺼번에 모든 코드를 대청소(대대적인 리팩토링)하려고 하지 마세요. 체크아웃할 때 내가 맡은 파일의 변수 이름 하나, 함수 구조 하나만이라도 원래보다 조금 더 깨끗하게 만들고 커밋한다면 프로젝트는 결국 점점 더 건강해질 것입니다.

시간이 부족하다는 핑계로 더러운 코드를 양산하지 맙시다. **기능 구현과 클린 코드 작성은 별개의 과정이 아니라 하나입니다.** 의식적으로 더 나은 구조를 고안하며 코딩하는 습관을 들여봅시다.

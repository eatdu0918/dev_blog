# 클린 코드(Clean Code): 이해하기 좋은 코드를 향한 고민

로버트 C. 마틴의 명저 **"클린 코드(Clean Code)"**는 수많은 개발자들의 바이블처럼 여겨집니다. 주니어 개발자로서 빠르게 기능을 쳐내는 데만 집중하다가, 몇 달 뒤 제가 짠 코드를 보고서도 유지보수를 할 수 없는 유지보수 지옥을 겪은 뒤에야 클린 코드의 원칙들을 다시 돌아보게 되었습니다.

당시 겪었던 기술 부채의 폐해와, 이를 개선하기 위해 의식적으로 체화하고자 한 몇 가지 기본 원칙을 회고해 봅니다.

---

## 1. 읽히지 않는 코드의 치명적 결과

일정에 쫓겨서 "일단 돌아가게만 만들고 나중에 고치자"라고 넘겼던 로직들이 있었습니다. 이른바 기술 부채(Technical Debt)를 쌓아둔 것입니다. 

시간이 흘러 해당 도메인에 기능을 추가해야 하는 요구사항이 들어왔습니다. 중첩된 조건문과 서로 책임을 떠미는 거대한 파일들을 분석하느라, 한 줄의 코드를 추가하기 위해 하루 종일 코드를 추적해야 했습니다. 한곳을 편법으로 수정하면 전혀 관계없어 보이는 화면에서 에러가 터졌습니다. 코드를 짜는 시간보다 '해독'하는 시간이 훨씬 많이 소모되는 악순환을 몸소 겪은 시기였습니다.

---

## 2. 당장 적용해 본 클린 코드의 5가지 실천

거창한 아키텍처 재설계를 떠나서, 당장 내일부터 시도해 보고 효과를 본 원칙들입니다.

### ① 의도가 분명한 네이밍 (Meaningful Names)
기능의 의미보다는 짧은 타이핑을 선호하여 포괄적인 변수(예: `data`, `info`, `type`, `flag`)를 남발했었습니다. 이를, 시간이 다소 걸리더라도 구체적으로 명시하는 습관으로 바꾸었습니다.

```javascript
// 과거의 혼란스러운 코드
const d = 10;
const f = true;

// 개선한 네이밍
const elapsedTimeInSeconds = 10;
const isUserAuthenticated = true;
```

### ② 함수는 하나의 일만 하도록 작게 분리 (Do One Thing)
하나의 함수 안에서 유효성 검사, 네트워크 데이터 패치, 상태 변경, 에러 로깅을 모두 넣었던 과거가 있었습니다. 
이후엔 함수의 길이를 최대한 줄이고 책임을 하나로 국한시켜 분리했습니다. 함수 분리는 곧 단위 테스트를 작성하기 쉬운 코드로 이어짐을 확인했습니다.

### ③ 주석을 지우고 코드로 설명하기 
복잡하게 꼬인 로직을 변명하려고 긴 주석을 남기곤 했습니다. 그러나 리팩토링 과정에서 코드가 변경되어도 주석은 현행화되지 않아, 오히려 향후 개발자를 속이는 잘못된 정보가 되어버리는 상황을 맞았습니다.
"어떻게" 동작하는지 설명하는 주석을 지우고, 대신 함수 이름을 의미 있게 추출하여 코드 자체로 읽히도록 바꾸려는 노력을 기울이고 있습니다. (비즈니스 결정의 "왜"를 설명하는 주석은 예외로 두었습니다.)

### ④ 포맷팅과 컨벤션 유지
개발자마다 띄어쓰기, 줄바꿈 등의 스타일이 다르면 코드를 읽을 때 가독성이 심하게 훼손되었습니다. 
팀 차원에서 `Prettier`와 `ESLint` 등을 프로젝트 배포 파이프라인과 커밋 훅(Husky 등)에 연결해 포맷팅을 강제하는 방식으로 통일성을 확보했습니다.

### ⑤ 에러 처리와 정상 로직의 분리 (Error Handling)
정상적으로 흘러가는 로직 중간중간에 수많은 `if/else` 에러 체크 코드가 끼어있으면 전체 흐름 파악이 너무 힘겨웠습니다. 비즈니스 로직과 예외 처리 로직(`try/catch`)을 명확히 분리하여 가독성을 높이는 훈련을 했습니다.

---

## 회고와 꾸준한 리팩토링

클린 코드를 지향하면서 깨달은 점은, 수만 줄의 프로젝트를 하루아침에 대대적으로 리팩토링할 수는 없다는 것이었습니다. 

대신 **"내가 수정한 파일은, 전보다 변수 이름 하나라도 더 깨끗하게 다듬고 커밋하자"**라는 규칙을 꾸준히 실천하는 것이 훨씬 현실적이고 강력한 방법임을 느꼈습니다. 여전히 바쁜 일정 앞에서는 유혹이 생기지만, 지금의 타협이 미래의 유지보수 비용으로 돌아올 것이라 생각하며 코드 품질을 유지하려고 노력하고 있습니다.

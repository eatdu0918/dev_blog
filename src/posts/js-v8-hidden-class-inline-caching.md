# V8 엔진의 객체 접근 최적화: 히든 클래스와 인라인 캐싱

![V8 히든 클래스 전환 과정 도식](/public/images/v8-hidden-classes.png)

자바스크립트 개발을 하며 객체의 속성을 마음껏 추가하고 삭제하는 유연함을 누릴 때마다, "런타임에 이런 유연함을 모두 감수하면서 어떻게 엔진이 성능을 낼 수 있을까?"라는 의문이 생기곤 했습니다. V8 엔진 내부를 조사하며 마주한 **Hidden Class(히든 클래스)**와 **Inline Caching(인라인 캐싱)**은 이러한 고민을 해결해 준 가장 혁신적인 기법이었습니다. 

동적인 자바스크립트 객체를 정적인 구조처럼 바라보려는 노력이 성능을 얼마나 끌어올릴 수 있는지 학습해 보았습니다.

---

## 1. 정적 구조: Hidden Class (Shapes)

자바스크립트 객체는 해시 맵(Hash Map)처럼 속성을 마음껏 담을 수 있습니다. 하지만 해시 맵 기반의 속성 접근은 매번 메모리 주소를 찾아야 하는 비용이 발생함을 보게 되었습니다.

- **히든 클래스 정의**: V8은 객체가 생성될 때 내부적으로 '히든 클래스'라는 정적인 구조 지도를 만듭니다. 같은 속성을 같은 순서로 가지는 객체들은 동일한 히든 클래스를 공유합니다.
- **클래스 전환(Transition)**: 객체에 새로운 속성을 추가하면, V8은 기존 히든 클래스에서 새로운 속성이 추가된 새로운 히든 클래스로 지도를 옮깁니다. "이 객체는 이제 속성 y를 더 가졌어(A' -> A'')"라고 기록하는 과정을 보며, 왜 코드를 짤 때 객체의 형태를 일정하게 유지하는 것이 엔진에게 큰 힌트가 되는지 실감했습니다.

---

## 2. 접근을 더 빠르게: Inline Caching (IC)

지도가 준비되었다면, 이제는 그 지도를 매번 다시 보지 않고도 목적지까지 갈 수 있는 지름길이 필요했습니다.

- **반복 접근 최적화**: 어떤 함수에서 객체의 속성에 반복적으로 접근할 때, V8은 그 접근이 성공했던 히든 클래스와 메모리 오프셋(Offset) 정보를 함수 호출 지점(Call Site)에 직접 '캐싱'해 버립니다.
- **성능의 극대화**: 다음 호출 때 객체의 히든 클래스가 캐시된 정보와 같다면, 복잡한 조회 과정을 생략하고 즉시 메모리 주소로 점프합니다. "타입이 같으면 이전 성공 경험을 기억하겠다"는 이 지능적인 방식이 자바스크립트의 속도를 정적 언어와 견주게 만듦을 알게 되었습니다.

---

## 💡 마치며: 엔진 친화적인 코드의 가치

V8의 히든 클래스와 인라인 캐싱을 학습하며 느낀 점은, 무차별적인 동적 속성 추가가 엔진 내부에서는 상당히 비싼 비용(Hidden Class의 잦은 전환 및 캐시 미스)을 요구한다는 사실이었습니다. 

앞으로는 객체를 생성할 때 생성자 함수나 클래스를 활용해 일관된 모양을 유지하고, 동적인 속성 추가를 최대한 지양하는 습관을 들여보려 합니다. 엔진이 우리 코드를 최적화할 수 있도록 명확한 구조를 제공하는 것, 그것이 바로 더 효율적인 애플리케이션을 만드는 개발자로서의 책임임을 마음속에 담아봅니다.

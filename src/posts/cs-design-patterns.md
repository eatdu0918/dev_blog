# 디자인 패턴(Design Patterns): 언어와 패러다임에 따른 변화

소프트웨어를 설계할 때 발생하는 흔한 문제들을 해결하기 위한 일종의 모범 답안이 바로 **디자인 패턴(Design Patterns)**입니다. 

처음 개발을 배울 때 1994년에 발표된 "Gang of Four(GoF)"의 23가지 패턴들을 교과서처럼 외우고 적용해 보려 애썼던 기억이 납니다. 하지만 이후 다양한 프로젝트를 경험하며 백엔드(Java)와 프론트엔드(JavaScript/React) 생태계를 오가다 보니, **자주 쓰이는 패턴 역시 언어와 환경에 따라 크게 달라진다는 것을 체감했습니다.**

---

## ☕ Java에서의 경험 (객체지향의 세계)

Java와 같은 정적 타입 기반의 객체지향 언어를 다룰 때는, 클래스와 인터페이스의 관계를 설계하는 **GoF 패턴이 여전히 매우 유효하다**고 느꼈습니다.

### 1. 싱글톤 패턴 (Singleton Pattern)
- **개념**: 애플리케이션 전체에서 클래스의 인스턴스가 오직 하나만 생성되도록 보장하는 패턴입니다.
- **실무 경험**: `DatabaseConnectionManager` 등을 직접 구현해 보기도 했고, 이후 Spring 프레임워크를 사용하면서 프레임워크가 내부적으로 `Bean` 객체를 관리할 때 이 패턴을 어떻게 활용하는지 이해하는 데 큰 도움이 되었습니다.

### 2. 팩토리 패턴 (Factory Pattern)
- **개념**: 객체 생성 로직을 팩토리 클래스에 위임하는 패턴입니다.
- **실무 경험**: 특정 인터페이스의 구현체를 동적으로 생성해야 할 때 적용해 보았습니다. 구체적인 클래스 이름에 의존하지 않고 객체를 찍어내게 되므로, 코드가 유연해지고 의존성 주입(DI)의 기본 원리를 깨닫는 계기가 되었습니다.

### 3. 전략 패턴 (Strategy Pattern)
- **개념**: 런타임에 실행할 알고리즘을 쉽게 교체할 수 있게 해주는 패턴입니다.
- **실무 경험**: 여러 종류의 결제 수단 처리를 구현할 때, 수많은 `if-else` 분기문을 작성하다가 전략 패턴을 도입하여 구조를 개선했던 적이 있습니다. 각 결제 수단을 '전략'으로 추상화하여 분리해 내니 코드의 가독성과 유지보수성이 크게 좋아졌습니다.

## 🌐 JavaScript / TypeScript에서의 경험 (유연함과 함수형의 조화)

자바스크립트 생태계(Node.js, React 등)로 넘어오니 1급 객체로서의 함수, 클로저(Closure), 비동기 처리 등 언어 특성이 달랐습니다. GoF 패턴을 무리하게 적용하기보다는, 언어적 특성에 맞는 변형된 패턴들을 더 자주 사용하게 되었습니다.

### 1. 모듈 패턴 (Module Pattern)
- **개념**: 클로저(Closure)를 활용하여 내부 상태를 캡슐화하고 공개 API만 노출하는 패턴입니다.
- **실무 경험**: ES6의 `import/export` 모듈 시스템이 도입되기 이전 환경의 코드를 분석하면서 빈번하게 마주쳤고, 자바스크립트에서의 스코프(Scope) 격리 개념을 잡는 데 도움이 되었습니다.

### 2. 옵저버 패턴 (Observer Pattern) / Pub-Sub 
- **개념**: 상태 변경 시 구독 중인 객체들에게 자동으로 알림을 보내는 패턴입니다.
- **실무 경험**: 프론트엔드에서 DOM 이벤트를 처리하거나 Redux, Zustand 같은 상태 관리 라이브러리를 사용하면서 이 패턴의 중요성을 체감했습니다. 단방향 데이터 흐름을 이해하는 기본기가 되었습니다.

### 3. 조합(Composition)과 훅스(Hooks)
- **개념**: UI 요소를 작은 컴포넌트로 나누고, 훅을 통해 로직을 재사용하는 형태입니다.
- **실무 경험**: 과거에 HOC(Higher-Order Component) 패턴으로 컴포넌트를 감싸며 로직을 재사용하려다 "Wrapper Hell"에 빠진 경험이 있습니다. 이후 React Hooks (`useXXX`)를 도입하면서, 객체지향적인 상속보다는 함수 단위의 조합(Composition)이 프론트엔드 생태계에 훨씬 자연스럽게 녹아든다고 느꼈습니다.

---

## 💡 결론 및 회고

디자인 패턴을 공부하며 느낀 점은, 무조건 특정 패턴을 적용해야 한다는 강박증을 가질 필요가 없다는 것입니다. Java 환경에서 교과서적인 정답이었던 패턴이 JavaScript 함수형 생태계에서는 오히려 불필요한 보일러플레이트 코드를 양산하는 원인이 될 수도 있었습니다. 

앞으로는 패턴의 이름이나 형태를 기계적으로 암기하기보다는, **"이 패턴이 어떤 문제를 해결하기 위해 고안되었고, 현재 내가 사용하는 언어의 패러다임과 잘 맞는가?"**를 먼저 고민하면서 아키텍처를 설계해 보려 합니다.

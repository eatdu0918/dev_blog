# 테스트 코드란 무엇이며, 왜 작성해야 할까?

개발을 진행하다 보면 "테스트 코드를 작성해야 한다"는 말을 수없이 듣게 됩니다. 스타트업이나 대기업을 막론하고 테스트 코드의 중요성은 나날이 강조되고 있습니다. 그렇다면 **테스트 코드(Test Code)**란 정확히 무엇이고, 바쁜 일정 속에서도 시간을 내어 왜 작성해야 하는 것일까요?

이번 포스팅에서는 테스트 코드의 기본 개념과 3가지 주요 종류, 그리고 테스트 코드를 작성했을 때 얻을 수 있는 강력한 이점들에 대해 알아보겠습니다.

---

## 1. 테스트 코드란?

**테스트 코드**는 우리가 작성한 **프로덕션 코드(실제 서비스에 동작하는 코드)가 의도한 대로 정확하게 작동하는지 검증하기 위해 작성하는 코드**를 말합니다. 사람이 직접 값을 입력하고 결과를 눈으로 확인하는 '수동 테스트'를 자동화한 것이라고 볼 수 있습니다.

### 대표적인 테스트의 종류 (Test Pyramid)

테스트는 검증하려는 범위에 따라 크게 3가지로 나눌 수 있습니다.

1.  **단위 테스트 (Unit Test)**
    *   **가장 작고 독립적인 테스트 단위**입니다. 주로 하나의 함수나 메서드, 또는 개별 클래스가 예상대로 동작하는지 확인합니다.
    *   빠르게 실행되며, 버그를 가장 먼저 발견할 수 있습니다.
2.  **통합 테스트 (Integration Test)**
    *   **여러 모듈들(또는 컴포넌트들)이 상호작용할 때 잘 동작하는지 검증**합니다. 데이터베이스 접근, 외부 API 호출 등 다양한 요소가 통합되었을 때의 시나리오를 테스트합니다.
3.  **E2E (End-to-End) 테스트**
    *   **사용자의 관점에서 전체 시스템의 흐름을 처음부터 끝까지 테스트**합니다. 브라우저 자동화 도구(예: Cypress, Playwright)를 사용하여 실제 사용자가 버튼을 클릭하고 페이지를 이동하는 과정을 시뮬레이션합니다.

---

## 2. 테스트 코드를 작성해야 하는 4가지 이유 🚀

"기능 구현하기도 바쁜데, 왜 테스트 코드까지 짜야 할까?"라는 의문이 들 수 있습니다. 하지만 테스트 코드는 장기적으로 보았을 때 훨씬 더 큰 이득을 가져다줍니다.

### ① 리팩토링과 기능 확장에 대한 '안전망(Safety Net)'
가장 큰 장점입니다. 새로운 기능을 추가하거나 기존 코드를 더 나은 구조로 리팩토링할 때, 혹시 다른 곳에서 버그가 발생하지 않았을까 하는 불안감이 생깁니다. 이때 잘 짜여진 테스트 코드가 있다면 **수정 후 테스트를 돌려보는 것만으로 기존 기능들이 망가지지 않았음을 확신**할 수 있습니다. 즉, 두려움 없이 코드를 개선할 수 있게 됩니다.

### ② 개발 과정에서의 빠른 피드백 (버그 조기 발견)
수동으로 브라우저를 띄우고, 로그인하고, 특정 페이지까지 들어가서 버튼을 눌러 테스트하는 과정은 매우 번거롭고 시간이 오래 걸립니다. 단위 테스트를 작성해두면 코드를 수정하자마자 수 초 내에 버그 여부를 피드백 받을 수 있습니다. 발견 시점이 빠를수록 버그를 고치는 비용도 기하급수적으로 줄어듭니다.

### ③ 테스트 코드가 곧 의사소통 수단이자 '문서(Doc)'
다른 사람(또는 미래의 나)이 작성한 코드를 이해하는 것은 어렵습니다. 하지만 테스트 코드를 보면 **"이 함수는 어떤 예외 상황이 있고, 어떤 값을 넣었을 때 어떤 결과를 반환하도록 설계되었구나"**를 바로 알 수 있습니다. 즉, 살아 숨 쉬는 명세서 역할을 합니다.

### ④ 높은 코드 품질 강제
테스트 코드를 작성하기 어려운 코드가 있습니다. 보통 너무 많은 역할을 하고 있거나 다른 객체와의 결합도가 높은 경우(강한 의존성) 그렇습니다. 테스트 코드를 짜려고 노력하다 보면 자연스럽게 함수를 잘게 쪼개고 결합도를 낮추는 방향으로 코드를 설계하게 되어, 전체적인 객체 지향적인 설계와 코드 품질이 향상됩니다.

## 3. 간단한 테스트 코드 예시 (JavaScript - Jest)

아주 간단한 계산기 함수를 예로 들어보겠습니다.

**프로덕션 코드 (`calculator.js`)**
```javascript
export function add(a, b) {
  return a + b;
}
```

**테스트 코드 (`calculator.test.js`)**
```javascript
import { add } from './calculator';

// 테스트 스위트 (그룹)
describe('Caculator add 함수 검증', () => {
    
  // 개별 테스트 (Test Case)
  test('1 + 2는 3을 반환해야 한다.', () => {
    // given (준비단계)
    const num1 = 1;
    const num2 = 2;

    // when (실행단계)
    const result = add(num1, num2);

    // then (검증단계)
    expect(result).toBe(3); // 결과가 3일 것이라 기대한다!
  });
  
});
```
위 코드를 실행하면 `Jest` 테스트 러너가 통과 여부를 즉각적으로 알려줍니다.

## 4. 마치며

테스트 코드 작성은 초기에는 개발 시간을 잡아먹는 비효율적인 일처럼 보일 수 있습니다. 하지만 소프트웨어의 생명 주기는 길고, 유지보수의 시간은 개발 시간보다 훨씬 깁니다. 

**"어제는 동작했는데 오늘 왜 안 되지?"**라는 고민에서 벗어나, 안심하고 기능 개발을 할 수 있는 든든한 환경을 만들기 위해 오늘부터 조금씩이라도 테스트 코드 작성을 시작해 보는 것은 어떨까요? 👍

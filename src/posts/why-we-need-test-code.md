# 바쁜 일정 속에서도 테스트 코드를 묶어둬야 했던 이유

실무 코드 베이스 위에서 새로운 기능을 얹어갈 때, 팀 단위 협업에서 "테스트 코드를 의무적으로 함께 붙여야 한다"는 지침은 일정 압박에 쫓기는 상황에서는 그저 짐스러운 생산성 저해 요소처럼 여겨졌던 시절이 있었다.

개발한 로직을 API 설계 툴(Postman)로 쏴보거나 브라우저에서 버튼을 눌러 수동 확인만 체크하면 충분하다 여겼지만, 모듈의 몸집이 비대해지면서 발생한 심각한 퇴보 사이드 이펙트를 직면한 이후부터, **자동화된 테스트 코드 명세망**의 존재 가치가 어떻게 소프트웨어 엔지니어 생태계를 구원하는지 맹위를 실감했던 과정을 고찰해 본다.

---

## 🛑 수동 테스트와 암흑기: "내 자리에서는 원래 잘 됐는데요"

새로운 요구사항으로 특정 유틸 함수 컴포넌트 내부 로직을 리팩토링 및 엎어치고 배포한 날, 수정 부위와 전혀 상관없다고 여겼던 결제 모듈 반대편 구간에서 `NullPointerException`이 속출하는 대참사가 터졌다.

변경한 코드의 파생 효과가 나도 모른 채 전역의 기존 모듈을 작살내 버렸으나, 매번 배포 빌드 시마다 수많은 기능 단위를 완벽히 전체 수동 손수 클릭 테스트(Regression Test)하며 검수하기란 물리적으로 절대 불가한 일이었다.

---

## 🛡️ 테스트 코드가 제공하는 막강한 강제 방어막 (Safety Net)

이런 폭탄 돌리기의 두려움을 척결하기 위해 프로젝트 통합 레벨에 단위 검증 파이프라인(Unit Test / Integration Test)을 코드 형태로 아키텍처링 하여 심기 시작했다.

### 1. 두려움 없는 리팩토링 개조 작업의 지원군
테스트 코드가 튼튼히 직조되어 있는 함수 파트는, 기존 로직 구성을 갈아엎고 디자인 패턴을 적용해 성능을 올리는 마개조를 진행하더라도 "혹시나 다른 기능이 터졌을까?" 하는 막연한 공포감을 말소시켰다. 수정을 마친 후 콘솔 창에 `npm test` 명령어를 타건했을 때 뜨는 통과 불빛만 확인하면, 최소한 100% 기존 로직들은 망가지지 않고 정상 동작할 것임을 보장받은 채 과감히 시스템을 밀어버릴 기술적 자신감을 얻어냈다.

### 2. 코드 자체가 완벽한 살아 숨 쉬는 설계 명세서(Docs) 등극
새로 합류한 팀원의 거대한 로직을 리뷰해야 하거나, 한 달 뒤의 내가 예전 로직을 분석할 때 어설픈 주석을 뒤적일 필요가 소멸했다. 단위 테스트 파일 껍데기에 적힌 명시적 로직만을 들여다보면, **"이 함수는 널(Null) 값과 음수 파라미터가 인입되었을 때 어떤 예외 코드를 무조건 뱉도록 설계되어 조립되었는가"**의 의도 시나리오(Given-When-Then 패턴)가 코드 레벨에서 명백한 입증 문서 역할을 겸하게 되는 신세계를 체감했다.

```javascript
// 단순하지만 강력한 방어선 문서
describe('결제 수수료 계산 모듈 아키텍처 검증망', () => {
    
  test('마이너스 금액 파라미터 인입 시, 무조건 에러 예외를 격발 반환해야 한다.', () => {
    // given 
    const invalidAmount = -500;
    // when & then
    expect(() => calculateFee(invalidAmount)).toThrow('Invalid Amount Data'); 
  });
});
```

### 3. 기능의 책임을 분리시키는 파생 효과
스파게티처럼 얽히고설켜 의존도가 높은 거대 컴포넌트는 태생적으로 구조상 고립된 깔끔한 테스트 코드를 짤 수가 없도록 강제 제약을 받는다. 테스트 블록을 작성하려고 몸부림치는 과정 그 자체만으로도, 개발자는 역설적이게도 "어떻게든 이 함수의 역할을 쪼개고 목업(Mock) 결합도를 분리해야만 한다"는 단일 책임 원칙(SRP) 설계 압박을 받게 되어, 구조적인 코드 개선의 나침반 역할을 해주었다.

## 💡 최종 회고 고찰

당장의 피처 스펙 오픈 일정을 맞추기 위해 테스트 코드 작성 스텝을 건너뛰는 것은 한 달 뒤 런타임 유지보수 단계에서 발생하는 디버깅 트러블 조달을 복리로 땡겨오는 기술 인프라 부채(Technical Debt)라는 확고함을 체화했다.

테스트 코드의 주도권 도입은 그저 프로젝트 세팅의 부가적인 절차가 아니었다. 수백만 줄로 비대해지는 닫힌 시스템 안에서 과거 내가 만들어둔 코드 시스템이 오늘 내가 수정한 코드에 의해 망가지지 못하게 차단선을 보장해 주는 유일하고도 합리적인 안전 인프라 방파제임을 통쾌하게 깨달았다.

# MSA(Microservices Architecture) 도입과 학습 회고

작은 규모의 프로젝트를 벗어나 여러 기능이 결합된 상거래 플랫폼 프로젝트를 기획하면서, 처음으로 **MSA(Microservices Architecture)**를 도입해 보았습니다.

그동안 익숙했던 거대한 단일 코드베이스(Monolithic)를 쪼개는 과정에서 겪은 수많은 아키텍처적 고민과, MSA의 본질에 대해 학습한 내용을 회고합니다.

---

## 1. MSA란 무엇이며, 왜 도입했는가?

MSA는 애플리케이션의 핵심 비즈니스 로직들을 **작고 독립적인 여러 개의 서비스 단위**로 분리하여 개발, 배포, 운영하는 아키텍처 가이드라인입니다.

기존 모놀리식 구조에서는 '주문' 쪽 코드를 수정하다가 생긴 버그 때문에 '상품 조회' 기능까지 멈추고 서버가 다운되는 경험을 한 적이 있습니다. 서비스의 규모가 커짐에 따라 파급력을 줄이고 **장애 격리(Fault Isolation)**를 달성하기 위해 MSA 분리를 시도했습니다.

---

## 2. 도입하며 체감한 장점 (Pros)

이론으로만 알던 장점들을 실제 구축 단계에서 일부 느낄 수 있었습니다.
1. **독립적인 배포**: 회원 서비스에 기능을 추가하고 싶을 때, 거대한 전체 프로젝트를 빌드하고 내릴 필요 없이 회원 서비스(컨테이너) 하나만 재배포하면 되어 속도감이 달랐습니다.
2. **기술 스택의 자율성**: 모든 서비스를 하나의 언어로만 짤 필요가 없었습니다. 대부분은 Java/Spring 기반으로 구축하되, 특정 데이터 수집 모듈은 가벼운 Node.js로 분리하는 유연함을 확보할 수 있었습니다.
3. **부분적 스케일 아웃**: 트래픽 부하 테스트 시, 가장 병목이 심한 주문 서비스만 여러 대의 서버로 증설하고 라우팅시킬 수 있어 인프라 자원의 효율성이 높아졌습니다.

---

## 3. 마주친 거대한 단점과 도전 과제 (Cons & Challenges)

단순히 파일만 분리하면 끝이라고 생각했지만, MSA의 진정한 어려움은 인프라 복잡도와 네트워크 통신에 있었습니다.

1. **테스트와 디버깅의 고통**: 하나의 요청 흐름이 API Gateway를 거쳐 주문 서버, 그리고 결제 서버까지 네트워크를 타고 넘어갑니다. 로컬 환경에서 테스트 한 번 하려면 수많은 서버를 동시에 띄워야 했고, 중간에 끊기면 어디서 문제가 발생했는지 추적하기가 기하급수적으로 복잡해졌습니다.
2. **트랜잭션(데이터 일관성) 분리**: 각 서비스가 자신만의 DB를 가지게 되므로(`Database per Service` 원칙), 단일 DB에서 편하게 쓰던 롤백(`@Transactional`)이 불가능해졌습니다. 분산 트랜잭션과 최종 정합성에 대한 깊은 고민(Saga 패턴 등)을 강제 받게 되었습니다.
3. **네트워크 오버헤드**: 내부 메서드 호출이면 곧방 끝날 연산이, HTTP 네트워크를 경유하면서 응답 지연 시간과 타임아웃 예외 처리 로직이라는 추가 비용을 발생시켰습니다.

---

## 4. MSA 핵심 생태계의 필요성 체감

위의 문제들을 방어하기 위해서는 순수 비즈니스 로직 외에도 **인프라 레벨의 패턴**들이 강제됨을 배웠습니다.
- **API Gateway**: 클라이언트 요청의 단일 진입점 및 라우팅.
- **Event Bus (Kafka)**: 서버 간의 강결합을 끊어내기 위한 비동기 메시징 큐.
- **Trace 모니터링 체계 (Zipkin 등)**: 분산된 로그를 하나의 흐름으로 묶어주는 인프라.

## 회고

MSA는 완벽한 해결책이 아니었습니다. 독립성과 확장성이라는 이점을 얻기 위해 **시스템 구조의 막대한 복잡성**을 감수해야 하는 전형적인 트레이드오프(Trade-off) 아키텍처였습니다. 개인적으로는 시스템의 확장 요구나 트래픽 규모가 모놀리식의 한계를 명백히 넘어서는 시점에 고려해야 하는 묵직한 설계 방식임을 깊이 느꼈습니다.

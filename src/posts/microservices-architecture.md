# MSA (Microservices Architecture)

## 1. 정의
MSA는 하나의 거대한 애플리케이션을 **작고 독립적인 여러 개의 서비스(Service)** 단위로 쪼개어 개발, 배포, 운영하는 아키텍처 스타일입니다. 각 서비스는 독립적인 기능을 수행하며, HTTP API나 메시지 큐(Message Queue) 등을 통해 통신합니다.

*   **기존 방식 (Monolithic)**: 모든 비즈니스 로직, 데이터베이스, UI가 하나의 거대한 코드베이스와 배포 단위로 묶여 있는 형태.

## 2. 주요 특징
1.  **독립적인 배포**: 특정 서비스만 수정하여 시스템 전체 중단 없이 배포가 가능합니다.
2.  **기술의 다양성 (Polyglot)**: 서비스마다 가장 적합한 언어, 프레임워크, 데이터베이스를 선택할 수 있습니다. (예: 주문 서비스는 Java/Spring, 추천 서비스는 Python/Django)
3.  **독립적인 확장 (Scale-out)**: 트래픽이 몰리는 특정 서비스(예: 주문)만 서버를 늘려 확장이 가능합니다.
4.  **분산 데이터 관리**: 각 마이크로서비스는 자신만의 전용 데이터베이스를 가집니다 (Database per Service 패턴).

## 3. 장점 (Pros)
*   **민첩성 (Agility)**: 서비스가 작아 코드를 이해하고 수정하기 쉬우며, 빠른 개발과 배포 주기를 가질 수 있습니다.
*   **유연한 확장성**: 리소스를 효율적으로 사용할 수 있습니다. 필요 없는 부분까지 확장할 필요가 없습니다.
*   **장애 격리 (Fault Isolation)**: 하나의 서비스(예: 회원 서비스)가 다운되어도 다른 서비스(예: 상품 목록 조회)는 정상 작동할 수 있습니다.
*   **조직 구조와의 일치**: 소규모 팀이 각 서비스를 전담하여 책임지고 운영하기 좋습니다 (Conway's Law).

## 4. 단점 및 도전 과제 (Cons & Challenges)
*   **복잡성 증가**: 분산 시스템이므로 개발, 배포, 테스트, 모니터링이 모놀리식보다 훨씬 복잡합니다.
*   **데이터 일관성**: 서비스 간 데이터가 분리되어 있어 트랜잭션 관리(ACID 준수)가 어렵습니다. (Saga 패턴 등으로 해결 필요)
*   **통신 비용**: 서비스 간 통신(Network Call) 오버헤드가 발생하며, 네트워크 장애 처리가 필요합니다.
*   **통합 테스트의 어려움**: 여러 서비스가 연계되어 있어 통합 테스트 환경을 구성하기 까다롭습니다.

## 5. MSA 구현을 위한 핵심 패턴 및 기술

MSA를 제대로 운영하기 위해서는 인프라와 플랫폼 차원의 지원이 필수적입니다.

| 구분                  | 역할                                                          | 대표 기술/패턴                            |
| :-------------------- | :------------------------------------------------------------ | :---------------------------------------- |
| **API Gateway**       | 모든 클라이언트 요청의 진입점. 인증, 라우팅, 로드밸런싱 담당. | Spring Cloud Gateway, NGINX, Netflix Zuul |
| **Service Discovery** | 동적으로 변하는 서비스들의 IP/Port를 등록하고 찾아줌.         | Netflix Eureka, Consul, Kubernetes DNS    |
| **Circuit Breaker**   | 장애가 난 서비스로의 요청을 차단하여 장애 전파를 막음.        | Resilience4j, Netflix Hystrix             |
| **Config Server**     | 분산된 서비스들의 설정 정보(DB 정보 등)를 중앙에서 관리.      | Spring Cloud Config                       |
| **Tracing**           | 분산 환경에서 요청의 흐름을 추적하여 모니터링.                | Zipkin, Jaeger, OpenTelemetry             |
| **Event Bus**         | 서비스 간 비동기 메시지 통신.                                 | Kafka, RabbitMQ                           |
| **Data Consistency**  | 분산 트랜잭션 처리.                                           | Saga Pattern, CQRS, Event Sourcing        |

## 6. 요약 비교

| 구분          | Monolithic                   | Microservices               |
| :------------ | :--------------------------- | :-------------------------- |
| **구조**      | 하나의 거대한 단일 프로세스  | 작은 서비스들의 집합        |
| **복잡도**    | 초기에는 단순, 커질수록 복잡 | 초기부터 복잡 (인프라 등)   |
| **배포**      | 전체 재배포 필요 (느림)      | 서비스별 개별 배포 (빠름)   |
| **확장**      | 전체 복제 방식 (비효율)      | 필요한 서비스만 확장 (효율) |
| **장애 영향** | 하나가 죽으면 전체 마비 위험 | 장애가 해당 서비스에 국한됨 |

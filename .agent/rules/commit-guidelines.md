# 커밋 가이드라인 (Commit Guidelines)

이 가이드라인은 프로젝트의 이력을 명확하게 관리하고, 의미 있는 단위로 작업을 나누어 기록하기 위한 규칙입니다.

## 1. 커밋 메시지 컨벤션
- **언어**: 커밋 메시지는 **한국어**로 작성합니다.
- **형식**: `태그: 요약` 형식을 사용합니다.
- **태그**:
    - `feat`: 새로운 기능 추가
    - `fix`: 버그 수정
    - `docs`: 문서 수정 (README, 블로그 포스팅 등)
    - `style`: 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음)
    - `refactor`: 코드 리팩토링
    - `test`: 테스트 코드 추가 및 수정
    - `chore`: 빌드 업무, 패키지 매니저 설정 등 (프로덕션 코드 변경 없음)
- **톤앤매너**: 명확하고 간결한 문장으로 작성하며, "~함", "~함 추가"와 같이 명사형 종결 어미를 권장합니다.

## 2. 작업 단위 그룹화 원칙
- **원자성(Atomicity)**: 하나의 커밋은 하나의 논리적인 변화만 담아야 합니다.
- **관련성**: 연관된 파일들(예: 기능 코드 + 해당 기능의 테스트 코드)은 하나의 작업 단위로 묶습니다.
- **독립성**: 각 커밋은 그 자체로 의미가 있어야 하며, 빌드가 깨지지 않는 상태를 유지해야 합니다.
- **대규모 변경 분할**: 한 번에 너무 많은 파일을 수정했다면, 이를 논리적인 단계(예: 데이터 모델 변경 -> 비즈니스 로직 적용 -> UI 반영)로 나누어 제안합니다.
